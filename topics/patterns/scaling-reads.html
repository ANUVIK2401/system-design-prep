Scaling Reads

Read replicas, caching strategies, and CDNs — the standard toolkit for read-heavy systems.

15-20 min Medium
Say this in the first 2 minutes
  • Most systems are read-heavy — solve reads before you ever think about sharding writes
  • Layer the levers: cache first (cheapest, fastest), then read replicas, then CDN for static/edge content
  • Cache-aside is the default caching pattern — check cache, miss falls to DB, populate cache, return
  • Read replicas introduce replication lag — call out the read-your-writes problem explicitly
  • Cache invalidation is the hard part, not the cache itself — say this and you sound experienced

Mental model

Scaling reads is like serving customers at a bakery: a sticky note on the counter (cache) answers the common "do you have croissants" question instantly. If the note's outdated, you check the back kitchen (replica). For city-wide chains, a nearby branch (CDN edge) is faster than calling headquarters (origin) every time.

Core concepts

TermDefinition
Read replicaA copy of the primary DB that serves reads; updated via async/sync replication
Replication lagDelay between a write on the primary and its visibility on a replica
Cache-asideApp checks cache first; on miss, reads DB and populates the cache itself
Write-throughEvery write goes to cache and DB together, synchronously
Write-behindWrite hits cache immediately; DB write happens asynchronously after
CDNGeographically distributed edge cache for static/semi-static content

Read replicas + cache-aside

Checked first — hot path
Source of truth — fallback on miss
  1. Check cache: app looks up the key in Redis first
  2. Hit: return immediately, ~1ms, no DB touched
  3. Miss: query a read replica, get the value
  4. Populate: write the value into the cache with a TTL, then return it

Tradeoffs

Wins
  • Cache-aside: simple, only caches what's actually requested
  • Read replicas: linearly scales read throughput, isolates read load from writes
  • CDN: removes cross-region latency entirely for cacheable content
Costs
  • Replication lag breaks "read your own write" without extra handling
  • Cache invalidation on update is easy to get wrong (stale reads)
  • Write-through adds write latency; write-behind risks data loss on cache crash

When to use

✓
Use cache-aside + replicas as the default for read-heavy APIs — product pages, profiles, feeds — where slight staleness is acceptable.
✓
Use a CDN for static assets and rarely-changing API responses (e.g. public catalog data) to cut latency for geographically distant users.
⚠
Watch for read-your-writes right after a user submits a form — route that specific read to the primary, or read from cache populated by the write itself.

Interview moves

  • 01
    "This is read-heavy, so I'll layer caching, then read replicas, then a CDN before I even consider sharding writes" — states the order of operations explicitly
  • 02
    "I'd use cache-aside as the default — populate on miss, so we only cache what's actually being read" — names the pattern and justifies it over write-through
  • 03
    "Replicas introduce lag, so right after a write I'd either read from primary or rely on the cache I just populated, to avoid the read-your-writes problem" — anticipates a classic follow-up
  • 04
    "Cache invalidation is the hard part — I'd use a short TTL plus explicit invalidation on write, rather than relying on TTL alone" — shows you know caching isn't free
  • 05
    "For static or rarely-changing data I'd push it to a CDN so cross-region users aren't paying our DB's round-trip latency" — connects CDN choice to actual latency numbers

Common follow-up questions

When does replication lag actually matter?
Right after a write, when the same user (or a dependent process) reads it back immediately and the replica hasn't caught up yet — causes a confusing "my change disappeared" bug.
How do you keep cache and DB in sync on update?
On write, invalidate (delete) the cache key rather than updating it in place — the next read repopulates it correctly. Updating the cache directly risks races with concurrent writers.
What's a cache stampede and how do you prevent it?
Many requests miss the same hot key simultaneously and all hit the DB at once. Prevent with a short-lived lock/single-flight on cache repopulation, or stagger TTLs with jitter.
Write-through vs write-behind — when would you pick write-behind?
Write-behind when write latency matters more than durability guarantees (e.g. view counters, analytics) — accept a small risk of losing the last few writes if the cache crashes before flushing.
How many read replicas can you add before it stops helping?
Each replica adds replication overhead on the primary and increases lag variance. Past 5-10 replicas, consider sharding or a dedicated read-optimized store (e.g. Elasticsearch, a CQRS read model) instead.
How does a CDN know when to refresh stale content?
Cache-Control headers set a TTL; on expiry the CDN revalidates with the origin (or you actively purge/invalidate specific paths on deploy/update).
