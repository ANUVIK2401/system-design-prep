Scaling Writes

Sharding, write buffering, async writes, and batching — how to absorb write volume a single DB can't handle.

20-25 min Hard
Say this in the first 2 minutes
  • Reach for write scaling only after caching and replicas are maxed out — it's the most operationally expensive lever
  • Sharding splits data across N databases by a shard key — pick the key based on your access pattern
  • Hash sharding spreads load evenly but kills range queries; range sharding enables range queries but risks hot shards
  • A queue in front of the DB absorbs write spikes — the DB drains it at a steady, sustainable rate
  • Async writes trade some durability for throughput — be explicit about what you're willing to lose

Mental model

Scaling writes is like a post office overwhelmed at one counter. Sharding opens multiple counters, each handling a fixed slice of the alphabet. A queue is the line that forms outside — it lets people arrive in bursts while clerks process at a steady pace.

Core concepts

TermDefinition
ShardingSplitting a dataset across multiple independent databases by a shard key
Hash shardingshard = hash(key) % N — evenly distributes load, destroys ordering
Range shardingShard boundaries are key ranges — keeps order, risks a hot shard
Hot partitionOne shard receives disproportionate traffic (e.g. a viral key)
Key saltingAppend a random suffix to a hot key to spread its writes across shards
Write bufferingA queue absorbs bursts; consumers write to the DB at a sustainable rate
BatchingGrouping multiple writes into one DB round-trip to cut per-write overhead

Tradeoffs

Hash sharding
  • Even distribution — no single shard naturally gets more traffic
  • Simple router logic, predictable capacity planning
  • Cost: range queries ("all orders this week") require fanning out to every shard
Range sharding
  • Range queries stay efficient — adjacent keys live on the same shard
  • Easy to reason about and rebalance by splitting a range
  • Cost: sequential/skewed keys (timestamps, celebrity IDs) create hot shards

When to use

✓
Use sharding once a single primary can't handle write QPS even after read offload, write buffering, and vertical scaling are exhausted.
✓
Use a write buffer/queue for bursty traffic (flash sales, IoT spikes) so the DB drains writes at a steady rate instead of falling over.
✗
Avoid sharding too early — it adds cross-shard query complexity, rebalancing operations, and multi-shard transaction headaches. Exhaust simpler levers first.

Common follow-up questions

How do you pick a shard key?
Pick the field most queries filter by, with high cardinality and even distribution. A bad key (e.g. status with 3 values) creates massive imbalance.
How do you handle a query that needs to join across shards?
Avoid it where possible by denormalizing. Otherwise, fan out to all shards in parallel and merge in the application layer — accept the latency cost.
What's key salting and when do you need it?
Appending a random suffix (e.g. user:482#3) to spread a hot key's writes across multiple shards/partitions, then aggregating reads across the salted variants. Needed when one key (a viral post, a popular product) dominates traffic.
How do you rebalance shards as data grows?
Hash sharding: use consistent hashing so adding a shard only remaps a fraction of keys. Range sharding: split an oversized range into two and migrate that slice only.
What's the durability tradeoff with async writes?
A write acknowledged to the client before it's durably persisted can be lost on crash. Mitigate with a write-ahead log or by accepting the loss for non-critical data (e.g. view counts).
When does batching writes help, and when does it hurt?
Helps when per-write overhead (network round-trip, transaction commit) dominates — batch 100s of rows per insert. Hurts latency-sensitive single writes, since you wait for the batch window to fill or flush.
