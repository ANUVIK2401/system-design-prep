Real-Time Updates

How to push fresh data to clients — polling, long-polling, WebSockets, and SSE — and how to justify the choice.

15-20 min Medium
Say this in the first 2 minutes
  • Ask: is this one-way server push, or does the client also need to send frequent messages?
  • One-way push (feeds, notifications, live scores) → SSE, not WebSockets — simpler infra, free reconnect
  • Bidirectional + low-latency (chat, games, collab editing) → WebSockets
  • WebSockets are stateful connections — horizontal scaling needs a pub/sub backplane (Redis) so any node can deliver to any client
  • Long-polling is the fallback when WS/SSE are blocked by a corporate proxy, not the primary design

Mental model

Polling is calling the restaurant every 5 minutes to ask if your order is ready. Long-polling is staying on hold until they tell you. SSE is the restaurant texting you the moment it's done. WebSockets is a walkie-talkie — both sides can talk anytime, on the same open channel.

Core concepts

TermDefinition
PollingClient repeatedly requests on a timer; mostly empty responses
Long-pollingServer holds the request open until data is ready, then responds
SSEServer-Sent Events — one persistent HTTP connection, server streams events
WebSocketFull-duplex socket over a single TCP connection after an HTTP upgrade
Sticky sessionLoad balancer routes a client to the same backend node every request
BackplaneShared pub/sub layer (Redis) so any server node can broadcast to any connected client
HeartbeatPeriodic ping/pong to detect dead connections behind proxies and NATs

Connection lifecycle: polling vs WebSocket

Client
Persistent connection (WS / SSE)
  1. Polling: client opens and closes a connection on every tick — most responses carry nothing new
  2. Long-polling: server delays the response until data exists or a timeout hits, then the client immediately reopens
  3. SSE / WebSocket: one connection stays open; server pushes data the instant it's available, no re-request

SSE wins
  • Just HTTP — works through normal proxies, load balancers, CDNs
  • Browser auto-reconnects with Last-Event-ID for free
  • Much less server-side state to manage than WebSockets
  • Simple to reason about: it's a stream, not a protocol
WebSocket costs
  • Stateful — needs sticky sessions or a shared backplane to scale out
  • No native reconnect or message replay — you build it
  • Harder to load balance and harder to debug in prod
  • Overkill if the client never needs to send much back

When to use

✓
Use SSE for live feeds, notification badges, stock tickers, progress bars — anything where only the server speaks.
✓
Use WebSockets for chat, multiplayer games, collaborative editing — frequent bidirectional, low-latency messages.
⚠
Watch out for corporate proxies/firewalls that block long-lived connections — keep a long-polling fallback path for those clients.

Interview moves

  • 01
    "Does the client need to send data back frequently, or just receive it?" — this single question picks SSE vs WebSocket for you
  • 02
    "I'd default to SSE — it's just HTTP, gets auto-reconnect for free, and is much easier to scale" — shows you don't reach for WebSockets reflexively
  • 03
    "WebSockets are stateful, so a load balancer alone can't route a push to the right client — I'd add a Redis pub/sub backplane" — the single most important WS scaling fact
  • 04
    "I'd keep long-polling as a fallback, not the primary path, for clients behind proxies that strip WebSocket upgrades" — shows pragmatism about real-world networks
  • 05
    "On reconnect, the client needs to resync missed state — either replay from a sequence number or just refetch a snapshot" — addresses the gap nobody mentions until asked

Common follow-up questions

How do you scale WebSockets across multiple servers?
Sticky sessions route a client to the same node. Use Redis pub/sub (or Kafka) as a backplane so any node publishing an event reaches the node holding that client's socket.
Why not just use WebSockets for everything?
Stateful connections are expensive to scale, harder to load balance, and need custom reconnect/heartbeat logic. SSE gets you 90% of the value with HTTP-level simplicity for one-way push.
How does a client recover missed messages after disconnect?
SSE: send Last-Event-ID header on reconnect, server replays from there. WebSocket: you implement a sequence number and a "resync" message yourself.
What's the failure mode of long-polling at scale?
Each held request ties up a server thread/connection slot. At high concurrency this exhausts connection pools faster than short polling — needs async I/O servers (Node, Netty) to hold many idle connections cheaply.
How do you detect a dead WebSocket connection?
Ping/pong heartbeat every 30s. If no pong within a timeout, the server closes and the client reconnects. Without this, dead sockets behind NATs linger silently.
Mobile clients drain battery on persistent connections — what do you do?
Use platform push notifications (APNs/FCM) to wake the app instead of holding an open socket in the background; only open WS/SSE while the app is foregrounded.
