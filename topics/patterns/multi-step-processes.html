Multi-Step Processes

The saga pattern — chaining local transactions across services with compensating actions instead of distributed transactions.

15-20 min Medium
Say this in the first 2 minutes
  • Distributed transactions (2PC) don't scale — they hold locks across services and fail badly on partition
  • Saga = sequence of local transactions, each with a compensating action to undo it
  • Orchestration = central coordinator calls each step; choreography = services react to each other's events
  • Compensating actions are business operations, not a generic rollback — "refund payment", not "undo INSERT"
  • Sagas give eventual consistency only — say this explicitly so the interviewer knows you're not claiming ACID

Mental model

A saga is like booking a multi-leg trip yourself instead of through one travel agent: you book the flight, then the hotel, then the car. If the car rental fails, you don't get a magic global undo — you cancel the hotel and refund the flight, one call at a time.

Core concepts

TermDefinition
SagaA sequence of local transactions across services, coordinated to act as one logical operation
Local transactionA normal ACID transaction scoped to a single service's own database
Compensating transactionA business-level action that semantically undoes a prior step (e.g. refund)
OrchestrationA central coordinator service explicitly calls each step and handles failures
ChoreographyEach service publishes events; the next service reacts — no central coordinator
2PCTwo-phase commit — a distributed transaction protocol that locks all participants until all agree

Orchestrated saga with compensating rollback

Orchestrator / confirm
Forward step (local transaction)
Compensating transaction (rollback)
  1. Forward path: orchestrator calls order → payment → inventory in sequence, each a local ACID transaction
  2. Failure detected: inventory step fails (out of stock) — orchestrator stops the forward path
  3. Compensate: orchestrator calls compensating actions in reverse order — refund payment, then cancel order
  4. End state: system reaches a consistent "cancelled" state — eventually, not instantly

Tradeoffs

Orchestration
  • One place to read the whole workflow — easier to understand and debug
  • Centralized retry/timeout/compensation logic
  • Easy to add monitoring and a saga state machine
  • Cost: the orchestrator is a new service and a potential bottleneck
Choreography
  • No central coordinator — services stay fully decoupled
  • No single point of failure for the workflow itself
  • Cost: hard to see the overall flow — logic is smeared across services
  • Cost: harder to add a new step without touching multiple services' event handlers

When to use

✓
Use orchestration for complex workflows with many steps, conditional branches, or where you need a clear audit trail (e.g. order checkout, loan approval).
✓
Use choreography for simple 2-3 step flows where services are already event-driven and you want to avoid adding a new coordinator service.
✗
Avoid 2PC across microservices — it requires all participants to hold locks until every vote is in, which kills availability and doesn't survive a partition.

Interview moves

  • 01
    "This spans three services with their own databases — I can't use one ACID transaction, so I'd use the saga pattern" — names the trigger and the pattern together
  • 02
    "I'd go with orchestration here since there are several steps and I want a single place to see the workflow state" — justifies the choice instead of defaulting blindly
  • 03
    "Each compensating action is a real business operation — refund the payment, release the inventory hold — not a generic database rollback" — shows you understand sagas aren't free
  • 04
    "Sagas only give eventual consistency — for a brief window the order exists in a partially-completed state, and I'd model that explicitly" — preempts the "what about consistency" follow-up
  • 05
    "Every step and compensation needs to be idempotent, since the orchestrator may retry after a timeout without knowing if the first call landed" — connects sagas to retry-safety

Common follow-up questions

Why not just use a distributed transaction (2PC)?
2PC holds locks on every participant until all vote commit — this blocks unrelated traffic and doesn't tolerate a coordinator or participant failure well. It doesn't scale to internet-scale microservices.
What happens if a compensating transaction itself fails?
Retry with backoff; if it keeps failing, push to a dead-letter queue for manual/ops intervention. Compensations must be designed to be retried safely (idempotent).
How do you give the user a consistent view during a saga in progress?
Show an explicit intermediate state like "processing" rather than a final state. Don't let the UI imply success until the saga's last step confirms.
Can every step be compensated?
No — some actions are irreversible (sending an email, an external non-refundable charge). Design those steps to be last in the sequence, after all reversible steps have succeeded.
How does choreography avoid a single point of failure?
Each service only knows about events it cares about and reacts independently via a message broker — there's no central process whose crash halts the whole workflow.
How do you debug a choreographed saga gone wrong?
Correlate all events with a single saga/trace ID and centralize logs (e.g. in a tracing system) — since there's no orchestrator state machine to inspect directly.
