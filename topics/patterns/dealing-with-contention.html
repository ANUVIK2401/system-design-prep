Dealing with Contention

Race conditions, optimistic vs pessimistic locking, and distributed locks — how to stop two writers from stepping on each other.

15-20 min Medium
Say this in the first 2 minutes
  • Name the race condition explicitly: "two requests read the same value, both write back, one update is lost"
  • Default to optimistic locking (version column) — it scales better because it doesn't hold a lock during I/O
  • Reach for pessimistic locking only when conflict rate is high and retries would thrash
  • Distributed locks (Redis/Zookeeper) are for cross-service coordination, not single-DB row updates
  • Any distributed lock needs a TTL — and any TTL needs a fencing token, or you risk two holders at once

Mental model

Contention is two people grabbing the same shopping cart in a store. Optimistic locking lets both reach for it and only complains if they collide — cheap when collisions are rare. Pessimistic locking puts a "do not touch" sign on the cart the moment one person approaches — safer when collisions are common, but everyone else waits.

Core concepts

TermDefinition
Race conditionOutcome depends on the unpredictable timing of concurrent operations
Lost updateOne write silently overwrites another because neither saw the other's change
Optimistic lockingRead with a version, write only if version still matches; retry on mismatch
Pessimistic lockingSELECT ... FOR UPDATE — acquire the lock before reading, hold until commit
Compare-and-swap (CAS)Atomic primitive: "set X to B only if X is currently A"
Distributed lockA mutex that works across processes/machines (Redis SETNX, Zookeeper)
Fencing tokenMonotonically increasing number issued with a lock, so stale lock holders can be rejected

Conflict path — retry with fresh read
  1. Read: fetch the row along with its current version number
  2. Modify: compute the new value in application code
  3. Write: UPDATE ... SET value=?, version=version+1 WHERE id=? AND version=?
  4. Check: if 0 rows affected, someone else won the race — re-read and retry

Tradeoffs

Optimistic locking wins
  • No lock held during I/O — much higher throughput under low contention
  • No deadlock risk between transactions
  • Scales horizontally — works fine across DB replicas/shards
Optimistic locking costs
  • Wasted work on every conflict — must retry the whole read-modify-write
  • Under high contention, retries can thrash and starve some requests
  • Requires every writer to participate (a version/CAS check) — easy to bypass by mistake

When to use

✓
Use optimistic locking as the default for most web app updates — profile edits, inventory counts, cart updates — where conflicts are rare.
✓
Use pessimistic locking for hot, narrow contention like seat booking or flash-sale checkout, where many requests target the exact same row at once.
⚠
Watch out with distributed locks — if a lock holder pauses (GC, network blip) past the TTL, another process can acquire the lock while the first still thinks it holds it. Use fencing tokens.

Interview moves

  • 01
    "I'll add a version column and use it as a compare-and-swap on write — that's optimistic locking, and it's my default" — names the pattern and justifies it as the baseline
  • 02
    "This row gets hammered by concurrent writers, so I'd switch to SELECT FOR UPDATE here specifically — not everywhere" — shows you scope pessimistic locking narrowly
  • 03
    "Across services I can't take a DB lock, so I'd use a Redis-based distributed lock with a short TTL" — shows you know locks don't cross service/DB boundaries
  • 04
    "Any TTL-based lock can expire mid-operation — I'd attach a fencing token so a late writer gets rejected by the resource itself" — the detail that separates senior answers from junior ones
  • 05
    "On conflict I'd retry with exponential backoff and a retry cap, then surface an error rather than retry forever" — shows you've thought about the failure path, not just the happy path

Common follow-up questions

Why is optimistic locking usually the default in interviews?
Most systems have low real contention — locks held during I/O block unrelated traffic and don't scale across replicas. Optimistic locking only pays a cost when conflicts actually happen.
What's a fencing token and why does it matter?
A monotonically increasing number issued alongside a distributed lock. The protected resource rejects any write carrying an older token than the last one seen — stops a "zombie" lock holder from corrupting state after expiry.
When would you choose Zookeeper over Redis for distributed locks?
Zookeeper gives strongly consistent, session-based locks with automatic release on disconnect — better correctness guarantees. Redis (Redlock) is faster and simpler but has known edge cases under clock drift and network partitions.
How do you prevent deadlocks with pessimistic locking?
Always acquire locks in a consistent global order across transactions, keep lock-held time short, and set a lock/statement timeout so a stuck transaction fails instead of hanging others.
How does optimistic locking handle high contention gracefully?
It often doesn't — retries pile up and some requests starve. At that point, switch that specific hot path to pessimistic locking or a queue that serializes writes to the contended resource.
Can you combine optimistic locking with a queue?
Yes — route writes to a contended key through a single-consumer queue (partitioned by key) so they're naturally serialized, eliminating the conflict instead of detecting and retrying it.
