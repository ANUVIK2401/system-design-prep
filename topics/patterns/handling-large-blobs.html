Handling Large Blobs

Object storage, pre-signed URLs, chunked uploads, and CDN delivery — how to move big files without your app server choking.

15-20 min Medium
Say this in the first 2 minutes
  • Large files never flow through your application servers — that's the single most important sentence on this topic
  • App server only issues a pre-signed URL; the client uploads bytes directly to object storage (S3)
  • Chunked/multipart upload makes large files resumable on flaky connections
  • Reads go through a CDN in front of object storage, not through the app server either
  • Object storage gives you durability and infinite scale for free — don't store blobs in your relational DB

Mental model

Your app server is a receptionist, not a loading dock. The receptionist hands you a signed pass (pre-signed URL) so you can drive your truck straight to the warehouse (object storage) — they never touch the cargo themselves.

Core concepts

TermDefinition
Object storageDurable, infinitely scalable blob store keyed by object name (S3, GCS)
Pre-signed URLA time-limited URL granting temporary upload/download permission without exposing credentials
Multipart uploadSplitting a large file into chunks uploaded independently, then assembled server-side
ResumabilityAbility to retry only the failed chunk instead of re-uploading the whole file
CDN originObject storage acts as the origin; CDN caches and serves reads at the edge
Metadata recordA DB row describing the blob (owner, size, status) — the blob itself never lives in the DB

Upload and delivery flow

Client uploads/downloads bytes directly
App server — auth + URL minting only
Object storage — durable blob store
CDN — edge reads
  1. Request: client asks the app server for permission to upload a specific file
  2. Authorize: app server checks auth/quota, asks object storage to mint a pre-signed URL, returns it
  3. Upload: client PUTs the bytes directly to object storage — the app server's job is already done
  4. Serve: reads go through a CDN sitting in front of object storage, never through the app server

Tradeoffs

Pre-signed direct upload
  • App server never holds large request bodies — no memory/bandwidth pressure
  • Upload throughput scales with object storage, not your fleet size
  • Client can upload in parallel chunks for speed and resumability
Proxying uploads through app server
  • Every upload consumes app server memory, CPU, and bandwidth
  • App server becomes the bottleneck and a single point of failure for uploads
  • Harder to scale — you're paying compute cost just to move bytes

When to use

✓
Use pre-signed URLs + object storage for any file upload over a few hundred KB — images, videos, documents, exports.
✓
Use multipart/chunked upload for files above ~50-100MB, or any upload from a mobile/unreliable network connection.
⚠
Watch for orphaned uploads — a pre-signed URL grants upload rights even if the client never tells your app it finished. Use storage event notifications (e.g. S3 → Lambda) to confirm and record completion.

Interview moves

  • 01
    "File bytes never flow through my app server — it only issues a pre-signed URL, the client uploads directly to S3" — the single sentence that signals you know this pattern
  • 02
    "For large files I'd use multipart upload, so a flaky connection only needs to retry the failed chunk, not the whole file" — connects chunking to resumability concretely
  • 03
    "I'd put a CDN in front of object storage so reads never round-trip to the app server or origin region either" — extends the bypass principle to reads, not just writes
  • 04
    "I'd store a metadata row in the DB — owner, size, status — and use a storage event webhook to flip status to 'uploaded' once S3 confirms" — shows you've thought about the upload-confirmation gap
  • 05
    "The pre-signed URL is scoped and time-limited so it can't be reused or shared beyond its purpose" — addresses the obvious security follow-up before it's asked

Common follow-up questions

Why never proxy file bytes through the app server?
It consumes app server memory and bandwidth per upload, turning a stateless compute tier into a data-transfer bottleneck. Direct-to-storage upload scales independently of your app fleet.
How do you know an upload actually succeeded?
Configure object storage to emit an event (e.g. S3 event notification) on object creation, consumed by a worker that updates the metadata record's status — don't trust the client to report success.
How do you validate file type/size before allowing an upload?
Pre-signed URLs can constrain content-type and max size via policy conditions. Re-validate server-side after upload (e.g. a Lambda that inspects the object) since client-side checks can be bypassed.
How do you handle resumable uploads after a connection drop?
Multipart upload tracks which chunk IDs have been acknowledged by storage; on reconnect, the client queries upload status and only resends missing chunks.
How do you serve private files securely through a CDN?
Use signed CDN URLs/cookies with short expiry, or keep private content origin-only and issue a fresh pre-signed download URL per request instead of caching it publicly.
How do you reduce storage cost for old/rarely accessed blobs?
Lifecycle policies that auto-transition objects to cheaper storage tiers (e.g. S3 Infrequent Access, Glacier) after N days of inactivity.
