Managing Long-Running Processes

Async job queues, status polling, webhooks, and idempotency keys — how to handle work that doesn't fit in one request.

15-20 min Medium
Say this in the first 2 minutes
  • If work takes longer than a request timeout, don't block — accept the job, return immediately, process async
  • Queue (SQS/Kafka) + worker pool decouples request latency from actual processing time
  • Client either polls a status endpoint or registers a webhook for completion — name both and pick one with a reason
  • Idempotency keys are mandatory the moment you have retries — a duplicate submission must not double-process
  • Webhooks push complexity (retries, security, ordering) onto you; polling pushes latency and waste onto the client

Mental model

A long-running job is a dry cleaner ticket. You drop off your clothes (submit the job), get a ticket number (job ID), and don't stand in the shop waiting. You either call to check ("is it ready?" — polling) or they call you when it's done (webhook).

Core concepts

TermDefinition
Job queueDurable buffer (SQS/Kafka) holding submitted work until a worker picks it up
Worker poolA fleet of processes that pull jobs off the queue and execute them
Status storeA DB/cache tracking each job's state: queued, running, done, failed
PollingClient repeatedly calls a status endpoint until the job completes
WebhookYour server calls a client-provided URL when the job finishes
Idempotency keyA client-supplied unique ID ensuring a retried submission isn't processed twice

Submit, process, and check status

Client / gateway
Database — source of truth
Async — queue / worker
  1. Submit: client sends the job with an idempotency key, gets back 202 Accepted + job_id instantly
  2. Queue: job sits in a durable queue until a worker is free
  3. Process: worker executes the job, updating status as it progresses
  4. Notify: client either polls GET /jobs/{id} or receives a webhook callback on completion

Tradeoffs

Polling
  • Simple to implement on both client and server — just a GET endpoint
  • No need to expose a public callback URL or verify inbound webhook auth
  • Cost: wasted requests and added latency between actual completion and detection
Webhooks
  • Near-instant notification, no wasted polling traffic
  • Cost: you must handle retries, ordering, and signature verification for security
  • Cost: client's callback endpoint must be reachable and reliable — added operational burden on them

When to use

✓
Use polling for short-lived jobs (seconds to a couple minutes) or when the client is a browser that can't easily expose a webhook endpoint.
✓
Use webhooks for server-to-server integrations with long jobs (video processing, large exports) where polling would waste significant resources.
⚠
Watch for duplicate processing on retry — if the client times out and resubmits, an idempotency key must make the second submission a no-op, not a second job.

Interview moves

  • 01
    "This work exceeds a normal request timeout, so I'd accept it async — return 202 and a job_id immediately, process in the background" — names the trigger and the pattern together
  • 02
    "I'd require an idempotency key on submission, so a client retry after a timeout doesn't create a duplicate job" — the single most important correctness detail in this pattern
  • 03
    "For this audience I'd offer a webhook, since these are server clients that can expose a callback — polling would just waste resources on both sides" — justifies the choice instead of picking one by default
  • 04
    "Webhook deliveries can fail, so I'd add retry with backoff and let the client fall back to polling the status endpoint if delivery fails repeatedly" — shows you've thought about webhook failure modes
  • 05
    "I'd sign webhook payloads so the receiver can verify the callback actually came from us" — addresses webhook security before it's asked

Common follow-up questions

Why are idempotency keys mandatory here?
Any async system has retries — network blips cause clients to resubmit. Without a dedup key, that resubmission becomes a second, redundant (or harmful) job execution.
How do you implement an idempotency key server-side?
Store submitted keys with a TTL in a fast store (Redis/DB unique constraint). On a new submission, check if the key exists — if so, return the original job's result instead of enqueueing again.
How do you handle a worker crashing mid-job?
Use visibility timeouts (SQS) so the message becomes available again if not acknowledged within a window — another worker picks it up. Design job execution to be safely re-runnable (idempotent).
How do you avoid polling becoming a load problem?
Use exponential backoff on the client (start at 1s, cap at 30s), and serve status reads from a cache, not the primary job DB, since they're high-volume reads of low-churn data.
How do you guarantee webhook delivery ordering?
Generally you don't — include a sequence number or timestamp in the payload and let the receiver reorder/dedupe, rather than relying on delivery order.
What if the client never polls and never receives a webhook successfully?
Keep job results in the status store for a retention window (e.g. 7-30 days) so a late client request can still retrieve the outcome instead of losing it.
