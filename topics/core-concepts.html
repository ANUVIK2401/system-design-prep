Core Concepts

CAP theorem, consistency models, latency numbers, and back-of-envelope math — the vocabulary of every system design interview.

15 min read High-density
Say these with confidence
  • CAP: you can only guarantee 2 of 3 — and during a partition, you choose CP or AP
  • Eventual consistency: reads may return stale data; all replicas converge given time
  • 1ms L1 cache → 100ms cross-datacenter — a 5-order-of-magnitude difference
  • 10M DAU × 1 write/day ÷ 86,400 = ~115 writes/sec — memorize this formula
  • A single machine handles ~10k concurrent connections; a single DB ~10k QPS

Mental model

Core concepts are the grammar of distributed systems. You don't need to quote papers — you need to use the words correctly when tradeoffs come up in your design.

CAP theorem

PropertyMeaning
Consistency (C)Every read returns the most recent write or an error — no stale reads
Availability (A)Every request gets a non-error response — even if the data might be stale
Partition tolerance (P)The system works even when network splits some nodes from others
In production, P is non-negotiable (networks always partition). So the real choice is: during a partition, do you return stale data (AP) or return an error (CP)?
ChoiceBehavior during partitionUse when
CP (consistent)Returns error rather than stale data — blocks until consistentFinancial systems, inventory, anything where stale = wrong decision
AP (available)Returns stale data rather than error — always respondsSocial feeds, caches, DNS, search indexes

Consistency models (spectrum)

ModelGuaranteeCost
Strong consistencyEvery read sees the latest write — alwaysHigh latency, low availability during failures
Read-your-writesYou always see your own writes immediatelyRoute your reads to primary DB or use sticky sessions
Monotonic readsOnce you see a value, you never see older valuesUser never sees a post "disappear"
Eventual consistencyAll replicas converge to same value — given enough timeLow latency, high availability; stale reads possible
Causal consistencyCausally related operations appear in orderHarder to implement; middle ground between strong and eventual

Scaling concepts

Vertical scaling (scale up)
  • Bigger machine — more CPU, RAM, disk
  • Simple: no code changes needed
  • Hard limit: largest machine available
  • Single point of failure
  • Good up to ~100k QPS; use it first
Horizontal scaling (scale out)
  • More machines behind a load balancer
  • Complex: stateless services required
  • No hard limit — add nodes as needed
  • Higher operational overhead
  • Required beyond ~100k QPS or HA
ConceptWhat it means
ShardingSplit DB rows across nodes by a key (user_id, region). Avoids hot spots. Complicates joins.
ReplicationCopy data to multiple nodes. Primary handles writes; replicas handle reads.
Consistent hashingDistribute keys across nodes such that adding/removing a node minimizes remapping
Load balancingDistribute traffic across servers — L4 (TCP) or L7 (HTTP, content-aware)
Rate limitingCap requests per client per time window — token bucket or sliding window
Circuit breakerFail fast if a dependency is degraded — prevents cascading failures
IdempotencySame request, same result if repeated — critical for retries in distributed systems

    "Given eventual consistency here, the read might be ~100ms stale — that's acceptable for a social feed but not for a payment" — shows you connect consistency models to real scenarios
  • 02

  • 03

  • 04
    "This is an AP system — during a partition we serve stale data rather than error out" — stating CAP preference explicitly shows distributed systems fluency

Common follow-up questions

When would you choose strong consistency over availability?
Any time a stale read causes a wrong business decision: inventory (overselling), payments (double charge), seat reservations. For everything else — social, search, analytics — eventual is fine.
What is PACELC and how does it extend CAP?
PACELC says: even when there's no partition (E), you still trade off latency (L) vs consistency (C). It captures the everyday tradeoff that CAP only covers during failure. Mention it if the interviewer is probing deep.
How do you handle hotspot keys in a sharded system?
Add a random suffix to the key to distribute across shards (shard spreading), then aggregate reads. Or use consistent hashing with virtual nodes to rebalance load automatically.
What's the difference between a cache and a replica?
A replica is a consistent copy of the DB — writes go there, lag is bounded. A cache is a best-effort snapshot — eviction can make it stale, and writes don't necessarily go through it.
Why does consistent hashing matter for distributed caches?
With naive modulo hashing, adding 1 node remaps ~N/nodes keys — causing a cache stampede. Consistent hashing remaps only ~K/n keys (K = total keys, n = nodes added), minimizing disruption.
What's the difference between rate limiting algorithms?
Token bucket: smooth traffic, allows bursts up to bucket size. Sliding window: strict — counts requests in the last N seconds. Fixed window: simple but allows 2x burst at window boundaries. Token bucket is the most common production choice.
