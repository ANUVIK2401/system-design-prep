Vector Databases

Approximate nearest neighbor search over embeddings — the retrieval engine behind RAG.

15-20 min Medium
Say this in the first 2 minutes
  • Embeddings are dense vectors (often 768-1536 dimensions) where semantic similarity becomes geometric distance — "nearby vectors mean similar meaning"
  • Exact nearest-neighbor search is O(n) per query and doesn't scale — ANN trades a small recall loss for orders-of-magnitude speedup
  • HNSW is the dominant production ANN algorithm — a multi-layer graph where search descends from sparse long-range edges to dense local edges
  • RAG exists because LLMs have a fixed context window and no built-in access to your private/current data — retrieval injects relevant facts at query time instead of fine-tuning
  • The recall/latency knob is real and interview-relevant — wider search (higher ef_search) means better recall at the cost of latency, tune per use case

Mental model

A vector database is a library where books aren't sorted by title or author, but placed physically close to other books about similar topics — you don't search by exact match, you walk to the right neighborhood on the shelf and grab everything nearby.

RAG pipeline with HNSW retrieval

Same embedding model must encode both docs and queries into one shared vector space
HNSW index — built once offline, queried with low latency online
Retrieved chunks ground the LLM in real data instead of pure model memory
  1. Offline: chunk source documents, embed each chunk, and insert vectors into an HNSW index (built incrementally, layer by layer).
  2. Online: embed the incoming user query with the same model so it lands in the same vector space as the indexed documents.
  3. HNSW search starts at the sparse top layer, takes long greedy hops toward the query vector, then drops down a layer and refines — like coarse-to-fine search.
  4. The bottom layer is dense and local — final hops here produce the approximate top-K nearest neighbors.
  5. Retrieved chunks are concatenated with the user's question into a single prompt, handed to the LLM to generate a grounded answer.

MemoryJust the raw vectorsVectors + graph edges — typically 1.5-2x raw vector size

Tradeoffs

Wins

  • Sub-linear query time at massive scale — the only way ANN search stays fast past millions of vectors
  • ef_search gives a tunable recall/latency knob per query without rebuilding the index
  • Purpose-built vector DBs (Pinecone, Weaviate, Qdrant) handle sharding, filtering, and hybrid search out of the box
  • pgvector lets you keep vectors next to relational data when you don't need dedicated-DB scale

Costs

  • Approximate by design — a small fraction of true nearest neighbors can be missed
  • HNSW graph memory overhead on top of raw vectors adds real infra cost at billions of vectors
  • Index build/insert is slower than a flat array — not ideal for extremely high-churn data
  • A dedicated vector DB is one more system to operate, secure, and keep in sync with the source of truth

When to use

✓
Use a dedicated vector DB when you're doing RAG, semantic search, or recommendation at scale (millions+ of vectors) and need fast, tunable ANN with metadata filtering.
⚠
Watch out for stale embeddings — if source documents change, you must re-embed and re-index, or RAG answers will be grounded in outdated context.
✗
Avoid a dedicated vector DB for small datasets (tens of thousands of vectors) — pgvector inside your existing Postgres, or even brute-force exact kNN, is simpler and fast enough.

Common follow-up questions

What happens if you increase ef_search?
HNSW explores more candidates per layer, increasing recall toward exact kNN at the cost of higher query latency — it's a runtime knob, no reindex needed.
How do you keep a RAG index fresh when source documents change?
Re-embed changed documents and upsert into the index; for high-churn data, batch updates on a schedule rather than per-write to avoid index thrashing.
Why not just fine-tune the LLM on your data instead of RAG?
Fine-tuning is expensive, slow to update, and doesn't cite sources. RAG updates instantly by re-indexing and lets you point to the exact retrieved source for grounding.
How do you combine vector search with traditional filters (e.g. "only docs from 2024")?
Hybrid search — most vector DBs support metadata filtering alongside ANN search, pruning the candidate graph traversal to only matching metadata.
What's the failure mode when embeddings are low quality?
Semantically similar text won't cluster correctly, so retrieval returns irrelevant chunks — garbage in, garbage out; embedding model choice matters as much as the index.
pgvector vs Pinecone — how do you decide?
pgvector for smaller scale or when you want vectors colocated with relational data and one less system to operate. Pinecone/Weaviate/Qdrant once you need dedicated ANN performance at large scale with built-in sharding.
