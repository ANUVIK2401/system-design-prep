Proximity Search

How "find nearby" actually works at scale — geohashing, quadtrees, and R-trees.

15-20 min Medium
Say this in the first 2 minutes
  • Naive lat/lng range queries don't work — a degree of longitude shrinks toward the poles, so a fixed box isn't a fixed physical radius
  • Geohashing turns 2D coordinates into a 1D string where shared prefix ≈ spatial proximity, so a normal B-tree index can do spatial range queries
  • Geohash has a boundary problem — two points 10 meters apart across a cell edge can share zero prefix characters, so you always query neighboring cells too
  • Quadtrees adapt to data density (fine-grained in cities, coarse in rural areas); R-trees group nearby objects into bounding boxes and are what production geo-databases like PostGIS actually use
  • Always post-filter candidates with exact haversine distance — the index gets you "probably close," not "definitely within 5km"

Mental model

Geohashing is like a postal address read left to right — "USA, California, San Francisco, Mission St" — each added character narrows the area, and two addresses sharing a long prefix are probably close together, even though the system has no idea what "distance" actually means.

Core concepts

TermDefinition
GeohashBase32 string encoding lat/lng; each extra char adds precision and shrinks the cell
Prefix proximityTwo geohashes sharing a long prefix are usually near each other (not guaranteed)
QuadtreeRecursively splits a square into 4 quadrants until each leaf holds few enough points
R-treeTree of nested minimum bounding rectangles (MBRs) grouping nearby geometries
GiST indexPostgres's generalized search tree — the index type PostGIS builds R-trees on top of
Boundary problemPoints near a cell edge can land in different cells with totally different prefixes
Haversine distanceExact great-circle distance formula used to post-filter index candidates
A 1° longitude box is ~111km wide at the equator but only ~78km wide at 45° latitude and 0km at the poles. Filtering with raw WHERE lng BETWEEN x AND y gives wildly inconsistent physical radii by latitude — this is why naive range queries fail.

Geohash cell subdivision

Each level adds a base32 char and divides the cell again
Boundary problem — adjacent points can land in sibling cells with different prefixes
  1. Encode lat/lng by interleaving binary subdivisions of latitude and longitude ranges into base32 characters.
  2. Each additional character roughly halves the cell's width and height — 1 char ≈ 5,000km, 6 chars ≈ 150m, 9 chars ≈ 5m.
  3. To find nearby points, query rows whose geohash starts with the user's prefix using a normal indexed LIKE 'prefix%' or range scan.
  4. Also query the 8 neighboring cells (geohash has a standard neighbor-calculation algorithm) to catch points split across a boundary.
  5. Post-filter every candidate with exact haversine distance and sort — the index narrows the field, it doesn't give exact distance.

Geohash vs quadtree vs R-tree

StructurePrecision controlQuery complexityWrite costReal-world usage
GeohashFixed steps via string lengthSimple prefix/range scan, needs neighbor lookupsCheap — just encode and insertRedis GEOADD/GEOSEARCH, Elasticsearch geo_point, Firestore
QuadtreeAdapts to point density automaticallyTree traversal, naturally handles uneven densityModerate — may trigger node splitsMongoDB 2d indexes (legacy), game engines, mapping tools
R-treeBounding boxes fit actual geometry, not a gridBest query performance for ranges/polygons, not just pointsHigher — box updates can cascade up the treePostGIS (via GiST), most production spatial databases
Redis specifically chose geohash because it can encode 2D location into a single 52-bit integer and reuse its existing sorted-set (ZSET) infrastructure — no new data structure needed. PostGIS chose R-trees/GiST because it already needed to index polygons and lines, not just points, where bounding boxes generalize better than a fixed grid.

Tradeoffs

Wins

  • Geohash reuses ordinary B-tree/string indexes — no custom spatial extension required
  • Quadtrees self-adjust to density — dense cities get fine cells, empty deserts stay coarse
  • R-trees give the tightest, most accurate candidate sets and support arbitrary shapes (polygons, routes)
  • All three turn an O(n) full scan into a narrow indexed lookup

Costs

  • Geohash boundary problem means you must always check neighbor cells, adding query complexity
  • Quadtree rebalancing on skewed inserts can be expensive and isn't trivially indexable in a normal RDBMS
  • R-trees are the most complex to implement from scratch — you'll use a library/extension (PostGIS), not hand-roll one
  • None of these give exact distance — every approach needs a post-filter pass

When to use which

✓
Use geohash when you're building on Redis/Elasticsearch already, need "nearby" search with simple infra, and points (not polygons) are enough.
⚠
Watch out for uneven data density with a fixed-precision geohash — choose precision carefully, or pair with quadtree-style adaptive depth if density varies wildly (rural vs. downtown).
✗
Avoid hand-rolling R-trees yourself — use PostGIS, MongoDB geospatial indexes, or a managed geo-search service (Elasticsearch, Algolia) instead of building the tree from scratch.

Common follow-up questions

How do you pick geohash precision (string length)?
Match it to your typical search radius — 6 chars (~150m cells) for "restaurants nearby," 5 chars (~5km) for "delivery zone." Shorter strings mean coarser cells and fewer rows scanned but more post-filtering.
Why not just use a hash map keyed by rounded lat/lng?
That's effectively a fixed-grid geohash with no prefix structure — you lose the ability to do range queries at multiple zoom levels with one index.
How does Redis's GEO command work internally?
It encodes lat/lng into a 52-bit geohash-like integer and stores it as the score in a sorted set (ZSET), reusing ZRANGEBYSCORE for proximity queries.
How would you shard a geo-indexed dataset across multiple DB nodes?
Shard by geohash prefix so physically nearby data lives on the same shard — keeps "nearby" queries from fanning out across every node.
What's the actual distance error from using geohash prefix match alone?
Unbounded in the worst case near a cell edge — two points meters apart can share zero prefix. That's exactly why neighbor-cell lookups plus an exact-distance post-filter are mandatory, not optional.
When would a quadtree beat a geohash?
When data density is extremely uneven (all points in cities, none elsewhere) — a quadtree naturally goes deep only where points cluster, while geohash precision is uniform everywhere by default.
