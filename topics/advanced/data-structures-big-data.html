Probabilistic Data Structures for Big Data

Trading a small, known error rate for huge memory savings — Bloom filters, HyperLogLog, Count-Min Sketch.

15-20 min Medium
Say this in the first 2 minutes
  • These exist because exact tracking at billion-event scale needs infeasible memory — a real hash set of 1B unique IDs costs tens of GB; HyperLogLog gets the same answer within ~0.81% error in ~12KB
  • Each structure answers a different question: Bloom = "have I seen this?", HyperLogLog = "how many unique things?", Count-Min Sketch = "how many times have I seen this?"
  • Bloom filters never produce false negatives, only false positives — "definitely not present" is always trustworthy, "probably present" needs a real lookup to confirm
  • All three trade a small, bounded error for O(1) fixed memory instead of memory that grows with input size
  • Name the concrete production use case immediately — Bloom filters in Cassandra/LSM-trees to skip disk reads, HLL in Redis for unique visitor counts, Count-Min Sketch for trending/heavy-hitter detection

Mental model

These structures are like a bouncer with a bad memory for faces but a great memory for vibes — they can't tell you exactly who's inside, but they can say "that person has definitely never been here" with total confidence, using a notebook that never gets any bigger no matter how many people walk through the door.

All three are write-once-read-many approximations — none of them support deletion cleanly (standard Bloom filters can't remove an element without risking false negatives for other elements that hash to the same bits).

Bloom filter — bit array + multiple hashes

Insert — set k bit positions to 1 per key, via k independent hash functions
Query — if ANY bit is 0, the element is definitely absent; if all are 1, it's only probably present
  1. Allocate a fixed-size bit array of m bits, all initialized to 0, plus k independent hash functions.
  2. To insert a key, run it through all k hash functions and set each resulting bit position to 1.
  3. To query a key, run it through the same k hash functions — if any bit is 0, the key was definitely never inserted (no false negatives).
  4. If all k bits are 1, the key was probably inserted — but a collision from other keys hashing to the same bits can produce a false positive.
  5. Tune m (array size) and k (hash count) against expected element count to hit a target false-positive rate — more bits and more hashes lower the false-positive rate at the cost of memory and CPU.

HyperLogLog and Count-Min Sketch — the other two

HyperLogLog

  • Hashes each element, then tracks the longest run of leading zero bits seen across many buckets
  • Longer zero-runs are exponentially rarer, so the longest observed run estimates how many distinct elements were hashed
  • Redis's implementation uses ~12KB regardless of whether you've counted 1,000 or 1 billion unique items, at ~0.81% standard error
  • Supports a merge operation — combine HLLs from multiple shards/days into one without re-scanning raw data

Count-Min Sketch

  • A small 2D array (width x depth); each row uses a different hash function
  • Incrementing a key bumps one counter per row; querying a key takes the MINIMUM across all rows it maps to
  • Taking the minimum cancels out most hash collisions — but the error is always an overestimate, never an undercount
  • Used to find heavy hitters (top-K trending items) in an unbounded stream without storing a counter per unique item

Tradeoffs

Wins

  • Fixed, predictable memory regardless of input cardinality — critical at billion-event scale
  • O(1) insert and query time, independent of how many elements have been added
  • Mergeable across shards/time windows without touching raw data again
  • Error bounds are mathematically known in advance, not a surprise in production

Costs

  • Approximate by design — wrong tool if you need an exact, auditable count (e.g. billing)
  • Standard Bloom filters can't delete elements without risking false negatives elsewhere
  • Error compounds with poor hash function choice or under-sized structures — needs tuning against expected scale
  • Debugging "why did this say present when it shouldn't be" is harder than with exact structures

When to use which

✓
Use a Bloom filter as a pre-check before an expensive lookup — e.g. "does this key possibly exist on disk?" before hitting an SSTable, avoiding most unnecessary disk reads.
✓
Use HyperLogLog for "how many unique X" dashboards (DAU, unique IPs) where ~1% error is completely acceptable and exact counts would need gigabytes of hash sets.
✗
Avoid all three for anything requiring exact, auditable numbers — financial transaction counts, inventory levels, or anywhere a false positive/overestimate has real consequences.

Common follow-up questions

Can you delete an item from a Bloom filter?
Not from a standard one — clearing a bit could create false negatives for other keys sharing that bit. A Counting Bloom Filter (counters instead of bits) supports deletion at the cost of more memory.
How does Redis implement HyperLogLog?
PFADD/PFCOUNT/PFMERGE commands store a ~12KB sparse-then-dense encoded structure per key, giving unique counts at ~0.81% error with constant memory.
Why does Count-Min Sketch only overestimate, never undercount?
Taking the minimum across rows means a true count can only be inflated by hash collisions with other keys in some rows — it can never be deflated below the real value.
How do you size a Bloom filter for a target false-positive rate?
Use the standard formula relating bit array size (m), expected element count (n), and hash count (k) — more bits and an optimal k lower the false-positive probability; tune against expected n upfront.
Why not just use a real HashSet if you have enough memory?
You can — these structures only earn their complexity at scale where exact memory cost becomes the bottleneck (billions of keys) or where mergeability across distributed nodes matters.
What's a concrete production example of each?
Bloom: Cassandra/RocksDB skip-disk-read filters. HyperLogLog: Redis unique visitor counters, Google's original use for ad click dedup. Count-Min Sketch: streaming top-K trending detection (e.g. Twitter-scale hashtag trends).
