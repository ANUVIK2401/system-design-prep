Time-Series Databases

Write-heavy, time-ordered data — downsampling, retention, and why metrics need their own database.

15-20 min Medium
Say this in the first 2 minutes
  • Time-series workloads are append-mostly, naturally ordered by time, and almost never updated after write — that shape enables compression a general-purpose OLTP DB can't exploit
  • Row-per-event in a regular relational DB falls over at scale — index bloat and no time-aware compression mean storage and query cost grow linearly with every single data point
  • Downsampling rolls up old high-resolution data into coarser aggregates over time — keep raw 1s data for days, 1min rollups for months, 1hr rollups forever
  • Retention policies auto-expire raw data after a TTL, deleting old high-resolution rows once they've been downsampled or are no longer useful
  • Cardinality explosion is the classic gotcha — too many unique tag/label combinations (e.g. tagging every request with a unique user ID) blows up a TSDB's index and kills performance

Mental model

A time-series database is like a security camera system that automatically degrades old footage from full HD to low-res thumbnails the further back in time you go — you keep perfect detail for "what just happened" and only a rough shape for "what happened last year," because nobody needs frame-by-frame detail from six months ago.

Core concepts

TermDefinition
Time-series dataTimestamped, append-mostly data naturally ordered by time (metrics, sensor readings, logs)
DownsamplingRolling up high-resolution old data into lower-resolution aggregates to save storage
Retention policyAutomatic expiry/deletion of raw data after a configured TTL
Delta encodingStoring the difference between consecutive values instead of the full value — exploits slow-changing metrics
Delta-of-deltaStoring the change in the rate of change — exploits near-constant time intervals between points
CardinalityNumber of unique tag/label combinations — high cardinality blows up index size
Tag/labelIndexed metadata on a metric (host, region, endpoint) used for filtering and grouping
Tagging every metric with a unique request ID or user ID turns a metric with a few hundred reasonable time series into millions — this is cardinality explosion, and it's the single most common way teams accidentally take down their own TSDB.

Downsampling and retention tiers

Recent data — full resolution for incident debugging
Mid-term — downsampled for trend analysis
Long-term — coarse rollups kept indefinitely at low cost
  1. Writes land at full resolution (e.g. 1s) in the hot tier — optimized for fast, append-only ingest.
  2. A background process periodically aggregates older raw points into coarser rollups (avg/min/max/p99 per minute, then per hour).
  3. Retention policies auto-delete each tier once it ages past its TTL and has already been captured in the next coarser tier.
  4. Queries automatically route to the appropriate tier — "last 1 hour" hits raw data, "last 6 months" hits hourly rollups — keeping query cost bounded regardless of time range.
  5. Delta and delta-of-delta encoding compress each tier further by storing changes instead of absolute values, since consecutive points and intervals barely change.

Tradeoffs

Wins

  • Massive compression ratios from exploiting the time-ordered, slow-changing nature of metrics
  • Built-in downsampling and retention remove a whole category of hand-rolled cron jobs
  • Time-bucketed queries (e.g. Prometheus PromQL) are first-class, not bolted on
  • Designed for the actual access pattern — append now, query recent at full detail, query old at low detail

Costs

  • Another specialized system to operate alongside your primary OLTP database
  • Cardinality explosion from over-tagging can degrade performance badly and is easy to introduce by accident
  • Not designed for arbitrary relational joins or transactional updates — wrong tool for general CRUD
  • Downsampled historical data loses precision — can't recover exact raw values once rolled up and the raw tier expires

When to use

✓
Use a TSDB for metrics, observability, IoT sensor data, or any workload that's overwhelmingly append-heavy, timestamp-ordered, and queried in time ranges/aggregates.
⚠
Watch out for cardinality explosion — never tag a metric with unbounded-cardinality fields like user ID or request ID; keep tags to bounded sets like host, region, or status code.
✗
Avoid a TSDB for general CRUD workloads needing relational joins, transactional updates, or strong consistency across unrelated entities — that's still a job for a regular OLTP database.

Common follow-up questions

Why does Postgres row-per-event fall over at scale specifically?
No time-aware compression means storage grows linearly with raw events, and a single growing B-tree index degrades — TimescaleDB fixes this by automatically partitioning into time-based chunks.
What exactly is cardinality explosion and why does it hurt?
Each unique tag combination creates a separate indexed time series — millions of unique combinations (e.g. per-user tags) means millions of tiny series, blowing up index memory and query latency.
How do continuous aggregates / rollup policies work?
A background job periodically (or incrementally) computes aggregates like avg/p99 over fixed time buckets and stores them in a separate, much smaller rollup table or materialized view.
What's lost when you downsample?
Individual data points within a bucket — you can't recover the exact raw value once it's rolled into an aggregate and the raw tier's retention has expired.
How would you handle a sudden spike in write throughput (e.g. IoT fleet doubling)?
Shard by time + a bounded tag (e.g. device group), scale ingest nodes horizontally, and ensure the hot/raw tier's retention window is short enough to keep storage growth manageable.
When would you NOT use Prometheus for metrics?
When you need long-term, high-cardinality historical storage — Prometheus is built for short-term local retention; pair it with a remote-write long-term store (e.g. Thanos, Mimir) for that.
