Questions

Design a Metrics Monitoring & Alerting System

Tests whether you separate a write-scale time-series storage problem (millions of points/sec) from a latency-critical alerting problem instead of serving both off the same slow query path.

45-55 min interview Hard
Say this in the first 2 minutes
  • The core insight: this is a time-series data problem at extreme write scale — write throughput and query flexibility pull in opposite directions, which forces specialized TSDB storage instead of a general-purpose database
  • Key scale number to anchor on: 10M metric data points/sec across the fleet, which rules out row-oriented storage and demands columnar, compressed, downsampled layout
  • Primary tradeoff being probed: query freshness vs storage cost — high-resolution data is expensive to keep forever, so old data gets progressively downsampled
  • Component most candidates forget: alerting cannot share the same query path as dashboards — a <1 minute evaluation latency target means a separate low-latency streaming path is mandatory
  • Failure mode most candidates skip: cardinality explosion — one bad label (like user_id) can take down the whole TSDB index, not just one metric

Step 4 of 6 — Data Flow

Data flow

Ingestion (blue, fanning into an async downsampling path in amber) and dashboard queries (mint) are fully separate flows that only meet at the TSDB.

Agents + dashboard UI — entry points
Kafka + downsample worker — absorb bursts, compact off the hot path
TSDB — shared store, written by ingestion, read by queries
  1. Agent emits a batch of points — pushed (or scraped, pull-model) every 10-15s from every host/service
  2. Ingestion tier buffers the burst — Kafka decouples producer rate from TSDB write rate
  3. TSDB writes raw points — sharded by series_id, compressed columnar layout
  4. Downsample worker compacts in the background — rolls old high-resolution data into lower-resolution long-term tiers
  5. Dashboard queries the query service — which picks raw or rolled-up data depending on the requested range

Step 6 of 6 — Deep Dives

Deep dive: time-series storage + downsampling

Raw 10s-resolution data is compacted over time into progressively coarser tiers — this is what makes both storage cost and long-range query performance tractable.

  1. Agent writes at 10s resolution — every point kept exactly as emitted for the first 24 hours
  2. Compaction worker runs on a schedule — reads a window of raw points, aggregates (avg/min/max/p99), writes one rolled-up point
  3. 1-minute tier retained for 7 days — enough detail for "what happened yesterday," at 1/6th the storage
  4. 1-hour tier retained for 13 months — enough for long-range trend and capacity charts, at a fraction of raw storage cost
  5. Query auto-picks the tier — a 6-hour dashboard query hits raw data; a 6-month capacity chart hits the 1hr tier automatically, transparent to the user
This isn't just a cost optimization — querying 13 months of 10s-resolution points would also be too slow to render a dashboard interactively. Downsampling solves storage cost and query latency at the same time.

Deep dive: cardinality explosion

A single high-cardinality label turns one metric into millions of unique time series, overwhelming the TSDB's index — the classic way a monitoring system takes itself down.

  1. A label is added without thinking about its value space — user_id or request_id looks harmless per-metric, but each unique value is a brand-new time series
  2. Series count multiplies combinatorially — every existing label combination now multiplies by every distinct user_id ever seen
  3. The TSDB index grows unbounded — the index tracking which series exist, not the data itself, is what runs out of memory first
  4. Every query against that metric slows down — and because shards are shared, other unrelated metrics on the same shard degrade too
  5. Mitigation is enforced at ingestion, not after the fact — reject or drop series once a per-metric cardinality limit is hit, and alert the emitting team immediately
High-cardinality dimensions like user_id or request_id belong in logs or traces, which are built to index arbitrary high-cardinality fields — not in a metrics system, which is optimized for a bounded, slowly-changing set of series.

Deep dive: alerting pipeline separate from dashboards

Alert evaluation reads the ingestion stream directly as it arrives — it never queries the same TSDB path dashboards use, because that path is both too slow and contended by expensive analytical queries.

  1. Ingestion stream is tapped twice — once into the TSDB for storage, once directly into the alerting engine
  2. Alerting engine keeps rule state in memory — per series, tracking how long a threshold has been breached against the configured evaluation window
  3. Evaluation happens as data arrives — no polling interval to wait out, no query queue to sit behind
  4. Breach triggers immediately — once the window condition is met, notification fires without a query round-trip
  5. Dashboards remain free to run expensive queries — because they no longer share a path with the latency-critical alerting workload
Polling-based alerting has two independent failure modes: the poll interval itself adds latency that can blow past a <1 minute target, and an expensive ad-hoc dashboard query can starve the same query path an alert poll is waiting on — exactly when an incident is generating more dashboard traffic too.

Follow-up questions

How do you decide the right downsampling schedule?
Balance query patterns against cost: keep raw resolution only as long as engineers actually zoom into second-level detail, then step down tiers where storage savings outweigh lost precision.
What happens if the downsample worker falls behind?
Raw data is retained until compaction catches up — queries transparently fall back to scanning raw data for that window, slower but not lossy, with lag surfaced as its own monitored metric.
How would you enforce a cardinality limit without silently losing important data?
Reject only the excess series past the limit, keep the rest flowing, and immediately alert the emitting team with the exact label combination that tripped the limit.
Why not just run alert queries more frequently to hit the latency target?
Frequent polling of an analytical query path multiplies load on shared TSDB infrastructure and still adds interval-length latency — streaming evaluation removes both problems at once.
How do you scale the alerting engine independently from ingestion?
Partition rule evaluation by series_id shard, same key space as the TSDB, so alerting horizontal scale tracks ingestion scale without a shared bottleneck.
How would this design change for push (agent-initiated) vs pull (scrape) metric collection?
Push suits ephemeral or high-cardinality-risk sources needing immediate delivery; pull suits stable long-running services and centralizes scrape-interval and target discovery control.
