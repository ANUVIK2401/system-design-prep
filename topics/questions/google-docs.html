Questions

Design Google Docs

Tests whether you can reason about real-time collaborative editing as a convergence problem — naive last-write-wins destroys concurrent edits, so you need Operational Transformation or CRDTs to guarantee every client ends up in the same state.

50-60 min interview Hard
Say this in the first 2 minutes
  • The core insight: this is a convergence problem, not a storage problem — two users editing the same region of text simultaneously will produce conflicting ops, and last-write-wins silently corrupts one user's edit
  • You need either Operational Transformation (server transforms concurrent ops against each other) or CRDTs (ops carry enough metadata that math guarantees convergence without a central arbiter) — pick one and justify it
  • Key scale numbers to anchor on: 1B documents, 100M DAU, ~3 concurrent editors per active document, and a <500ms propagation target for one editor's keystroke to appear for everyone else
  • Component most candidates forget: the revision log + snapshot/compaction worker — without it, reconstructing a document replays every operation since creation, which gets slower forever
  • Failure mode most candidates skip: an OT server crashing mid-transform — clients must resend un-acked ops after reconnect via a sequence-number protocol, or edits silently vanish

Step 4 of 6 — Data Flow

Data flow

Op submission (blue) is the hot write path that must complete in under 500ms; doc load and presence (mint) are simpler reads that don't touch the OT transform logic.

Clients — local OT engine applies ops optimistically
OT Server — the hot-path convergence logic
Revision log + snapshots — durable, replayable history
  1. Client applies the op locally — optimistic update, no waiting on the server, so typing feels instant
  2. Op is sent over the open WebSocket — tagged with the client's base revision and local sequence number
  3. OT server transforms it — against any concurrent ops it has already accepted since that base revision
  4. Transformed op is appended to the revision log — durable, ordered by server-assigned sequence number
  5. Transformed op is broadcast to every other connected client — each applies it locally, all converge to the same state

Step 6 of 6 — Deep Dives

Deep dive: Operational Transformation — the hardest diagram on this site

Two users type near the same position at the same time, before either has seen the other's edit. The server must adjust — transform — one operation against the other so that no matter which order each client applies them locally, everyone converges to the exact same final string.

Setup: document is "Hello orld" (10 characters, missing the "w" in World). User A inserts "W" at position 6. User B, at the same instant, inserts "!" at position 10 (end of string). Neither has seen the other's op yet — both applied their own op locally and sent it to the server optimistically.
The transform function, concretely: op_A = insert("W", pos=6), op_B = insert("!", pos=10). Since A's insert happens before B's position in the string, B's operation needs its position shifted right by the length of A's inserted text: op_B' = transform(op_B, op_A) = insert("!", pos=11). Meanwhile A's own operation is untouched relative to B, since B inserted after A's position: op_A' = transform(op_A, op_B) = insert("W", pos=6).
  1. Both clients apply their own op immediately and optimistically — this is what makes typing feel instant even before the server responds
  2. Both ops are sent to the server tagged with their base revision (40) — the revision the client had last synced against
  3. The server picks an arrival order — say op_A arrives first, is applied cleanly, doc moves to rev 41
  4. op_B references a now-stale base revision — the server transforms it: op_B' = transform(op_B, op_A), shifting its position to account for op_A already being applied
  5. The server also computes op_A' = transform(op_A, op_B) — needed so Client B, which already applied op_B locally, can correctly apply A's change on top
  6. Server broadcasts the transformed complement to each client — Client A receives op_B' (not raw op_B), Client B receives op_A' (not raw op_A)
  7. Both clients converge to the identical final document — despite applying operations in opposite local orders, because the transform function guarantees it mathematically
This is the entire reason naive last-write-wins fails: if the server just applied whichever op arrived second and discarded position awareness, op_B's "!" would land at raw position 10 in a document that now has an extra "W" — landing one character too early, silently corrupting the insert. The transform step is what prevents this.

Deep dive: CRDT vs OT

OT depends on a central server to pick an order and transform. CRDTs bake enough metadata into each operation that any two replicas converge mathematically, regardless of order or whether a server is even involved.

OT server — mandatory central authority that serializes and transforms
CRDT clients — can sync with any peer, in any order, and still converge
DimensionOperational TransformationCRDT
Ordering authorityCentral server required to pick order + transformNone needed — metadata makes order irrelevant
Offline / peer-to-peer supportWeak — needs a round trip to the server to resolve conflictsStrong — two peers can sync directly, whenever they reconnect
Metadata overheadLow — position + text, transform logic lives server-sideHigher — every character/op carries a unique fractional or sequence ID
Implementation complexityTransform functions are notoriously hard to get exactly right for all op-type pairsConvergence proof lives in the data structure, less bespoke logic per op-type pair
  1. OT: server receives concurrent ops — picks an arrival order and computes a transform for every out-of-order pair
  2. CRDT: no server decision needed — each character/op is tagged with a unique identifier (e.g. a fractional position between neighbors) at creation time
  3. CRDT convergence is structural — merging two replicas' op sets, in any order, deterministically produces the same result because the identifiers form a total order on their own
  4. Tradeoff in practice — Google Docs uses a server-centered OT-style model since it already assumes an always-connected client and a central service; CRDTs shine when offline-first or fully peer-to-peer sync matters more than metadata size

Deep dive: revision history + snapshots

The operation log grows forever. Reconstructing a document at any point in time uses the nearest prior snapshot plus replaying only the ops after it — never replaying from the beginning.

  1. Every accepted operation is appended to the log — durable, ordered by server sequence number, never mutated
  2. Every N operations (e.g. 500), the snapshot worker materializes a full document body — stored alongside the sequence number it represents
  3. To reconstruct the document at any revision — find the nearest snapshot at or before that revision, then replay only the ops between the snapshot and the target
  4. Old ops before the oldest retained snapshot can be safely archived or trimmed — nothing downstream needs to replay past a snapshot boundary
Why not always replay from op #0? Because replay cost is O(ops since creation) — on a document that's been live for years with tens of thousands of edits, every single page load or history view would get slower and slower forever. Snapshotting bounds replay cost to O(ops since the nearest snapshot), which is a controllable, tunable constant.
Snapshot frequency is the dial: snapshotting every 100 ops means faster reconstruction but more storage and worker overhead; every 5,000 ops is cheap to store but means longer replay chains. Hot documents (high edit velocity) should get more frequent snapshots than a document nobody has touched in a year.

Follow-up questions

Why not just use a global lock so only one person can edit at a time?
It defeats the entire point of collaborative editing — users expect simultaneous typing, and a lock would serialize what should feel real-time and concurrent.
What happens if the OT server itself becomes a single point of failure for a very popular document?
Its state is minimal and reconstructible from the revision log, so a replacement instance can rebuild in-memory state from the last known sequence number and resume.
How do you handle formatting operations (bold, font size) alongside text insert/delete?
Each formatting change is its own operation type with its own transform rules against every other op type — the transform function grows combinatorially, which is why CRDTs are often preferred as op-type variety increases.
Why doesn't Google Docs use a fully peer-to-peer CRDT model?
It already assumes an always-connected client talking to a central service for storage, permissions, and history — a central OT server fits that model with lower per-op metadata overhead.
How would you support a user editing offline for an extended period?
That's explicitly out of scope for this design — a true offline-first engine would push further toward CRDTs, since OT's server-dependent transform doesn't handle long offline divergence well.
How do you keep revision history storage from growing without bound?
Archive or delete raw operations older than the oldest snapshot you intend to keep retrievable — anything before that boundary is never needed for replay again.
