Step 4 of 6 — Data Flow

Data flow

A new message triggers a write/batch path (blue write, amber async batching) that queues the request for GPU inference, and a read path (mint) that streams generated tokens back as they're produced.

Client — sends message, renders streamed tokens
Conversation DB — durable message history
Scheduler + GPU worker — batching and generation, off the synchronous request path
  1. Client sends the message, opens a stream — POST with an SSE connection kept open for the reply
  2. Moderation filters the input — blocks disallowed content before it reaches the model
  3. Conversation DB records the turn and assembles context — recent history is packed into the prompt
  4. Scheduler queues the request into a batch — joins an in-progress continuous batch rather than waiting for a dedicated GPU call
  5. GPU worker streams tokens as generated — each token is pushed to the client within milliseconds of being produced, not buffered until the end

Step 6 of 6 — Deep Dives

Deep dive: continuous batching for GPU efficiency

A naive one-request-per-GPU-call model wastes most of the GPU's capacity, because each forward pass can process many requests' tokens at once for nearly the same cost as one.

  1. New request arrives at the scheduler — it doesn't wait for a fresh batch to form
  2. It's inserted into the current in-flight batch — the next GPU forward pass processes it alongside whatever requests are already generating
  3. Each request advances one token per pass — requests are not forced to share the same total length
  4. A request that finishes generating leaves the batch immediately — its GPU slot is freed for the next arrival
  5. The batch composition changes continuously — this is what "continuous" or "dynamic" batching means, as opposed to static batches that wait to fill up before starting
The key mental model: a GPU forward pass costs roughly the same whether it's processing 1 request's token or 64 requests' tokens simultaneously — so the entire scheduling problem is about keeping that batch as full as possible, at all times, without forcing requests to align in length.

Deep dive: token streaming to the client

Full generation can take several seconds, but the user should see the first token in well under a second — streaming decouples perceived latency from total generation time.

  1. Client sends the prompt and opens a streaming connection — SSE or WebSocket, held open for the duration of generation
  2. GPU worker emits each token the instant it's produced — it does not wait to accumulate a full sentence or the full response
  3. Stream router forwards every token with no added buffering — its only job is to not introduce latency between worker and client
  4. Client renders tokens as they arrive — the user perceives the response as "already happening," not "loading"
  5. A final event signals completion — carries metadata like token usage so the client can close the stream cleanly
Time-to-first-token and total generation time are different metrics that get solved differently: TTFT is mostly about queue/batch wait time at the scheduler, while total time is bounded by tokens/sec throughput per request — optimizing one doesn't automatically fix the other.

Deep dive: conversation context management

Every new turn needs the prior conversation as context, but the model has a fixed context window — once a conversation grows past it, something has to give.

  1. New message arrives for an existing conversation — the system needs prior turns to respond coherently
  2. Prior turns are retrieved in order — read from the Conversation DB by conversation_id
  3. Context is assembled and checked against the token budget — the model's context window is a hard limit, not a soft target
  4. If it fits, the prompt is sent as-is — the common case for most conversations
  5. If it doesn't fit, a strategy kicks in — truncation drops oldest turns, summarization compresses them, or a sliding window keeps only the most recent K turns — each trades detail for staying in budget
Summarization has the best user experience — it approximates "the model still remembers" — but costs an extra inference call to produce the summary and risks compounding drift if summaries are repeatedly re-summarized across a very long conversation.

Follow-up questions

Why not just give every request its own dedicated GPU?
GPUs are expensive and mostly idle during a single request's memory-bound decode step — continuous batching lets one GPU serve many concurrent requests for nearly the cost of one.
How do you keep time-to-first-token low when the queue is deep?
Prioritize scheduling so new requests slot into the next batch cycle quickly, and shed or queue-signal load rather than letting wait times grow unbounded.
What happens to in-flight generations if a GPU worker needs to be taken down for a deploy?
Drain the worker — stop admitting new requests to its batch, let in-flight ones finish, then take it offline; abrupt kills should be reserved for actual crashes.
How would you reduce moderation's added latency on the critical path?
Run it as a small, fast classifier co-located with the gateway rather than a full model call, and run input and output checks in parallel with generation where safe.
How do you decide between truncation, summarization, and a sliding window for context overflow?
Sliding window is simplest and cheapest but loses early detail abruptly; summarization preserves gist at the cost of an extra inference call — most production systems use summarization once a conversation crosses a length threshold.
How would you scale this to support much longer responses (e.g. long-form documents)?
The streaming and batching model doesn't change — you'd mainly need higher per-request token budgets, longer-lived connections, and schedulers tolerant of more variance in request length within a batch.
