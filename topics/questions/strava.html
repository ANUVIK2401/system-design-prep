Design Strava

Tests whether you can turn noisy GPS streams into clean segment times and live leaderboards without falling into the trap of treating this as just another CRUD app.

30-40 min interview Medium
Say this in the first 2 minutes
  • The core insight: this is a write-heavy time-series + geospatial problem, not a social-feed problem — GPS ingestion and segment matching dominate the design
  • Key scale number to anchor on: a single activity can carry 3,000-10,000 GPS points, and raw tracks are append-only, never updated in place
  • Primary tradeoff being probed: exact segment matching (expensive, precise) vs approximate spatial-index matching (cheap, tolerant of GPS noise)
  • Component most candidates forget: a spatial index (geohash/R-tree) for segment matching — without it, matching degrades to a full scan per activity
  • Failure mode most candidates skip: leaderboard re-rank storms when a popular segment gets thousands of new times in a single morning commute window

Data flow

  2. Track written to S3, metadata to Postgres — both durable before responding
  3. 202 returned immediately — client sees "processing," never waits on matching
  4. Matching job enqueued — spatial index snaps the track against nearby known segments
  5. Segment efforts written to Cassandra, then ZADD into Redis — leaderboard updates within seconds, not on the critical path

Step 6 of 6 — Deep Dives

Deep dive: segment matching algorithm

Raw GPS points are noisy — matching snaps them onto a canonical segment path using a spatial index instead of comparing against every known segment.

  1. Bucket track points into geohash cells — bounds the search to segments physically near the route
  2. Query spatial index for candidate segments — returns a handful, not every segment in the system
  3. Run a curve-similarity check per candidate — Frechet distance (or similar) scores how closely the noisy track follows each candidate's canonical path
  4. Accept the best-fit match above a distance threshold — reject candidates that don't clear it, avoiding false positives on parallel roads
  5. Compute elapsed time from the snapped entry/exit points — not the raw first/last GPS timestamps, which can be off by a few seconds
GPS drift near tunnels, tree cover, or urban canyons is the real failure mode: a track can appear to cut through a building. The curve-similarity threshold exists specifically to reject a false match rather than award an impossible segment time.

Deep dive: leaderboard update flow

Once a segment effort is computed, the leaderboard must re-rank without ever making the athlete wait on it.

Cache — hot path
Database — source of truth
Async — queue / worker
  1. Matcher computes elapsed_time_s for the segment — happens seconds after upload, fully async
  2. Write to Cassandra first — durable source of truth, survives a Redis restart
  3. ZADD into the segment's Redis sorted set — O(log N), the set stays sorted automatically, no batch re-rank job needed
  4. ZRANK gives the athlete's live position — same O(log N) cost whether the set has 100 or 100,000 entries
  5. Conditional notification fires — only if the new rank cracks the top 10, avoiding notification spam on low-traffic segments

  • 02
    "I'll return a 202 the instant the track is durably stored, and run segment matching fully async — the athlete shouldn't wait seconds for their upload to complete" — separates the durable write from the expensive computation
  • 03
    "Raw GPS is noisy, so matching isn't a database lookup — it's a spatial index query followed by a curve-similarity check to reject false positives" — shows awareness that this isn't a simple point-lookup problem
  • 04
    "Redis sorted sets give me ZADD and ZRANK in O(log N), so I never need a separate batch re-rank job — the leaderboard is always sorted" — picks the right data structure for the access pattern, not just "cache it"
  • 05
    "A legendary climb near a major city could get thousands of efforts in one morning — I'd batch ZADDs per segment rather than writing on every single finish" — anticipates a hot-key problem unprompted
  • 06
    "Cassandra stays the source of truth and Redis is a rebuildable cache — if I lose the sorted set, I can regenerate it from segment_id partition scans" — states a concrete recovery story instead of assuming the cache never fails

Follow-up questions

How do you handle an athlete uploading a manually-edited or spoofed GPS file?
Apply plausibility checks (max speed for activity type, elevation consistency) at matching time and flag outliers for review rather than blocking the upload outright.
Why Cassandra for segment efforts instead of Postgres?
Writes are append-only and reads are "ordered scan within one partition" — Cassandra's partition model fits that natively without the row-lock contention a relational table would see at this write volume.
How would you support "following" leaderboards, not just all-time?
Maintain a separate Redis sorted set per (segment_id, athlete's follow-graph) is too expensive — instead fetch the all-time top N, filter client-side against the follow list, and fall back to a targeted ZSCORE lookup for followed athletes not in that page.
What happens if two athletes finish a segment with the exact same elapsed time?
Redis ZSET breaks ties lexicographically by member, which is arbitrary — add a secondary tiebreaker (earlier activity timestamp wins) enforced at write time before the ZADD.
How do you keep the spatial index from growing unbounded as new segments are created?
Geo-partition the index by region/city so a lookup only touches the local shard, and archive segments with near-zero traffic out of the hot index into cold storage.
How would you scale this to 10x the daily upload volume?
Shard the match queue and Cassandra by segment_id hash, add more matcher workers scaled on queue depth, and pre-bucket hot segments' Redis keys across multiple nodes.
