Questions

Design a Distributed Cache

Tests whether you see this as a consistent-hashing + replication problem, not a "store key/value in a hash map" problem — the hard part is remapping keys cheaply and surviving node failure without a thundering herd.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: distributing keys across nodes is a consistent-hashing problem — the goal is that adding/removing a node remaps the minimum number of keys, not the whole keyspace
  • Key scale number to anchor on: 10M ops/sec across the cluster at sub-millisecond p99 — this rules out any design that hits disk or crosses a network hop more than once per request
  • Primary tradeoff being probed: replication factor and consistency for replica reads vs raw throughput and memory cost
  • Component most candidates forget: a coordination/membership layer that tracks ring topology — without it, nodes disagree about who owns which key
  • Failure mode most candidates skip: thundering herd — when a hot key expires or a node dies, thousands of clients miss cache at once and hammer the database simultaneously

Step 4 of 6 — Data Flow

Data flow

Writes (blue) hash to the owning node then replicate to a standby; reads (mint) hash to the same owning node and return directly — no replication hop needed on the read path.

Client library — hashes locally, no proxy hop
Cache nodes — primary owns the shard, replica is standby
  1. Client hashes the key — onto the consistent-hashing ring using the locally cached topology
  2. Request routes directly to the owning node — no intermediate proxy or load balancer hop
  3. Write replicates to the shard's replica(s) — synchronously or asynchronously depending on the consistency dial
  4. Read returns straight from the primary's memory — sub-millisecond since it's a single RAM lookup
  5. On primary failure, client retries against the promoted replica — using the same ring lookup with updated topology

Step 6 of 6 — Deep Dives

Deep dive: consistent hashing ring

  • Nodes and keys hash onto the same circular keyspace.
  • Key owner = first node clockwise from the key.
  • Vnodes scatter each node's ownership so load stays even.
Node A's 3 vnodes — spread around the ring for even load
Node B's 3 vnodes
Node C's 3 vnodes
  1. Every physical node gets many vnodes — hashing the node ID with several salts places dozens-to-hundreds of points per node around the ring
  2. A key is hashed to a single ring position — the same hash function used for nodes, so keys and nodes share one keyspace
  3. Ownership is "next vnode clockwise" — walk the ring from the key's position until a vnode is found; that vnode's physical node owns the key
  4. Adding a node only remaps its adjacent arc — new vnodes slice into existing arcs, and only the keys that fell in the sliced portion move; everything else is untouched
  5. Removing a node merges its arc into the next vnode clockwise — again, only that node's keys remap, bounded to roughly 1/N of the keyspace, not all of it
This is the entire reason to use consistent hashing over a plain hash(key) % N scheme: with modulo hashing, changing N reshuffles almost every key. With a ring, changing node count only touches the keys owned by the adjacent arc — typically ~1/N of the keyspace.

Deep dive: eviction and invalidation

Two separate questions: which entries get evicted under memory pressure (LRU vs LFU), and how the cache stays consistent with the backing store (write-through vs write-back vs cache-aside).

Cache-asideApp checks cache, on miss reads DB and populates cache; writes go straight to DB and the cache entry is invalidatedSimplest pattern, but a brief inconsistency window exists between DB write and cache invalidation.
Write-throughEvery write goes to the cache first, which synchronously writes through to the DB before acknowledgingCache and DB never disagree, but every write pays DB latency — no write-speed benefit from the cache.
Write-backWrite lands in the cache and is acknowledged immediately; DB write happens asynchronously afterwardFastest writes, but a cache node failure before flush can lose data the client thinks is durable.
TTL expiry and eviction are independent triggers for the same outcome — a key can be evicted early under memory pressure even with time left on its TTL, or it can sit untouched at low memory and still expire exactly at its TTL. Both paths must trigger the same invalidation/refetch logic, or a candidate leaves consistency bugs at the boundary.

Deep dive: thundering herd protection

A hot key expiring or a node dying can turn thousands of simultaneous cache misses into thousands of simultaneous identical database queries — the database sees a spike that looks like an attack.

Clients — many simultaneous requesters for one key
Cache + coalescer — collapses concurrent misses into one fetch
Database — protected from duplicate simultaneous load
  1. First miss on a key registers an in-flight marker — request coalescing/single-flight: subsequent misses for the same key see the marker and wait instead of re-querying
  2. Only one request actually reaches the database — its result is cached and fanned out to every client that was waiting
  3. TTLs are staggered with jitter — instead of every replica of a dataset expiring at exactly T+60s, each gets T+60s ± random jitter so misses spread out over time
  4. Stale-while-revalidate serves the old value during refresh — a client gets the slightly-stale cached value immediately while one background request refreshes it, instead of everyone blocking on a fresh fetch
  5. Node failure triggers the same coalescing on the promoted replica — the replica that takes over a dead primary's shard applies identical single-flight logic rather than accepting every re-routed request as a fresh miss
Request coalescing must be scoped correctly — coalescing across different keys (not just the same key) either does nothing useful or, worse, serves the wrong value to the wrong request. The single-flight lock key must be the exact cache key.

  • 02

  • 04

  • 05

  • 06
    "The coordination layer tracking cluster topology can't be a single point of failure itself — I'd lean toward gossip over a single centralized coordinator at this scale" — extends the design's own failure-tolerance principle to its own control plane

Follow-up questions

How do virtual nodes actually improve load distribution?
Without vnodes, one unlucky hash placement can give a node a disproportionate arc; with dozens of vnodes per physical node, individual bad placements average out across the whole node.
How would you handle a client with a stale view of the ring topology?
The owning node detects it doesn't actually own the key, returns a redirect/hint with the correct node, and the client refreshes its topology copy from the coordination layer.
Why not just always use strong consistency for replica reads?
Strong consistency means every read waits on cross-replica confirmation, which conflicts directly with a sub-millisecond latency target — eventual consistency is the deliberate tradeoff for speed.
What happens if the coordination service itself goes down?
With gossip-based membership, nodes keep operating on their last-known topology and keep converging peer-to-peer; a centralized coordinator instead needs its own HA setup (e.g. a Raft-backed quorum).
How do you decide between write-through and write-back?
Write-through when losing an unflushed write is unacceptable; write-back when write latency matters more and a short data-loss window on cache-node crash is tolerable.
How would you detect a hot key before it becomes a problem?
Sample per-key access counts on each node with a rolling window; a key crossing a threshold relative to its shard's average triggers client-side caching or key-splitting proactively.
