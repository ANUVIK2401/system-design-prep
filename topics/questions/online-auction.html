Questions

Design an Online Auction

Tests whether you treat concurrent bidding as a compare-and-swap problem and auction close as a distributed coordination problem, not just another CRUD write.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: bid submission under contention is an optimistic-locking problem — two bids racing for the same auction must resolve via compare-and-swap, never last-write-wins
  • Key scale number to anchor on: a hot auction in its final 10 seconds can see 200-400 bid attempts/sec, almost all against the same row
  • Primary tradeoff being probed: strong consistency on the current highest bid vs low-latency real-time updates to every watching bidder
  • Component most candidates forget: the auction close worker — deciding the exact moment an auction ends and settling exactly once is a distributed coordination problem, not a cron job
  • Failure mode most candidates skip: "bid sniping" — a bid arriving in the last millisecond before close must not be dropped just because the clock ticked over mid-flight

Step 4 of 6 — Data Flow

Data flow

Read path (mint) — viewing an auction — and write path (blue) — submitting a bid — share the same Auction row but diverge everywhere else.

Read path — cache + CDN first
Write path — CAS on Postgres, no distributed lock needed
Broadcast — decoupled from the write's response
  1. Client submits a bid with expected_version — the version it last saw for this auction
  2. Bid Svc validates the increment — must exceed current high bid by the minimum step
  3. Conditional UPDATE against Postgres — WHERE version = expected_version, atomically bumps version
  4. Loser gets a 409 — client re-fetches the new current_high_bid and version, may re-bid immediately
  5. Winner's bid is published to Redis Pub/Sub — every WebSocket-connected watcher receives the new high bid within milliseconds

Step 6 of 6 — Deep Dives

Deep dive: bid race condition

Two bids arrive within milliseconds of each other for the same auction. Only one can win — a compare-and-swap on the version column decides, not arrival order at the network layer.

✓ Winning bid — CAS succeeds
✗ Losing bid — CAS fails, clean retry
This is why the client sends expected_version, not a raw amount comparison: comparing against a possibly-stale current_high_bid read on the client is itself a race. The database's atomic conditional UPDATE is the only safe arbiter.

Deep dive: auction state machine

Every transition is guarded and enforced in the same transaction as the triggering write. Dashed transitions are irreversible.

TransitionReversible?Guard condition
Scheduled → ActiveYes (seller can still edit before start)Current time reaches the listing's start_at
Active → ClosingNo — irreversibleClose worker acquires the auction's Redis lock exactly once past ends_at
Closing → Closed/SoldNo — terminalHighest accepted bid exists and meets reserve price
Closing → Closed/UnsoldNo — terminalNo bids, or highest bid below reserve
Bid sniping is the classic edge case: a bid that starts processing at t-5ms before ends_at must still be evaluated against the correct pre-close version. The close worker must only flip Active → Closing after acquiring the row lock, so an in-flight CAS either completes first or is correctly rejected — never silently dropped.

Interview moves

  • 01
    "Bid submission under contention is fundamentally a compare-and-swap problem — I'll use a version column so only one of two racing bids can win" — states the core insight before being asked
  • 02
    "The client shouldn't compare its own view of the current price against a new bid — that comparison itself can be stale. The conditional UPDATE in Postgres is the only safe arbiter" — shows understanding of why CAS beats client-side checks
  • 03
    "Auction close needs a distributed lock per auction_id so two worker instances can never both settle the same auction" — names the exact mechanism for exactly-once coordination
  • 04
    "I'd decouple the real-time bid broadcast from the bid write itself — Redis pub/sub fans out to watchers after the CAS succeeds, never blocking the write" — demonstrates instinct for isolating hot paths from side effects
  • 05
    "Bid sniping is a real edge case — a bid in flight at the exact close moment must be resolved against the correct pre-close state, not silently dropped because the clock ticked over" — proactively surfaces a subtle correctness bug
  • 06
    "At 300+ bid attempts/sec on one auction, some 409s are expected and healthy — I'd rather have a clear, bounded retry loop than pretend I can eliminate contention entirely" — sets realistic expectations instead of over-engineering

Follow-up questions

Why optimistic locking instead of a row-level pessimistic lock (SELECT ... FOR UPDATE)?
Pessimistic locks serialize all bidders on one auction, worsening latency under contention. Optimistic CAS lets rejected bidders fail fast and retry instead of queuing behind a lock.
How do you prevent shill bidding (seller bidding on their own item)?
Out of scope for this design's core, but the guard would live in Bid Service validation: reject any bid where bidder_id equals the auction's seller_id.
How would you support proxy/auto-bidding (bid up to a max on my behalf)?
Store a max_amount per bidder; on any new competing bid, a background rule auto-submits the next valid increment on behalf of any proxy bidder whose max still exceeds it, up to their cap.
What happens if the auction close worker pool falls behind during a traffic spike?
Auctions queue in Closing slightly longer than ends_at, which is acceptable since bids are already frozen by the state transition — settlement delay doesn't affect bid correctness.
How do you handle a WebSocket client that misses updates due to a dropped connection?
On reconnect the client calls GET /auctions/{id} to resync current_high_bid and version before resubscribing — pub/sub is best-effort, REST is the consistency backstop.
How would this change for a "Buy It Now" hybrid auction?
Buy It Now is itself just a bid at a fixed amount that triggers an immediate CAS-guarded transition straight to Closing, reusing the same compare-and-swap and state machine.
