Design Dropbox

Tests whether you can design chunked, deduplicated file sync across unreliable clients without a naive re-upload-everything approach.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight is content-addressed chunking — split files into 4MB blocks, hash each one, and only upload chunks the server hasn't seen before
  • Metadata (file tree, versions, permissions) and block storage are two entirely separate systems with different access patterns and scaling needs
  • Sync is a notification-plus-pull problem, not a polling problem — a lightweight long-poll tells clients "something changed," then they pull the delta
  • Conflicts are inevitable with offline edits — I'd resolve by keeping both versions rather than silently picking a winner
  • Mobile clients are on flaky connections, so every chunk upload must be independently resumable and idempotent

Data model

-- PostgreSQL: file tree + version metadata (small rows, hot reads)
CREATE TABLE files (
  file_id     UUID PRIMARY KEY,
  owner_id    BIGINT NOT NULL,
  parent_dir  UUID,                    -- self-referencing tree
  name        TEXT NOT NULL,
  is_dir      BOOLEAN DEFAULT FALSE,
  latest_version_id UUID
);

CREATE TABLE file_versions (
  version_id  UUID PRIMARY KEY,
  file_id     UUID REFERENCES files(file_id),
  chunk_hashes TEXT[] NOT NULL,        -- ordered list of SHA-256 chunk hashes
  size_bytes  BIGINT,
  created_at  TIMESTAMP DEFAULT NOW(),
  device_id   UUID                    -- which client produced this version
);

-- Content-addressed block index: many files can point to the same chunk
CREATE TABLE blocks (
  hash        CHAR(64) PRIMARY KEY,   -- SHA-256 of chunk content
  s3_key      TEXT NOT NULL,
  size_bytes  INT,
  ref_count   BIGINT DEFAULT 1        -- incremented per file referencing it
);

-- S3: raw chunk bytes, keyed by hash — content-addressed storage
-- s3://blocks/{hash[0:2]}/{hash[2:4]}/{hash}
Why split metadata from blocks: metadata is small, relational, and read constantly (file tree, permissions) — a fit for Postgres. Blocks are large, immutable once written, and accessed by hash — a fit for S3. Coupling them into one system would force one storage engine to serve two very different access patterns badly.

Core design decisions

Why: only the changed chunks of an edited file need re-upload, and identical chunks across different users' files (the same PDF uploaded a million times) are stored exactly once.
Tradeoff: more metadata overhead (a chunk list per version instead of one blob pointer) and more round-trips to check existence before upload.
Why: plain polling wastes requests on "nothing changed" checks at 700M-user scale; a lightweight long-poll signal ("something changed, go pull") keeps the push side cheap while letting the client control the pull.
Tradeoff: long-poll connections are stateful and need connection-holding infrastructure — more operationally complex than stateless polling, though far cheaper at scale.

Why: silently discarding one user's offline edits is a data-loss bug from the user's perspective — surfacing both as "file.txt" and "file (conflicted copy).txt" preserves all data and lets the human resolve it.
Tradeoff: users occasionally see duplicate conflicted files, which is a worse UX moment than a clean merge — but far better than silent loss.
Why: a version vector (device_id, version_id pairs) lets the sync protocol detect concurrent edits from two devices, not just "which edit came later" — necessary because clock time across devices is unreliable.
Tradeoff: more complex comparison logic than a simple incrementing counter, but a counter alone can't distinguish "sequential edit" from "concurrent edit" — which is exactly the case that needs a conflict resolution.
Why: each chunk upload is keyed by its content hash, so retrying an interrupted upload after a dropped connection is naturally safe — re-sending the same chunk is a no-op if it already landed.
Tradeoff: requires the client to track per-chunk upload state locally (which chunks are confirmed vs pending) — added client-side complexity in exchange for network resilience.

Deep dive: chunked upload flow

Watch the sequence: split, hash, check existence, upload only the new chunks, then commit metadata.

  1. Split: client divides the file into ~4MB chunks locally, before touching the network
  2. Hash: compute SHA-256 for each chunk — this becomes both its identity and its dedup key
  3. Check existence: a single batched request asks the block service which of these hashes already exist in S3
  4. Upload only what's missing: chunks the server already has are skipped entirely — this is the mechanism that makes "1M people upload the same PDF" cost one chunk of storage
  5. Commit: once every chunk hash is confirmed present, the client commits a new file version referencing the ordered chunk-hash list — the file itself is never reassembled server-side until download
The upload isn't "done" from the metadata layer's perspective until every referenced chunk hash is confirmed durable in S3 — committing metadata before that risks a version pointing at a chunk that never actually landed.

Deep dive: sync protocol between two clients

Client A edits offline, reconnects; Client B is watching and receives only a delta, not the whole file.

  1. Offline edit queues locally: Client A's changes are journaled on-device until connectivity returns
  2. Reconnect and commit: changed chunks upload (deduplicated as usual), a new file version is committed to metadata
  3. Fan-out notification: the notification service pushes a small "file X has a new version" message to every other online device for that account — not the file content itself
  4. Client pulls the delta: Client B requests the metadata diff since its last known version, getting back the new chunk-hash list
  5. Selective chunk fetch: Client B diffs the new chunk list against what it has locally and downloads only the chunks it's missing — often just one changed chunk out of many
A device that was offline for days follows the same protocol on reconnect as one that missed a single notification — it just requests a larger delta. The protocol doesn't need a special "long offline" case.

Follow-up questions

How do you handle a file with millions of chunks (very large file)?
Store the chunk-hash list itself as a separate blob rather than an inline array once it exceeds a threshold, and paginate existence checks in batches instead of one giant request.
What stops a malicious user from claiming they have a chunk they don't, to skip upload?
The server verifies chunk hashes server-side after any real upload and can spot-check by requesting proof (e.g. a byte range) before trusting an existence claim for sensitive dedup paths.
How do you delete a file's storage when the last reference is removed?
Decrement each chunk's ref_count on file/version deletion; a background garbage collector reclaims chunks whose ref_count hits zero, avoiding synchronous delete-time S3 calls.
How would you support very large team folders shared across thousands of users?
Shard metadata by folder_id, batch and debounce notification fan-out, and consider a pub/sub tree so updates propagate through a small number of hops rather than one service fanning out to everyone directly.
How do you keep chunk boundaries stable when a user inserts a byte at the start of a file?
Fixed-size chunking would shift every subsequent chunk boundary. Content-defined chunking (e.g. a rolling hash like Rabin fingerprinting) picks boundaries based on content, so only the chunks actually touched by the edit change.
How is version history stored without duplicating unchanged chunks per version?
Each file_versions row stores an ordered chunk-hash list; unchanged chunks are simply referenced again by hash. Only the new/changed chunks physically exist once, and old versions remain valid as long as their referenced chunks aren't garbage collected.
