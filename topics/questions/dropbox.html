Design Dropbox

Tests whether you can design chunked, content-addressed file sync without a naive re-upload-everything approach.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: chunked upload with content-hash dedup means the same file uploaded by 1M users is stored exactly once
  • Key scale number to anchor on: at 700M users, even a 40% dedup rate on raw storage is the difference between feasible and impossible cost
  • Primary tradeoff being probed: metadata and block storage are two separate systems with different access patterns — coupling them serves neither well
  • Component most candidates forget: the notification service is a lightweight signal, not a push of file content — the client still pulls the delta
  • Failure mode most candidates skip: two offline devices editing the same file concurrently — silently picking a winner is a data-loss bug

Long-poll instead of WebSocket: sync events are infrequent per device and one-directional (server → client signal), so holding a full-duplex socket open per idle device isn't worth the infra cost. Long-poll gives near-real-time delivery at a fraction of the connection overhead.
Step 4 of 6 — Data Flow

Data flow

Upload path — only new chunks transfer
Download path — CDN accelerates popular blocks
  1. Client splits and hashes file locally — before any network call
  2. Batch existence check against Block Service — server returns which hashes are missing
  3. Only missing chunks upload — chunks already in S3 are skipped entirely
  4. Metadata commit — once every chunk is confirmed durable, a new file version is written
  5. Download reverses it — metadata gives the chunk list, chunks fetch in parallel through CDN

Step 6 of 6 — Deep Dives

Deep dive: chunked upload sequence

Client / gateway
Cache — hot path
Database — source of truth
Async — queue / worker
  1. Split — client divides the file into ~4MB chunks locally
  2. Hash — SHA-256 each chunk, becomes its identity and dedup key
  3. Check existence — one batched request against the block service
  4. Upload only what's missing — this is the mechanism that makes 1M identical uploads cost one chunk
  5. Commit — a new version is written referencing the ordered chunk-hash list
The upload isn't "done" from the metadata layer's view until every referenced chunk hash is confirmed durable in S3 — committing metadata earlier risks a version pointing at a chunk that never landed.

Deep dive: conflict resolution

  1. Both clients edit offline — each queues its own new version locally, tagged with its device_id
  2. Both reconnect and attempt to commit — the second commit's parent version no longer matches the current latest_version_id
  3. Server detects divergence — the version vector comparison shows two branches from the same parent, not a linear history
  4. Keep both — the losing client's version is saved as "file (conflicted copy — device B).txt" rather than discarded
  5. User resolves manually — both versions remain fully recoverable, no silent data loss

Interview moves

  • 01
    "The core insight here is content-addressed chunking — I'd split files into 4MB blocks, hash each with SHA-256, and only transfer chunks the server hasn't seen before" — names the actual hard problem unprompted
  • 02
    "I'm keeping metadata and block storage as two completely separate systems since their access patterns don't match" — shows you won't force one database to do two incompatible jobs
  • 03
    "Sync is notify-then-pull, not poll-then-diff — a long-poll tells the client something changed, and the client decides what to fetch" — rules out a naive polling design
  • 04
    "On conflict, I'd keep both versions rather than pick a winner — silently discarding a user's edit is worse than a visible duplicate file" — a mature, user-first tradeoff call
  • 05
    "Every chunk upload needs to be idempotent since mobile clients are on flaky connections — retrying a chunk should always be safe" — proactively addresses a realistic failure mode
  • 06
    "The same PDF uploaded by a million users should cost one chunk of storage, not a million copies — that's what the content hash buys us" — makes the dedup payoff concrete

Follow-up questions

How do you handle a file with millions of chunks?
Store the chunk-hash list as a separate blob rather than an inline array past a threshold, and paginate existence checks in batches.
What stops a malicious user from claiming they have a chunk they don't?
The server verifies hashes server-side after any real upload and can spot-check with a byte-range proof before trusting an existence claim for sensitive dedup paths.
How do you delete storage when the last reference is removed?
Decrement ref_count on file/version deletion; a background garbage collector reclaims chunks whose ref_count hits zero, avoiding synchronous delete-time S3 calls.
How would you support very large team folders shared across thousands of users?
Shard metadata by folder_id, batch and debounce notification fan-out, and consider a pub/sub tree so updates propagate through fewer hops than one service fanning out to everyone.
How do you keep chunk boundaries stable when a user inserts a byte at the start of a file?
Fixed-size chunking shifts every later boundary. Content-defined chunking (rolling hash) picks boundaries based on content, so only chunks actually touched by the edit change.
How is version history stored without duplicating unchanged chunks per version?
Each version stores an ordered chunk-hash list; unchanged chunks are simply referenced again by hash — only new/changed chunks physically exist once.
