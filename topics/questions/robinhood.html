Design Robinhood (Trade Execution)

Tests whether you treat trade execution as a financial-correctness problem first and a scale problem second — a network retry must never turn into a duplicate buy or sell.

45-55 min interview Hard
Say this in the first 2 minutes
  • The core insight: order execution must be idempotent and exactly-once — a client retry after a timeout must never re-execute the same trade
  • Key scale number to anchor on: market open/close creates a ~100x QPS spike in seconds, not minutes — this is a burst problem, not a steady-growth problem
  • Portfolio reads outnumber writes roughly 1000:1, so the read path (balance/positions) and the write path (order placement) need completely different scaling strategies
  • Component most candidates get wrong: account balance must never be a single mutable column — it has to be derived from an append-only ledger or concurrent updates silently lose money
  • Failure mode most candidates skip: an exchange execution confirmation that's delayed or lost leaves an order stuck in "submitted" — you need a reconciliation job, not just hope

Step 4 of 6 — Data Flow

Data flow

Order submission (blue, low volume, idempotency-checked) and portfolio reads (mint, ~1000x the volume) never touch the same hot path.

Client app — order submission and portfolio view
Idempotency check + portfolio cache — the hot-path components
Postgres — order, ledger, and derived balance, source of truth
  1. Client submits an order with an idempotency key — generated once per attempt, reused verbatim on any retry
  2. Order Service validates buying power — rejects immediately if the account can't cover the order
  3. Idempotency check runs before execution — a seen key short-circuits straight to the cached response, no re-execution
  4. Order and ledger entry commit atomically — both succeed or both fail, never one without the other
  5. Portfolio reads almost always hit cache — Postgres is only touched on a cache miss or TTL expiry

Step 6 of 6 — Deep Dives

Deep dive: idempotency key flow

The client generates the key once per order attempt. A timeout-triggered retry reuses that exact key — the server recognizes it and returns the cached response instead of executing a second trade.

orders (idempotency columns)Postgres
order_idUUID PRIMARY KEY
idempotency_keyTEXT NOT NULL, UNIQUE INDEX
statusTEXT — submitted/filled/rejected
response_bodyJSONB — the exact response to replay
created_atTIMESTAMPTZ
The UNIQUE index on idempotency_key is what makes correctness structural: the second INSERT attempt with the same key fails at the database level, and that failure is the signal to fetch and return response_body instead of executing anything.
  1. Client generates one idempotency key per logical order attempt — a UUID created client-side before the first network call
  2. Server attempts an INSERT keyed on idempotency_key — the UNIQUE index makes a duplicate insert fail fast, before any exchange interaction
  3. Trade executes exactly once — only the winning INSERT proceeds to validation and routing
  4. Response is persisted alongside the order — not just returned to the caller, but stored for replay
  5. On retry with the same key, the conflict is caught — server fetches the stored response_body and returns it, skipping execution entirely
The key must be generated once per user intent and reused across retries of that same intent — if the client mints a new key on every retry, the UNIQUE index provides zero protection and duplicate trades become possible again.

Deep dive: portfolio consistency via ledger pattern

A single mutable balance column is unsafe under concurrency. An append-only ledger with a derived balance is not.

  1. Never store balance as a single mutable field — a naive UPDATE accounts SET balance = balance - X under concurrency has a read-then-write gap where two transactions can both read the same stale value
  2. Model every change as an append-only ledger entry — a deposit, fee, or trade fill is an INSERT of a signed delta, never an overwrite
  3. Derive balance by summing the ledger — correctness follows from arithmetic on immutable rows, not from lock timing
  4. Materialize the derived balance for hot-path reads — recompute periodically or on every write and cache it, so buying-power checks don't sum the whole ledger per request
  5. Reconcile derived balance against the ledger sum on a schedule — catches drift from a lost materialization write before a user notices
The lost-update problem isn't a rare edge case here — every order fill, fee, and deposit is a concurrent write candidate on a busy account, and a single mutable balance column will eventually silently drop one of them.

Deep dive: market-hours burst traffic

Traffic is flat for most of the day, then jumps ~100x within seconds at 9:30am ET — reactive autoscaling reacts on a timescale of minutes, far too slow for a spike that arrives in seconds.

  1. Baseline QPS holds steady for most of the trading day — normal reactive autoscaling would be adequately sized here
  2. At 9:30am ET, order and quote-subscription volume jumps ~100x in seconds — not over minutes, so metric-triggered autoscaling (CPU/queue-depth thresholds) hasn't even finished evaluating before the spike has already landed
  3. Reactive autoscaling's reaction time (minutes to provision + warm up) exceeds the spike's arrival time (seconds) — by the time new capacity is healthy, the worst of the burst has already been served in a degraded state or dropped
  4. Pre-scale on a schedule instead — provision extra capacity a fixed window (e.g. 15-30 min) before every known market-open and market-close time, the same way ticket-sale platforms pre-scale ahead of a published on-sale time
  5. Scale back down on a schedule after the post-open volume settles — avoids paying for peak capacity all day
Autoscaling triggered by observed load is fundamentally reactive — it is the wrong tool for a spike whose timing is known in advance. Market open/close times are published and fixed, so this should be a scheduled pre-scale, not a monitored threshold.

Follow-up questions

What if two requests arrive with the same idempotency key at the exact same moment?
The UNIQUE index on idempotency_key makes the database the arbiter — only one INSERT wins; the other fails and its handler fetches and returns the winner's stored response.
How long do you retain idempotency keys?
Keep them as long as the order row itself lives, since a legitimate late retry (e.g. after a long network partition) should still resolve safely rather than be treated as a brand-new order.
Why not just use optimistic locking on a mutable balance column instead of a ledger?
Optimistic locking fixes the race but not the audit trail — a ledger also gives you a full, immutable history for disputes, compliance, and reconciliation, which a single column can never provide.
How do you keep the materialized balance in sync with the ledger cheaply?
Update it incrementally on every ledger insert within the same transaction, then run a periodic full-sum reconciliation job as a correctness backstop, not the primary mechanism.
What if pre-scaling for market open isn't enough for an unusually volatile day?
Layer reactive autoscaling on top of the scheduled pre-scale as a secondary defense, and apply load-shedding/backpressure on non-critical reads (e.g. serve slightly stale portfolio data) before ever rejecting order writes.
How would you extend this design to support limit orders that don't fill immediately?
The order stays in a "working" state until the exchange sends a fill or cancel event; the same execution-confirmation consumer and reconciliation job handle it, just over a longer, indefinite window.
