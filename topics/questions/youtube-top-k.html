Design Top-K Trending Videos (YouTube)

Tests whether you know that exact top-K at global scale is a myth — the real design is a probabilistic counting structure feeding a bounded heap, not a giant counter-per-video table.

35-45 min interview Medium
Say this in the first 2 minutes
  • The core insight: exact top-K over hundreds of millions of videos and billions of daily events doesn't scale — you need approximate counting (Count-Min Sketch) instead of an atomic counter per video
  • Key scale number to anchor on: 5B views/day is ~58k events/sec average, with the top-K query needing to answer in under 100ms
  • Primary tradeoff being probed: exactness vs memory — a sketch trades a small, bounded overestimation error for constant memory regardless of how many distinct videos exist
  • Component most candidates forget: a bounded min-heap of size K sitting on top of the sketch — the sketch alone doesn't give you "top-K," it only gives you fast estimates you still have to rank
  • Failure mode most candidates skip: a single viral video can overload one counter shard — needs local pre-aggregation before it ever touches the shared sketch

Step 4 of 6 — Data Flow

Data flow

The write path (amber) streams every view event through Kafka into the sketch asynchronously; the read path (mint) only ever touches the periodic snapshot, never the live sketch.

Kafka + Counter Service — decoupled, high-throughput write path
Redis CMS — the hot, ephemeral counting structure
Snapshot DB — what every read actually hits
  1. View fires — the watch page/app emits a view event once playback crosses a minimum threshold
  2. Event lands on Kafka — partitioned by video_id so one video's events stay ordered and co-located
  3. Counter Service pre-aggregates — batches counts locally per partition before incrementing the shared sketch
  4. Sketch cells increment — one increment per hash function row, not one key per video
  5. Snapshot job reads the sketch — periodically extracts the current top-K and writes it durably
  6. Query Service serves reads — every trending request hits the snapshot, completely decoupled from ingestion load

Step 6 of 6 — Deep Dives

Deep dive: Count-Min Sketch mechanics

A view event for a video is hashed by d independent hash functions, each mapping to one column in its own row of a d × w counter matrix. One event increments d cells at once — one per row. To estimate a video's count, hash it the same way and take the minimum across those d cells — collisions can only inflate a cell, never deflate it, so the min is the tightest available upper bound.

  1. Event arrives for a video — the video_id is fed into d independent hash functions
  2. Each hash picks one column per row — d cells total are touched, one per row, possibly shared with other videos
  3. All d cells increment by one — a single view event is a single write fanned out across rows
  4. Estimate reads hash the same video_id — landing on the same d cells deterministically
  5. Take the MIN across those cells — collisions only inflate a cell, so the minimum is the tightest upper bound on the true count
Because two different videos can collide into the same cell, every cell is a sum of possibly multiple videos' counts — that's exactly why you take the min across rows instead of trusting any single cell: it's the row least corrupted by collisions.

Deep dive: heavy hitters — exact vs approximate top-K

Exact top-K needs a full count per distinct video and a full sort (or a global atomic counter per video) — both fall apart at hundreds of millions of videos. Approximate top-K bounds memory to a fixed sketch size plus a fixed-size heap.

Exact top-K (impossible at this scale)
  • One counter per distinct video — memory grows with the number of videos, not a fixed budget
  • Every query needs a full sort (or a full scan) across all counters to find the top K
  • A single global atomic counter per video becomes a hot lock under viral load
  • Memory for hundreds of millions of videos is untenable to keep hot in memory
Approximate top-K (what actually ships)
  • Count-Min Sketch: fixed d × w matrix, independent of how many distinct videos exist
  • A bounded min-heap of size K tracks only current candidates, updated incrementally as counts change
  • No global lock — only local shard pre-aggregation plus periodic merges
  • Error is small, known, and one-sided (only ever overestimates)
  1. Stream has unbounded cardinality — hundreds of millions of distinct video_ids, no way to hold one counter each affordably
  2. Sketch compresses cardinality to fixed memory — d × w cells absorb the entire stream, accepting collisions as the cost
  3. Heap tracks only K live candidates — every sketch update checks if a video's new estimate beats the heap's current minimum
  4. Losing candidates are evicted — the heap never grows past size K, so its memory cost is constant
  5. Error is tunable, not accidental — picking w and d before launch is a deliberate memory-vs-accuracy trade, not a bug to fix later
The error bound is one-sided by construction — a video's estimated count can be inflated by collisions but is mathematically guaranteed to never be undercounted, which is exactly why taking the min across rows is safe and why a video can rank into the top K it shouldn't, but never rank out of one it deserves.

Deep dive: windowed vs all-time top-K

"Trending in the last hour" and "most-viewed of all time" are different questions that need different data structures — a single persistent sketch can't cheaply answer both.

  1. Windowed queries need decay — a video's popularity 3 hours ago shouldn't count toward "trending now," so old buckets must expire
  2. Time-bucketed sketches solve this — open a fresh small sketch every few minutes, merge the last N buckets by elementwise sum for the current window
  3. All-time queries need permanence — a single sketch that's never reset, growing (in accuracy demand, not physical size) over the video's life
  4. One structure can't do both cheaply — resetting periodically (needed for freshness) destroys history; never resetting (needed for permanence) can't express "just the last hour"
  5. Two structures, two purposes — bucketed+merged sketches for any rolling window, one persistent sketch (or exact aggregate, since all-time cardinality growth is slower) for all-time
All-time counts are often worth computing exactly via a batch job (e.g. nightly aggregation from durable logs) instead of a sketch — the write rate against all-time is the same, but the tolerance for a slower, exact answer is much higher than for "trending right now."

Follow-up questions

Why not just use an exact counter with atomic increments per video?
Memory scales linearly with distinct video count — hundreds of millions of hot keys is untenable to keep in fast storage, and a full sort for top-K would be too slow regardless.
Can the Count-Min Sketch ever undercount a video?
No — collisions only add extra weight to a cell, they never remove it, so every cell is an upper bound and the minimum across rows is the tightest available upper bound.
How do you size the sketch's width and depth?
Width (w) controls the error magnitude epsilon, roughly e/epsilon; depth (d) controls the failure probability delta, roughly ln(1/delta) — pick both from your acceptable error and confidence targets.
What happens if the snapshot job falls behind?
The trending list goes stale, not wrong or unavailable — the API exposes snapshot age so clients can decide whether to show a "may be outdated" indicator.
How would you support trending broken out by region or category?
Key the sketch and heap by (region, window) or (category, window) instead of a single global instance — same mechanism, partitioned differently, more memory in exchange for finer granularity.
Could a malicious actor game the trending list with bot views?
Yes without mitigation — add a lightweight anomaly filter upstream of the Counter Service (rate limits per viewer/IP, watch-time thresholds) before events ever reach the sketch.
