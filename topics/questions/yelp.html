Design Yelp

Tests whether you treat "search nearby businesses" as a geo-indexing and ranking problem first, and reviews as ordinary CRUD second.

30-40 min interview Medium
Say this in the first 2 minutes
  • The core insight: geo search + ranking is the hard part of this system, reviews and business profiles are standard CRUD on top
  • Key scale number to anchor on: 12M businesses indexed, ~8k search QPS average with 5-10x spikes in dense metros at peak hours
  • Primary tradeoff being probed: a precomputed geo index (fast reads, stale on writes) vs querying live (always fresh, slow at scale)
  • Component most candidates forget: search ranking is a multi-signal blend — distance alone produces bad results next to a 2.1-star business
  • Failure mode most candidates skip: a dense downtown core can have 50x the business density of a suburb, breaking uniform-grid geo indexes

Step 4 of 6 — Data Flow

Data flow

Search reads (mint) hit the geo index directly; business/review writes (blue) land in Postgres first, then propagate to the index async.

Read path — cache then geo index
Write path — strong consistency in Postgres first
Reindex — decoupled, eventually consistent
  1. Client searches by location + keyword — hits Search Svc with lat/lng, radius, category
  2. Redis checked for a hot query cache — keyed on geohash cell + category + query text
  3. Cache miss → Elasticsearch geo query — filters by radius, scores by the ranking blend
  4. Business/review writes land in Postgres first — the transactional source of truth, response returns immediately
  5. Change event flows through Kafka to reindex — search results reflect the update within seconds, not synchronously

Step 6 of 6 — Deep Dives

Deep dive: quadtree vs geohash for the geo index

Both turn a 2D radius query into a fast lookup — they differ in how they handle uneven business density across a city.

Quadtree — adapts to density
Geohash — uniform grid
ApproachWhen it winsTradeoff
QuadtreeUneven density — dense downtown cores next to sparse suburbs, which is exactly a city's business distributionMore complex to implement and rebalance; tree traversal is slower than a string-prefix lookup
GeohashSimplicity and uniform load — easy to shard by prefix, trivial to reason about, works well when density is roughly evenA single dense cell becomes a hotspot with no built-in way to subdivide it further
In practice: Elasticsearch's geo_point uses a geohash-like structure internally, but the engine shards and balances automatically — most teams don't hand-roll either structure. The interview signal is knowing why density matters, not implementing a quadtree from scratch.

Deep dive: search ranking signals

Distance alone produces bad results — a 4.8-star cafe 400m away should usually beat a 2.1-star cafe 150m away. Ranking blends four signals into one score.

  1. Geo filter narrows the candidate set — Elasticsearch geo-radius query returns everything within the search radius first, typically a few hundred to a few thousand businesses
  2. Distance decay score — closer businesses score higher, but the curve flattens past ~1km so distance stops dominating once "nearby enough"
  3. Star rating score — average rating weighted by review count, so a single 5-star review doesn't outrank a business with 500 reviews at 4.6
  4. Review recency score — a business with recent reviews signals it's currently open and active, a small boost over one with reviews from 3 years ago
  5. Text relevance score — keyword match against name, category, and review text (BM25-style), only applied when the query has search terms
  6. Weighted blend produces final rank — score = w1·distance_decay + w2·rating_score + w3·recency + w4·relevance, weights tuned per query type (browse vs keyword search)
Weighting is context-dependent: a "pizza near me" query should weight distance heavily since intent is clearly proximity-driven; a "best pizza in [neighborhood]" query should weight rating and relevance more heavily since the user already named a location.

Interview moves

  • 01
    "Before I design anything, let me split this into two problems — geo search is the hard part, business profiles and reviews are standard CRUD" — states the core insight unprompted and sets scope
  • 02
    "I wouldn't run a geo-radius query against Postgres at 12M rows — I'd maintain a dedicated search index that's async-updated from the source of truth" — names the read/write split explicitly
  • 03
    "Ranking can't be distance-only — a 2-star business next door shouldn't outrank a 4.8-star business three blocks away, so I'd blend distance, rating, recency, and relevance" — shows the ranking is a designed system, not an afterthought
  • 04
    "A uniform geohash grid breaks down in a dense downtown core — I'd want variable-depth cells there, closer to a quadtree, while suburbs stay coarse" — proactively surfaces the density problem
  • 05
    "Reviews go straight into Postgres for strong consistency on the one-review-per-user rule, then flow into the search index async — a slow reindex should never block a review submission" — separates write correctness from read freshness
  • 06
    "I'd flag review-bombing as a real failure mode — a sudden spike in reviews with an extreme rating delta on one business should be rate-limited and queued for fraud review, not trusted immediately" — demonstrates abuse-case thinking unprompted

Follow-up questions

How do you handle autocomplete/typeahead for the search box?
A separate trie or Elasticsearch completion suggester over business names and categories, served from an in-memory cache — a different latency budget than full search.
How would you support filtering by "open now"?
Store structured hours as a JSONB field, evaluate against the request's local time server-side, and add it as a post-filter after the geo/rank query rather than indexing every possible time slot.
How do you prevent duplicate business listings for the same physical location?
Fuzzy-match name + geo-proximity + phone number at creation time, flag likely duplicates for manual merge rather than auto-rejecting, since false positives block legitimate new businesses.
How would this change to support 10x the scale?
Geo-shard Elasticsearch by region so each shard's working set fits in memory, add more Redis cache capacity for hot queries, and move to variable-depth cells more broadly, not just downtown cores.
Why not compute avg_rating live with a JOIN instead of denormalizing it?
A profile page and every search result would need a COUNT/AVG aggregate over potentially thousands of reviews — denormalizing into a cached summary makes that a point read instead.
How do you rank sponsored/ad businesses without making search feel untrustworthy?
Keep organic ranking signals untouched, inject sponsored results into clearly labeled slots, and cap how much an ad boost can override a very low organic rating.
