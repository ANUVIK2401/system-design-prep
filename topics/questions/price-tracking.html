Questions

Design a Price Tracking Service

Tests whether you can keep a scraper fleet alive against sites actively trying to block it, not whether you can store a time series.

30-40 min interview Medium
Say this in the first 2 minutes
  • The core insight: getting blocked by target sites is the primary operational risk, not storage or alert delivery — anti-detection is a first-class design concern
  • Key scale number to anchor on: 50M tracked products polled on independent per-domain schedules means the scheduler itself is a hard distributed-systems problem
  • Primary tradeoff being probed: crawl frequency vs politeness — hitting a domain too often gets the whole fleet IP-banned, not just one request
  • Component most candidates forget: a per-domain rate limiter that throttles the scraper fleet itself, independent of the site's own limits
  • Failure mode most candidates skip: a site changes its HTML structure — the scraper must fail loudly (alert a human) rather than silently store garbage prices

Step 4 of 6 — Data Flow

Data flow

Scrape path (amber, async/scheduled) and user read path (mint) never touch the same hot component — a slow scrape never blocks a chart load.

Scrape path — scheduled, per-domain throttled
Read path — direct query, fully decoupled from scraping
  1. Scheduler computes due jobs — every product whose poll_interval_min has elapsed since its last scrape
  2. Jobs land in per-domain queues — isolates a slow or blocking retailer from delaying every other domain
  3. Worker checks Redis throttle state — skips or delays the request if the domain is in backoff
  4. Worker fetches and extracts price — via the domain's configured CSS/JSON selector
  5. Price point written, alerts evaluated — a write to TimescaleDB triggers a synchronous check against PriceAlert rules for that product

Step 6 of 6 — Deep Dives

Deep dive: scraper anti-detection techniques

Every layer exists to make the scraper fleet look like normal, distributed human traffic rather than a single bot hammering a site.

✓ Well-behaved fleet — sustained access
✗ Naive fleet — banned within minutes
  1. Rotate proxy IPs per domain — a pool of residential/datacenter proxies, never hammering from one IP
  2. Rotate user-agent strings — mimics a mix of real browsers instead of one static bot signature
  3. Jitter request timing — randomized delay instead of a suspiciously exact fixed interval
  4. Per-domain rate limiting — the scraper throttles itself below the site's own detection threshold, not just above it
  5. Respect robots.txt where reasonable — reduces adversarial posture and legal risk, though it isn't a technical requirement for public price pages
This is an arms race, not a solved problem: sites tighten detection over time, so the anti-bot layer needs ongoing tuning per domain, not a one-time configuration.

Deep dive: price alert evaluation

Two threshold types cover nearly every real use case, evaluated synchronously on every new price point.

Absolute threshold
  • "Alert me when this drops below $50"
  • Simple comparison: new price_cents < threshold_value
  • Best for a specific target price the user has in mind
Percentage drop threshold
  • "Alert me on any 15%+ drop from the price I started tracking at"
  • Compares against a stored baseline price, not the previous point
  • Best for catching sales without knowing a target price upfront
  1. New price point written — the scraper worker's write to TimescaleDB completes
  2. Alert Engine looks up active rules — indexed query on PriceAlert.product_id, typically a handful of rows
  3. Each rule evaluated against the new price — absolute or percentage-drop comparison
  4. Matching rules enqueue a notification — deduplicated so a user isn't alerted twice for the same drop within a cooldown window

  • 02
    "I'd queue scrape jobs per-domain, not in one global queue, so one retailer's slowdown or ban never delays every other domain" — shows isolation instinct before being prompted
  • 03
    "A shared Redis-backed throttle means every worker in the fleet respects the same per-domain rate limit, even across hundreds of instances" — names the exact mechanism, not just the concept
  • 04
    "If extraction starts returning null or wildly different prices, I want that to page a human, not silently write garbage into the price history" — demonstrates data-integrity instincts
  • 05
    "I'd separate the scrape pipeline from the read API entirely — a user loading a price chart should never be slowed down by scraper backpressure" — correctly identifies the two independent hot paths
  • 06
    "Poll interval shouldn't be fixed — a $2,000 item a user is actively watching deserves more frequent checks than a $10 item nobody's alerted on" — connects a technical scheduling decision to product value

Follow-up questions

How do you avoid tracking the same product twice under slightly different URLs?
Normalize URLs at track time — strip tracking/query params, resolve redirects — then enforce a UNIQUE constraint on the normalized form before creating a new TrackedProduct.
How would you prioritize which products to scrape first when the worker pool is saturated?
Weight the per-domain queue by number of active alert subscribers and how overdue the product is relative to its poll interval, not simple FIFO.
What if a site requires JavaScript rendering to show the real price?
Route that domain to a separate headless-browser worker tier — it's far more expensive per request, so only sites that actually need it should use it.
How do you handle currency and regional pricing differences?
Store price_cents alongside a currency code and normalize to the user's display currency at read time using a periodically refreshed FX rate table, never at write time.
Could you reduce cost by sharing scrape results across users tracking the same product?
Yes — TrackedProduct is already deduplicated by normalized URL, so one scrape serves every user watching that product; only PriceAlert rules are per-user.
How would you extend this to support 10x the tracked products?
Shard TimescaleDB by product_id range, scale out per-domain queue partitions, and widen poll intervals for long-tail low-alert-subscriber products first.
