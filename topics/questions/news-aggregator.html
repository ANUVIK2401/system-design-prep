Design a News Aggregator

Tests whether you can turn crawl scheduling and cross-source dedup into a first-class design problem instead of hand-waving "we scrape the internet."

30-40 min interview Medium
Say this in the first 2 minutes
  • The core insight: crawl scheduling and dedup are the hard problems here, not storage — any store handles the write volume fine
  • Key scale number to anchor on: ~150k source feeds, ~40M new articles/day, but re-crawl checks dwarf that at ~2B fetches/day
  • Primary tradeoff being probed: freshness (crawl aggressively) vs crawl cost (bandwidth, politeness, source rate limits)
  • Component most candidates forget: a URL frontier that prioritizes re-crawls by source velocity, not a flat round-robin queue
  • Failure mode most candidates skip: a Bloom filter false positive silently drops a real article — must state the accepted error rate out loud

  2. Crawler fetches feed/page — respects robots.txt, per-domain rate limits, conditional GET (ETag)
  3. Bloom filter check on url_hash — likely-duplicate URLs are dropped before any parsing happens
  4. New article persisted — raw HTML to S3, metadata row to Postgres, event fired to Kafka
  5. Async consumers index and cluster — Elasticsearch gets the text, clusterer groups same-story duplicates

Step 6 of 6 — Deep Dives

Deep dive: URL frontier priority queue

150k sources can't be polled on a flat round-robin — a wire feed publishing every 30s and a monthly blog need wildly different crawl cadences.

  1. Track rolling publish rate per source — exponential moving average of articles/hour, updated after every crawl
  2. Compute next due_time — due_time = now + clamp(3600 / avg_publish_rate, 30s, 6h), so a high-velocity wire gets checked every 30s and a quiet blog every few hours
  3. Push (source_id, due_time) into a Redis sorted set — score is due_time, giving an O(log n) priority pop
  4. Scheduler pops due sources in batches — dispatches to the crawler fleet respecting a per-domain concurrency cap
  5. Re-score after crawl — a source that just published gets re-prioritized; a source with zero new articles across 5 checks backs off further
Politeness beats greed: a source with no rate cap will get hammered by an eager frontier and rate-limit or block the crawler entirely. Every source carries a max-concurrency and min-interval floor that the priority score can never override.

Deep dive: Bloom filter dedup — the false-positive math

A Bloom filter trades a small, tunable false-positive rate for a massive memory win over storing every seen URL as a real set.

2B
expected inserted URLs (n)
~34.2 Gb
bit array size (m) for target FP rate
7
optimal hash functions (k)
0.1%
target false-positive rate (p)

Sized from the standard formulas m = -(n · ln p) / (ln 2)² and k = (m/n) · ln 2. At n=2B and p=0.1%, that's m ≈ 2.87×10¹⁰ bits ≈ 3.4GB.

Versus roughly 128GB+ to store 2B full SHA-256 hashes (32 bytes each) in a real set — a ~40x memory reduction for a 1-in-1000 chance of a false duplicate.

A false positive here is silent by design — the article just never gets crawled, with no error and no retry. Mitigate by rotating the filter on a 30-day TTL window (old URLs age out, keeping n and thus m bounded) and by treating the Bloom check as advisory-only for high-priority sources, backed by a real url_hash lookup in Postgres before a final drop.

Deep dive: feed ranking pipeline — signals → score → rank

Ranking runs as a batch job every ~60s per topic, not per-request — computing a personalized score for 80k QPS in real time doesn't fit the latency budget.

✓ Pre-ranked cache hit — ~2-5ms
✗ Cold/niche topic miss — ~40-80ms
  1. Collect signals per article — recency, source trust score, cluster size (how many outlets ran the story), early click-through rate
  2. Compute a weighted score — score = 0.4·recency + 0.25·source_trust + 0.2·cluster_size + 0.15·ctr, weights tuned offline
  3. Sort and truncate per topic — top 500 articles/topic written to Redis, far more than any single page needs
  4. Feed Svc paginates from the cached list — cursor-based, no re-ranking per request

Follow-up questions

How do you detect that two articles from different outlets are the same story?
Title/text shingling (n-gram overlap) catches near-identical wire copy cheaply; embedding similarity on the extracted text catches reworded coverage the shingling misses.
What happens if a source's RSS feed goes down but the site still publishes?
Frontier tracks consecutive feed failures and falls back to an HTML sitemap/homepage crawl after a threshold, flagging the source for manual review if that also fails.
How would you extend this to support 10x the source count?
Shard the frontier's sorted set by source_id hash, horizontally scale the crawler fleet, and shard Elasticsearch by time range so old indices can be archived off the hot cluster.
How do you avoid re-processing the same article if the crawler retries after a timeout?
The Bloom filter check happens before any parsing or storage write, and the final Postgres insert is guarded by the unique constraint on url_hash, making the write path idempotent.
Why not rank every feed request live instead of batching every 60 seconds?
Live ranking means recomputing scores across thousands of candidate articles per request at 80k QPS — the freshness lost by batching is a better tradeoff than that compute cost.
How do you handle a source that's flagged as low-quality or spammy?
Source.avg_publish_rate and a separate trust score feed directly into the ranker's weighting, and a trust score below threshold can drop a source from the frontier entirely.
