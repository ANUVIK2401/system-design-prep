Questions

Design a URL Shortener

Tests whether you can identify the real hard problem — ID generation, not redirect logic — and build a read-heavy system around it.

30-40 min interview Easy
Say this in the first 2 minutes
  • The core insight: ID generation is the hard problem here, not the redirect — the redirect is a one-line cache lookup
  • Key scale number to anchor on: reads outnumber writes roughly 100:1, so every decision optimizes the redirect path
  • Primary tradeoff: base62 counter vs hash vs Snowflake for ID generation — collision risk vs coordination cost
  • Component most candidates forget: a CDN in front of the redirect service, not just a Redis cache behind it
  • Failure mode most candidates skip: what happens to click analytics if the async pipeline backs up — it must never block the redirect

Step 4 of 6 — Data Flow

Data flow

Read path — cache-first, CDN in front
Write path — low volume, strong uniqueness
  1. Client requests /{short_code} — hits CDN edge first
  2. CDN cache hit — returns cached 302 immediately, request never reaches origin
  3. CDN miss → Redirect Svc → Redis — checks in-memory cache for short_code
  4. Redis miss → Postgres — indexed lookup on short_code, result written back to Redis
  5. 302 returned to client — browser follows to long_url

Step 6 of 6 — Deep Dives

Deep dive: ID generation

Base62 counter
  • Simple: increment a counter, encode to base62
  • No collisions by construction
  • Single point of contention — one DB sequence
  • Codes are sequential — guessable, minor info leak
MD5/SHA hash of URL
  • No coordination needed — pure function of input
  • Collision risk requires a uniqueness check on every write anyway
  • Truncating the hash for a short code raises collision odds further
At this scale (~12 writes/sec), a Postgres sequence with base62 encoding is genuinely sufficient — reach for Snowflake only once write throughput or multi-region writes make a single sequence a bottleneck (see the Bitly page for that scale).

Follow-up questions

How would you prevent short code enumeration attacks?
Rate limit per IP/user at the gateway. 62^7 ≈ 3.5 trillion combinations already makes brute-force scanning impractical without added protection.
What if the same long URL is submitted twice?
Default: generate a new code each time. Dedup by hashing long_url adds a read on every write to save storage — a real tradeoff, not a free win.
How do you handle 10x the current scale?
Shard Postgres by short_code prefix, switch ID generation to Snowflake, extend the code to 8 characters, make CDN mandatory rather than optional.
301 or 302 redirect?
302 by default — keeps every click hitting the server. 301 lets browsers cache it, which is faster but kills click visibility after the first visit. See the Bitly page for the full tradeoff.
How would you implement expiration efficiently?
Store expires_at in both Postgres and as a Redis TTL. The TTL handles the hot-path check for free; a daily background job hard-deletes expired rows in batches.
How do you keep the CDN and Redis cache consistent after a delete?
Short TTL (~60s) on both as a safety net, plus an explicit purge/DEL call fired synchronously on delete rather than relying on TTL expiry alone.
