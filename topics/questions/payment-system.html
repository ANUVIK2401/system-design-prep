Questions

Design a Payment System

Tests whether you can coordinate state across multiple parties — buyer, seller, platform, external bank/card network — when the external confirmation is async, sometimes delayed, and sometimes duplicated.

45-55 min interview Hard
Say this in the first 2 minutes
  • The core insight: this is a multi-party state coordination problem — buyer, seller/merchant, platform, and an external processor all need a consistent view, and the processor's confirmation arrives async, not synchronously
  • Key scale number to anchor on: 10M transactions/day, but processor confirmation latency ranges from milliseconds (card) to seconds or minutes (bank transfer/ACH) — the design must not block on the slow end
  • Idempotency is mandatory, not optional: every charge request carries a client-generated idempotency key, because a client retry after a timeout must never double-charge a buyer
  • Component most candidates get wrong: balances must never be a mutable column — a double-entry, append-only ledger is what makes multi-party settlement (buyer debit, merchant credit, platform fee credit) provably net to zero
  • Failure mode most candidates skip: processor webhooks can be delayed, retried, or arrive out of order — you need webhook dedup by processor event ID plus a nightly reconciliation job against the processor's own statement, not blind trust in the webhook stream
Step 1 of 6 — Requirements

Requirements

Functional
  • Buyer initiates a charge against a merchant for an order amount
  • Payment is authorized and captured via an external processor (card network, bank rail)
  • On success, funds are split: merchant payout amount, platform fee, both tracked separately
  • Buyer, merchant, and platform can each query the current state of a transaction
  • Refunds and disputes can be issued against a previously completed transaction
Non-Functional
  • Idempotent charge handling — a retried request must never double-charge the buyer
  • 10M transactions/day, processor confirmation latency from ~200ms to 30+ seconds depending on rail
  • Never lose or double-apply a webhook — dedup by processor event ID, tolerate out-of-order delivery
  • Ledger must always net to zero across all parties on every settled transaction
  • Nightly reconciliation against the processor's statement catches any drift within 24 hours
What NOT to build: card tokenization/PCI-compliance implementation details (assume a PCI-compliant vault/tokenization provider sits in front of us), a fraud/risk scoring ML pipeline, and multi-currency FX conversion logic. This is transaction state coordination, ledger accounting, and processor reconciliation only.

Balance view — materialized, absorbs read volume off the raw ledger
External processor + ledger — durable sources of truth, one external, one internal
Webhook ingestion + ledger write — decoupled from the synchronous charge call

  2. Orchestration service creates the transaction in "pending" — before the processor has confirmed anything
  3. Processor call returns an immediate ack, not a final result — actual authorization/capture confirmation arrives later via webhook

  5. Ledger entries are written only after a deduped, verified webhook confirms success — never optimistically on the initial request

Step 6 of 6 — Deep Dives

Deep dive: idempotent charge flow with external processor

Client / gateway
Cache — hot path
Database — source of truth
Async — queue / worker
transactions (idempotency columns)Postgres
transaction_idUUID PRIMARY KEY
idempotency_keyTEXT NOT NULL, UNIQUE INDEX
statusTEXT — pending/authorized/completed/failed
response_bodyJSONB — the exact response to replay
The UNIQUE index on idempotency_key makes correctness structural at the database level: a second INSERT with the same key fails before any processor call is even attempted, and that failure is the signal to fetch and replay response_body.
  1. Client generates one idempotency key per logical charge attempt — created once, before the first network call
  2. Orchestration attempts an INSERT keyed on idempotency_key — the UNIQUE index makes a duplicate insert fail fast, before touching the processor
  3. Transaction is created in "pending" immediately — the processor hasn't confirmed anything yet, and doesn't need to for this row to exist
  4. Processor's webhook is the only thing that can move the transaction out of "pending" — and that webhook itself must be deduped by processor_event_id, since it can be retried or arrive out of order
  5. Only after a deduped, verified webhook confirms success does the ledger get a confirmed entry — never optimistically on the initial request
The client's idempotency key and the processor's event ID solve two different halves of the same problem — one protects against the client retrying the request, the other protects against the processor retrying its confirmation. Skipping either one reopens a double-charge or double-credit path.

Deep dive: double-entry ledger for multi-party settlement

Cache — hot path
Database — source of truth
  1. Never store per-account balance as a single mutable field — a 3+ party split (buyer, merchant, platform fee) has no way to be expressed as one atomic mutation across three separate columns
  2. Model one payment as multiple linked ledger entries sharing a transaction_id — debit buyer, credit merchant, credit platform fee, all inserted in a single DB transaction
  3. Enforce that every transaction's entries sum to exactly zero — this invariant is checkable and is what makes the accounting provably correct, not just probably correct
  4. Refunds and disputes become new offsetting entries, never mutated history — a refund is a fresh entry crediting the buyer and debiting the merchant, referencing the original transaction_id, so the full history stays intact for audit
  5. Derive each account's balance by summing its entries — materialize it for fast reads, but the ledger sum is always the source of truth
A naive "balance" column per account fails specifically at 3+ parties: there is no single row that can represent "these three changes are one atomic unit that must net to zero." The ledger's transaction_id grouping is what makes that invariant enforceable at write time instead of hoped for.

Deep dive: webhook reliability and reconciliation

Database — source of truth
Async — queue / worker
  1. Webhook delivery is at-least-once at best, and not even guaranteed at all — the processor's endpoint call can fail outright, arrive minutes to hours late, or be delivered more than once
  2. A timeout-based status check is the first backstop — if a transaction has sat in "pending" past a defined SLA, the system proactively polls the processor's status API instead of waiting indefinitely for a webhook
  3. A nightly reconciliation job is the second, stronger backstop — it pulls a full settlement statement/report from the processor's API, not just recent webhook events, and diffs it against internal transaction state
  4. Matches auto-resolve silently — the vast majority of transactions agree between the two sources and require no action
  5. Mismatches are flagged, and where safe, auto-corrected — e.g. a transaction confirmed on the statement but stuck "pending" internally gets its webhook effect replayed; anything ambiguous (e.g. an internal "completed" the processor has no record of) pages a human instead of guessing
Trusting the webhook stream alone means trusting that delivery is guaranteed, in-order, and exactly-once — none of which any processor actually promises. Reconciliation exists precisely because the webhook path is a convenience for latency, not a durability guarantee.

Follow-up questions

What if two charge requests arrive with the same idempotency key at the exact same moment?
The UNIQUE index on idempotency_key makes the database the arbiter — only one INSERT wins; the other fails and its handler fetches and returns the winner's stored response.
How long do you retain idempotency keys and webhook event IDs?
As long as the transaction row itself lives, since a legitimate late retry or delayed webhook after a long network partition should still resolve safely rather than be treated as new.
Why not just use optimistic locking on mutable balance columns instead of a ledger?
Optimistic locking could fix a two-party race, but it can't express a 3+ party atomic split, and it gives you no audit trail — the ledger solves both at once.
What happens when reconciliation finds a mismatch?
Auto-resolvable cases (e.g. a confirmed statement line with no local effect applied) get the missed webhook effect replayed automatically; anything ambiguous pages a human rather than guessing.
How do you handle a partial refund?
A new transaction referencing the original, with its own linked ledger entries — debit merchant, credit buyer, adjust the platform fee proportionally — never editing the original entries.
How would this design change for a bank transfer that takes days to settle instead of a card charge that confirms in milliseconds?
Same state machine and same webhook/reconciliation mechanics, just with a much longer expected "pending" window — the timeout-based status check threshold is rail-specific, not universal.
