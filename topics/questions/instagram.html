Design Instagram

Tests whether you split photo storage (write-once, read-many blob storage) from feed generation (a fan-out problem) instead of treating "Instagram" as one monolithic system.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: photo storage and feed generation are two completely separate scaling problems — storage is write-once read-many (durable blob store + CDN), feed generation is a fan-out problem
  • Feed generation is the same shape as Facebook News Feed: push (fan-out-on-write) for normal users, pull-on-read (fan-out-on-read) for celebrity accounts, merged at read time
  • Key scale to anchor on: 2B users, 100M photos/day uploaded, 500M DAU viewing feed — feed reads dwarf photo writes by orders of magnitude
  • Component most candidates forget: the upload call must return immediately — thumbnailing, CDN pre-warm, and feed fan-out all happen asynchronously via a queue
  • Failure mode most candidates skip: a celebrity with 100M followers posting — fan-out-on-write for that one post would be a catastrophic write storm, so celebrity posts are pulled at read time instead

Step 6 of 6 — Deep Dives

Deep dive: photo upload pipeline

The upload call is synchronous only up to the S3 write. Everything after that — thumbnailing, CDN propagation — is async, off a queue, and invisible to the client's latency budget.

  1. Client sends the photo in chunks — resumable multipart upload, tolerant of a flaky mobile connection
  2. Upload Svc streams the blob to S3 — the only synchronous, latency-sensitive write in the whole pipeline
  3. 202 Accepted returns to the client immediately — the original photo is already viewable on the owner's own profile at full resolution
  4. A processing job is enqueued — decouples the fast path from everything downstream
  5. Thumbnail worker generates multiple sizes and pre-warms the CDN — happens entirely in the background, typically completing well under the 2s "visible to followers" target
The key move: the upload call's latency is bounded by one S3 write, not by thumbnailing or fan-out. Anything that can happen after the client stops caring about the response belongs on the queue, not in the request path.

Deep dive: feed generation — push vs pull hybrid

Same pattern as the Facebook News Feed problem: fan-out-on-write for normal users, fan-out-on-read for celebrity accounts, merged at serve time.

Push (fan-out-on-write) — normal users, written at post time
Pull (fan-out-on-read) — celebrity accounts, fetched at view time
  1. Feed Generation Worker checks follower count — below the celebrity threshold, pushes a feed entry into every follower's cache immediately
  2. Above the threshold, the post is marked pull-only — no write fan-out happens; writing to tens of millions of caches for one post would be catastrophic
  3. Follower opens their feed — Feed Read Service reads their precomputed cache (push-based entries already there)
  4. Feed Read Service also live-queries the small set of celebrities they follow — cheap because a user follows only a handful of celebrities, not millions
  5. Both sources are merged and ranked by recency — the follower sees one unified feed, unaware of which posts were pushed vs pulled
The threshold isn't a hard science — it's tuned against write amplification: a user with 5,000 followers fanning out on write costs 5,000 cache writes; a user with 100M followers doing the same would be a full write storm across the fleet. Pick a threshold (e.g. ~1M followers) where push cost stops being negligible.

Deep dive: follow graph storage

Modeled as an adjacency list in a wide-column store, not a relational join table — because the only query that matters at billions of edges is a fast single-partition lookup.

Client / gateway
Database — source of truth
  1. Model as an adjacency list — one row per (follower, followee) edge, partitioned by follower_id
  2. "Who does X follow" becomes a single-partition read — no join, no cross-partition scan, sub-millisecond
  3. A separate table, partitioned by followee_id, answers "who follows X" — denormalized on purpose, written twice per follow action
  4. Celebrity flag stored on the followee's row — cheap to check at fan-out decision time without a separate lookup
  5. No multi-row transactions needed — a follow/unfollow is a single-partition write, naturally atomic
The relational instinct is to model this as a join table with foreign keys — but at billions of edges, joins don't help because there's exactly one query shape ("give me all edges for this key"). A wide-column store turns that into what it actually is: a fast key-value lookup, denormalized in both directions to avoid ever needing a join.

Follow-up questions

How do you decide the follower-count threshold for push vs pull?
Tune it against write amplification cost — pick the count where fan-out write volume per post starts to meaningfully strain the cache fleet, typically in the hundreds of thousands to low millions.
What happens if a user follows several celebrities — does every feed read get slow?
No — a user follows only a handful of celebrities even with a huge follow graph, so the live pull query touches a small, bounded set of accounts, not their whole feed.
How do you delete a photo and make sure it disappears everywhere, including the CDN?
Mark it deleted in Postgres immediately (stops new reads), issue a CDN cache invalidation for that key, and let feed caches skip it lazily on next read via a soft-delete check.
How would you support re-ordering the feed by engagement instead of pure recency?
Swap the sorted-set score from timestamp to a computed ranking score, recomputed periodically by a ranking worker — the cache structure doesn't change, only what "score" means.
How do you avoid unbounded growth of a user's feed cache?
Cap each per-user sorted set at a fixed size (e.g. ~800 entries) and evict the oldest on insert — old enough posts are past the point anyone scrolls to them anyway.
What's the impact of a follow-graph write during an active fan-out?
Fan-out is computed from a snapshot of followers at post time; a follow that lands mid-fan-out simply misses that one post and catches up on the next one — acceptable staleness, not a correctness bug.
