Questions

Design a Rate Limiter

Tests whether you understand that rate limiting is fundamentally a distributed counting problem — get the shared-state model wrong and every instance silently multiplies the real limit.

30-40 min interview Medium
Say this in the first 2 minutes
  • The core insight: local in-memory counters per instance are wrong at any scale >1 instance — 10 instances behind a load balancer each enforcing "100/min" independently allows 1000/min in aggregate
  • Key scale number to anchor on: gateway sees ~200k req/s across ~50 backend instances, so the limiter check must add well under 1ms and never become the bottleneck it's supposed to protect
  • Primary tradeoff being probed: token bucket (smooths bursts, allows saved-up capacity) vs sliding window (precise but costs more memory) vs fixed window (cheap but bursts at boundaries)
  • Component most candidates forget: Redis itself needs to be fast and available for every request — if it's down, decide up front whether to fail open (allow) or fail closed (deny)
  • Failure mode most candidates skip: the check-then-increment race — two concurrent requests both read count=99 under a limit of 100 and both get allowed, so the increment must be atomic, not read-modify-write in app code

Step 4 of 6 — Data Flow

Data flow

Two outcomes for the same path — the ALLOW row (mint) and the DENY row (amber) — both hit Redis, they just diverge on the response.

Redis — shared atomic counter, source of truth for the decision
Gateway + middleware — stateless, horizontally scalable
Postgres — cold rule config, polled not queried per-request
  1. Gateway authenticates the request — resolves the identifier (user ID, API key, or IP) used as the rate-limit scope
  2. Middleware builds the counter key — e.g. rl:user:4471:2026-06-30T14:22 for a fixed window, or tb:user:4471 for token bucket
  3. Atomic Redis operation executes — INCR+EXPIRE or a Lua script, resolving in under 1ms with no read-modify-write race
  4. Decision made from the returned count — under limit → forward to backend; at/over limit → short-circuit with 429
  5. Response includes Retry-After — computed from the window's remaining TTL or the bucket's refill rate, so clients can back off correctly

Step 6 of 6 — Deep Dives

Deep dive: token bucket algorithm

Tokens refill continuously into a fixed-capacity bucket; each request consumes one. This is what allows a client to burst up to the bucket size, then throttles to the steady refill rate.

  1. Lua script runs atomically in Redis — reads tokens and last_refill_ts from the hash in one round trip
  2. Compute elapsed time — now - last_refill_ts, multiplied by the refill rate (e.g. 1.67 tokens/sec for 100/min)
  3. Add refilled tokens, capped at bucket capacity — a bucket never exceeds its max, so idle time doesn't grant unlimited future bursts
  4. Check if tokens >= cost — if yes, subtract and allow; if no, deny and return time-until-next-token as Retry-After
  5. Write back tokens and timestamp — same Lua script, same round trip, no separate write call
Memory cost: one Redis hash per active identifier, ~100 bytes each. At 10M active users that's ~1GB — trivial for a Redis cluster, and idle users' keys just expire via TTL.

Deep dive: fixed window boundary burst vs sliding window

Fixed windows are cheap (one INCR per key) but let a client burst up to 2x the limit right at the window edge. Sliding window log fixes it at the cost of storing a timestamp per request.

✗ Fixed window — boundary burst
✓ Sliding window log — smooth

Implementation: a Redis sorted set per identifier, score = request timestamp. Each check does ZREMRANGEBYSCORE (drop entries older than the window) then ZCARD (count what's left) then ZADD (record this request) — all in one Lua script for atomicity. Cost: O(log n) per op and one sorted-set entry per request in the window, vs a single integer for fixed window — real memory tradeoff at high per-user limits.

Interview moves

  • 01
    "The first thing that breaks a rate limiter design is per-instance in-memory counters — with N backend nodes you're actually enforcing N times the stated limit" — names the single most common mistake before being asked
  • 02
    "The check-and-increment has to be one atomic operation, not a read followed by a write, or two concurrent requests both slip through under the limit" — shows you understand the race condition, not just the happy path
  • 03
    "I'd use token bucket for a public-facing API since it tolerates legitimate bursts, but sliding window log for something like a login endpoint where boundary bursts are a security problem" — ties the algorithm choice to a concrete requirement, not a default
  • 04
    "When Redis is down, failing open vs failing closed isn't one global answer — I'd fail open on read traffic and fail closed on anything that touches money or auth" — demonstrates the failure mode isn't binary, it's per-endpoint risk
  • 05
    "Rule config lives in Postgres but is polled into an in-memory cache every 30 seconds — the hot path never makes a database call, only Redis" — separates cold config from hot enforcement unprompted
  • 06
    "I'd put the rate limit check in the request path but everything downstream of the decision — metrics, alerting — goes through an async pipeline so it never adds latency" — reuses the hot-path/side-effect isolation pattern that shows up across every well-designed system

Follow-up questions

Why not just use a local in-memory counter per instance with sticky sessions?
It works only if every request from a given user always lands on the same instance, which load balancers don't guarantee cheaply — and it silently breaks the moment you autoscale or a node restarts.
How do you rate limit by IP when clients sit behind a shared NAT or corporate proxy?
Pure IP-based limiting punishes everyone behind that IP. Prefer API-key or user-ID scoping where possible, and treat IP-based limits as a coarser, higher-threshold safety net layered on top.
How would you support per-endpoint limits without a rule lookup on every request?
Bake the endpoint into the counter key itself (e.g. rl:user:4471:POST:/orders) and load all rules for a scope into the middleware's in-memory cache at startup/refresh, not per-request.
What's the tradeoff between fixed window, sliding window log, and sliding window counter?
Fixed window: cheapest, allows boundary bursts. Sliding log: exact, but O(n) memory per identifier. Sliding counter: approximates the log using two fixed windows weighted by time — good middle ground.
How do you rate limit across multiple regions with regional Redis clusters?
Either accept eventual consistency with async cross-region replication (limit is approximate but latency stays low), or route all counting for a given identifier to one home region at the cost of extra hops.
How would you let a client know how close they are to their limit before being denied?
Return X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset headers on every response, not just on the 429 — lets well-behaved clients self-throttle proactively.
