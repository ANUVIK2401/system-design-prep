Design a Local Delivery Service

Tests real-time geospatial matching, exactly-once state transitions, and hot-partition handling under bursty local demand — think DoorDash, Gopuff, or Instacart.

40-50 min interview Hard
Say this in the first 2 minutes
  • This is fundamentally a geospatial matching problem — finding the nearest available driver can't be a full table scan, it needs a geo-index
  • Driver location writes are extremely high frequency (every ~4s per driver) — this is a write-heavy hot path, separate from the order flow entirely
  • The order lifecycle is a strict state machine — every transition must be atomic and exactly-once, or you get double-dispatched drivers or lost orders
  • ETA is a composition of several signals (distance, traffic, prep time, driver load), not a single distance calculation
  • Lunch-rush demand concentrates geographically — I'd expect a hot-partition problem in popular neighborhoods and plan for it upfront

Why Postgres for orders: a delivery is a financial transaction with strict state and exactly-once requirements — ACID guarantees matter more than raw write throughput here. Why Redis GEO for locations: positions are ephemeral and read/written constantly — durability of any single ping doesn't matter, only the current position and query speed do.

Core design decisions

Why: geohashing turns 2D proximity into a string-prefix problem — nearby locations share a hash prefix, so "find drivers near me" becomes a cheap indexed lookup instead of computing distance to every driver.
Tradeoff: geohash cells have edge effects (two points can be physically close but in different cells) — mitigated by querying the cell plus its 8 neighbors.
Why: at ~2,500 writes/sec of transient data where only the latest position matters, an in-memory GEO index gives sub-millisecond radius queries — durability of any individual ping is irrelevant.
Tradeoff: a Redis node failure loses recent positions — acceptable since drivers re-ping every ~4s, so the gap self-heals within one ping cycle.

Why: prevents two concurrent updates (e.g. a driver marking "picked up" while a system timeout tries to cancel) from silently overwriting each other — the update only succeeds if the version matches what was read.
Tradeoff: a losing writer must retry against the new state, adding a small amount of complexity versus a naive blind UPDATE — worth it given the cost of a double-dispatch or lost transition.
Why: demand spikes are hyper-local (one office district at lunch) — pricing at the geohash-cell granularity reads local demand density and reacts to it directly, rather than overcharging an entire city for one busy block.
Tradeoff: requires a streaming aggregation pipeline (orders-per-cell over a rolling window) instead of a simple global counter — more moving parts, but far more accurate incentive signal for driver supply.
Why: a single fixed-precision geohash grid means a dense downtown cell can get hammered with far more orders and drivers than a suburban cell of the same size — dynamically using finer precision in dense areas balances load per partition.
Tradeoff: variable cell precision complicates the "query neighbors" logic slightly, since neighbor cells may be at different precision levels — solvable but adds an edge case to handle explicitly.

Deep dive: geohashing for driver matching

A city is divided into a grid of geohash cells. A driver ping lands in exactly one cell; an order query expands outward from its own cell until enough drivers are found.

  1. Driver ping lands in a cell: a GPS coordinate is encoded to a geohash string; GEOADD places the driver in that cell's index
  2. Order arrives with its own cell: the customer's location (or merchant's, for pickup) is encoded the same way
  3. Query expands outward ring by ring: check the order's own cell first, then its 8 neighbors, then the next ring out — stopping as soon as enough available drivers are found
  4. Rank candidates: among drivers found, rank by a combination of actual road distance (not just geohash proximity) and current driver load
  5. Dispatch and lock: the top candidate is offered the order with a short accept window; on timeout or decline, the next candidate is tried
A pure distance-only match can repeatedly offer orders to the same nearby driver, ignoring one who just finished a delivery two blocks further out — ranking should blend distance with idle time or fairness to avoid this.

Deep dive: order state machine

Every transition is atomic via optimistic locking. Dashed transitions are irreversible — there's no path back once crossed.

TransitionReversible?Guard condition
Placed → MatchedYes (can revert to Placed on driver decline)Matching service finds and confirms an available driver
Placed / Matched → CancelledNo — terminalCustomer or system-initiated cancellation before pickup
Matched → Picked UpNo — irreversibleDriver confirms pickup at merchant (geofence or manual tap)
Picked Up → En RouteYes (informational, not state-critical)Driver location leaves merchant geofence
En Route → DeliveredNo — terminalDriver confirms drop-off (geofence + explicit confirmation)
Once an order reaches "Picked Up," cancellation is no longer offered as a self-service option — food is already in the driver's possession. This must be enforced at the state machine level, not just in the UI, or a race condition could let a cancel request slip through after pickup.

Follow-up questions

How do you prevent two orders from being matched to the same driver simultaneously?
Optimistic locking on the driver's assignment record — the matching service's UPDATE only succeeds if the driver is still unassigned; a losing request immediately retries against the next candidate.
What happens if a driver goes offline mid-delivery?
The order enters a stalled state after a location-staleness timeout; the system either waits for reconnect within a grace period or reassigns to a backup driver, notifying the customer of the delay.
How would you detect and handle a hot geohash cell in real time, not just at lunch as a known pattern?
Stream order and driver counts per cell through a rolling window aggregator; when a cell's order-to-driver ratio crosses a threshold, trigger both surge pricing and dynamic cell-splitting for that region.
How do you avoid always sending orders to the same closest driver, starving others?
Blend distance with idle time or a fairness score in the ranking function so drivers who've been waiting longer get preference among similarly-close candidates.
How is surge pricing computed without it feeling arbitrary or exploitative to users?
Base it on a transparent, bounded multiplier tied to a real demand-density signal (orders per available driver per cell), capped at a maximum, and shown to the customer before they confirm the order.
How would this design change for scheduled (non-immediate) orders?
Scheduled orders skip immediate matching and instead enter a time-based queue that triggers the matching flow shortly before the scheduled window, reusing the same state machine and geo-index from that point on.
