Design a Local Delivery Service

Tests whether you separate a write-heavy geospatial problem (driver location) from a read-heavy one (order matching) instead of forcing both into one store.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: driver location is a write-heavy geospatial problem, order matching is a read-heavy geospatial problem — they need different stores
  • Key scale number to anchor on: 10k drivers pinging every ~4s is 2,500 location writes/sec, an entirely separate hot path from order volume
  • Primary tradeoff being probed: durability vs write speed for driver location — losing one ping is fine since the next one is 4s away
  • Component most candidates forget: the order state machine needs atomic, exactly-once transitions or you get double-dispatched drivers
  • Failure mode most candidates skip: a hot geohash cell during lunch rush — one neighborhood can get 5-10x the load of the rest of the city

Step 4 of 6 — Data Flow

Data flow

Driver location writes (blue) and order matching reads (mint) hit the same Redis GEO index but are otherwise fully separate flows.

Driver/customer apps — persistent WebSocket
Redis GEO — shared index, written by one path, read by the other
  1. Driver pings location every ~4s — over the already-open WebSocket connection
  2. Location Svc writes to Redis GEO — GEOADD updates the driver's position, TTL refreshed
  3. Customer places an order — Order Service creates the order row in "placed" state
  4. Matching Svc queries Redis GEO — finds nearby available drivers by expanding outward from the order's cell
  5. Best candidate is offered the order — pushed over their WebSocket connection

Step 6 of 6 — Deep Dives

Deep dive: geohashing for driver matching

A city is divided into a grid of geohash cells. A driver ping lands in exactly one cell; an order query expands outward until enough drivers are found.

  1. Driver ping lands in a cell — GPS coordinate encoded to a geohash string, GEOADD places the driver
  2. Order arrives with its own cell — customer/merchant location encoded the same way
  3. Query expands outward ring by ring — own cell first, then 8 neighbors, then the next ring
  4. Rank candidates — actual road distance plus current driver load, not just geohash proximity
  5. Dispatch and lock — top candidate offered with a short accept window, next candidate tried on decline
A pure distance-only match can repeatedly offer orders to the same nearby driver, ignoring one who just finished two blocks further out — ranking should blend distance with idle time to avoid this.

Deep dive: order state machine

Every transition is atomic via optimistic locking. Dashed transitions are irreversible.

TransitionReversible?Guard condition
Placed → MatchedYes (revert on driver decline)Matching service finds and confirms an available driver
Placed / Matched → CancelledNo — terminalCustomer or system-initiated cancellation before pickup
Matched → Picked UpNo — irreversibleDriver confirms pickup at merchant (geofence or manual tap)
En Route → DeliveredNo — terminalDriver confirms drop-off (geofence + explicit confirmation)
Once an order reaches "Picked Up," cancellation is no longer offered — food is already in the driver's possession. This must be enforced at the state machine level, not just in the UI, or a race condition could let a cancel request slip through after pickup.

Follow-up questions

How do you prevent two orders from being matched to the same driver simultaneously?
Optimistic locking on the driver assignment — the UPDATE only succeeds if the driver is still unassigned; a losing request immediately retries the next candidate.
What happens if a driver goes offline mid-delivery?
The order enters a stalled state after a location-staleness timeout; the system waits for reconnect within a grace period or reassigns to a backup driver.
How would you detect a hot geohash cell in real time?
Stream order/driver counts per cell through a rolling window aggregator; a crossed threshold triggers both surge pricing and dynamic cell-splitting for that region.
How do you avoid always sending orders to the same closest driver?
Blend distance with idle time in the ranking function so drivers waiting longer get preference among similarly-close candidates.
How is surge pricing computed transparently?
A bounded multiplier tied to a real demand-density signal (orders per available driver per cell), capped at a maximum, shown to the customer before confirming.
How would this change for scheduled (non-immediate) orders?
Scheduled orders skip immediate matching and enter a time-based queue that triggers matching shortly before the scheduled window, reusing the same state machine.
