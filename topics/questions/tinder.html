Questions

Design Tinder

Tests whether you treat recommendation as a geospatial + preference filter problem — narrow a city down to a few hundred candidates before any ranking model ever runs.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: candidate discovery is geospatial retrieval first, ranking second — you never run a model over an entire city's users
  • Key scale number to anchor on: 75M DAU swiping ~150 times/day each is ~130k swipes/sec at peak, but geo-filtered candidate pools per query are only a few hundred rows
  • Primary tradeoff being probed: precomputed daily candidate batches vs on-demand geo queries — batching wins on read latency, loses on freshness
  • Component most candidates forget: the swipe write path needs idempotency — a retried swipe must never double-count or trigger a duplicate match
  • Failure mode most candidates skip: a match is a two-writer race — both users can swipe right in the same instant, so match detection needs an atomic check, not a read-then-write

Step 4 of 6 — Data Flow

Data flow

Deck read path (mint) pulls from a precomputed candidate cache; swipe write path (blue) records the action and checks for a match inline.

Deck read — precomputed batch first, live geo fallback
Swipe write — logged then checked for a match inline
Match — only fires on a detected mutual like
  1. Client requests a deck — Deck Svc first checks the precomputed candidate cache for this user
  2. Cache hit (common case) — batch was built overnight by the recommendation pipeline, returned in low tens of ms
  3. Cache miss (new/relocated user) — falls through to a live Redis GEO radius query, expanding rings until enough candidates are found
  4. Client swipes right — Swipe Svc writes the row to Cassandra, keyed by (swiper_id, swipee_id)
  5. Inline match check — Swipe Svc reads for the reverse pair; if it's also a like, Match Svc commits a Match row and pushes to both sockets

Step 6 of 6 — Deep Dives

Deep dive: geo-filter before you rank

A city is divided into geohash cells. Candidate discovery filters by cell first, shrinking the pool to a few hundred rows — only then does a ranking model run.

  1. User location lands in a cell — GPS coordinate encoded to a 6-char geohash, GEOADD places the user in Redis GEO
  2. Nightly batch job runs per user — starts from their home cell and expands ring by ring until ~2,000 candidates are collected
  3. Hard filters applied first — age range, gender preference, max distance, already-swiped exclusion — all cheap boolean checks on the geo-filtered set
  4. Lightweight ranking model scores the survivors — activity recency, mutual-interest signals, photo quality — runs over hundreds, never millions, of rows
  5. Top ~150 written to the candidate cache — this is exactly what Deck Svc serves the next day, no per-request ranking cost
Running the ranking model before the geo filter is the classic mistake: scoring every user in a metro against every other user is O(n²) and will never fit a nightly batch window, let alone a <200ms request. Geo-filter to a few thousand candidates first — ranking only ever touches that shrunk set.

Deep dive: swipe state machine and match race

Every profile a user sees moves through seen → liked/passed, and a match only fires when both directions resolve to "liked."

Seen — no write yet, purely a "don't show again" flag
Matched — only reachable via an atomic reverse-pair check
TransitionReversible?Guard condition
Seen → Liked / PassedNo — a swipe is final for that pairIdempotency key on the swipe row prevents a client retry from double-recording
Liked → Pending CheckN/A — internal stepSwipe Svc reads Cassandra for the (swipee_id, swiper_id) reverse row inline, same request
Pending Check → MatchedNo — terminalReverse row exists and is "like" — Match Svc does a conditional INSERT on the sorted pair, unique constraint rejects a duplicate
The match race is the sharpest edge case in this system: both users can swipe right within milliseconds of each other, so two concurrent requests may both detect "the other side already liked me." The unique constraint on the sorted (user_a_id, user_b_id) pair in Postgres is what prevents two Match rows — the losing INSERT fails cleanly and that request just treats it as "match already exists," same outcome delivered to the user.

Interview moves

  • 01
    "Before I design ranking, let me separate candidate discovery from candidate scoring — geo-filter to a few thousand users first, then only rank that shrunk set" — states the core insight unprompted, prevents an O(n²) design
  • 02
    "I'd precompute each user's daily candidate batch offline rather than run the recommendation model on every deck request — that's the only way to hit <200ms at 75M DAU" — justifies a caching decision with a concrete latency number
  • 03
    "A swipe needs an idempotency key — a mobile client retrying a failed request must never double-write or trigger a phantom match check" — shows awareness of real-world client behavior, not just the happy path
  • 04
    "Match detection is a two-writer race — I'd enforce it with a unique constraint on the sorted user pair rather than a read-then-write check, so it's atomic at the database level" — names the exact concurrency mechanism, not a vague "handle it"
  • 05
    "Swipes and profile edits have wildly different access patterns — 130k writes/sec of append-only swipes go to Cassandra, low-frequency strongly-consistent profile data stays in Postgres" — demonstrates picking storage per access pattern, not one store for everything
  • 06
    "If the candidate cache is cold for a brand-new signup, I'd trigger an on-demand mini batch rather than make them wait for the next nightly run" — proactively covers a cold-start edge case most candidates miss

Follow-up questions

How do you prevent showing the same profile twice in one session?
The candidate batch excludes anyone already in the user's Swipe table; "seen" state is also cached client-side for the current session to avoid a re-fetch showing stale entries.
How would you handle a user who travels to a new city?
Location POST refreshes the Redis GEO key immediately; if the candidate cache has no batch for the new region, trigger an on-demand mini geo-filter job instead of waiting overnight.
How do you scale the "who liked me" feature given Swipe is keyed by swiper_id?
Maintain a second denormalized table keyed by swipee_id, written asynchronously off the same swipe event — trades a bit of write amplification for a fast reverse lookup.
How would you rate-limit or throttle abusive swiping behavior?
A daily swipe cap enforced at Swipe Svc, backed by a Redis counter with a 24h TTL — cheap to check inline before the write, resets automatically.
Why not rank candidates in real time instead of precomputing batches?
Real-time ranking over millions of geo-filtered users per request can't hit <200ms at 75M DAU; precomputing trades some freshness for a latency floor the live path can't reach.
How do you extend this to support video profiles or stories?
Store media in S3/CDN as with photos, keep only URLs in Profile — the deck-serving and matching logic is entirely unaffected since it never touches media bytes directly.
