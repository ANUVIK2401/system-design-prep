Questions

Design an Ad Click Aggregator

Tests whether you can build a real-time stream-aggregation pipeline with financial-grade exactly-once correctness at extreme write volume, not just a generic "count things with Kafka" answer.

45-55 min interview Hard
Say this in the first 2 minutes
  • The core insight: this is a real-time stream-aggregation problem where correctness is a financial requirement, not an analytics nicety — advertisers are billed off these numbers, so double- or under-counting is a bug, not noise
  • Key scale number to anchor on: ~100B events/day is ~1.2M events/sec average, with peak traffic (evening, campaign launches) running several times higher — this rules out any single-node aggregation
  • Primary tradeoff being probed: real-time streaming aggregates are fast but approximate under failure/replay; you need a batch reconciliation path against the immutable raw log to guarantee exactness
  • Component most candidates forget: a dedup key (event_id) at ingestion — Kafka only guarantees at-least-once delivery, so without idempotent aggregation a retried producer double-counts a click
  • Failure mode most candidates skip: late-arriving events after a window has "closed" — watermarks and a grace period are required or you silently undercount every window boundary

Step 4 of 6 — Data Flow

Data flow

Ingestion (amber, async) and dashboard reads (mint) share the aggregate store but are otherwise independent — one writes continuously, the other reads on demand.

Ad server + dashboard — entry points
Kafka + stream processor — continuous async write path
Aggregate store — shared, written by ingestion, read by dashboard
  1. Ad server emits an event — every click/impression gets a unique event_id generated server-side
  2. Event lands on Kafka — partitioned by ad_id so all events for one ad are processed in order by one consumer
  3. Stream processor aggregates — dedups by event_id, folds the event into its 1-minute window's running counts
  4. Aggregate store is upserted — per-(ad_id, window_start) row updated in place, idempotent on replay
  5. Dashboard reads the aggregate — query service serves the latest window, labeled approximate until reconciled

Step 6 of 6 — Deep Dives

Deep dive: exactly-once stream aggregation

Kafka only guarantees at-least-once delivery — a producer retry or consumer rebalance can redeliver the same event. Correctness comes from making the aggregation idempotent, not from eliminating redelivery.

Kafka — source of at-least-once redelivery
Dedup check + window state — hot-path processing
Checkpoint + upsert sink — durable, idempotent by key
  1. Event arrives, possibly a redelivery — Kafka's at-least-once guarantee means this is expected, not exceptional
  2. Dedup check against event_id — a bounded window of recently-seen ids (backed by the checkpointed state) catches duplicates
  3. New events apply to the window — folded into the running count for that (ad_id, window_start)
  4. State is checkpointed periodically — on a crash, the processor restores from the last checkpoint and replays only what's since it
  5. The sink itself is upsert-based — even if a duplicate slips past dedup, writing the same (key, value) twice is a no-op, not a double-count
Two independent layers of defense: dedup at the processing stage catches most duplicates cheaply, and an idempotent upsert sink keyed by window makes the rare slip-through harmless — correctness doesn't depend on either one being perfect alone.

Deep dive: windowed aggregation and watermarks

Events are grouped into fixed 1-minute tumbling windows. A watermark tracks how far event-time has progressed, and a grace period decides how long a window stays open for stragglers before it's considered final.

Tumbling 1-minute windows — non-overlapping, fixed size
  1. Events are bucketed by event-time — the timestamp the ad server assigned, not when the processor receives it
  2. Watermark tracks progress — an estimate of "no more events earlier than this will arrive," derived from observed event-time lag
  3. A window closes when the watermark passes its end — triggering emission of that window's aggregate
  4. A grace period keeps it updatable briefly after — late events within the grace window still get folded in via the idempotent upsert
  5. Events later than the grace period are dropped or routed to reconciliation — the batch job is what ultimately catches these instead
Grace period is a direct latency-vs-completeness knob: a longer grace period catches more late/network-delayed events but delays when a window is considered "final" for the dashboard — a 0s grace period is fastest but systematically undercounts every window by its straggler rate.

Deep dive: streaming vs batch reconciliation

Two paths compute the same numbers on different timelines: streaming is fast but can drift under replay, rebalance, or a dedup edge case; batch is slow but reads the immutable log directly, so it's always exact.

Streaming: Kafka + processor — fast, continuous, approximate
Batch: raw log + reconciler — slow, exact, overwrites in place
  1. Streaming produces a live number within seconds — good enough for a dashboard graph, labeled approximate
  2. Every raw event is also durably archived — the same event that feeds Kafka lands in the immutable raw log
  3. A batch job runs on a schedule — hourly or nightly, reading directly from the raw log, independent of any streaming state
  4. It recomputes the exact aggregate per window — no dedup ambiguity, no checkpoint edge cases, just a full deterministic recompute
  5. The reconciled value overwrites the streaming value — same store, same key, marked approximate: false; billing only ever reads this value
Why not just pick one: streaming alone can silently drift — a rare checkpoint-recovery edge case or a dedup-window eviction under extreme skew can under- or over-count a window, and you'd never know without an independent check. Batch alone is correct but hours stale, useless for a live dashboard. You need both: streaming for speed, batch as the financial-grade backstop.

Follow-up questions

How do you handle a stream processor crashing mid-window?
Checkpointed state is restored from the last durable snapshot; the processor replays events since that checkpoint, and idempotent upserts make any reprocessed events a no-op.
What if the same event_id somehow gets two different aggregate values applied?
It shouldn't under correct upsert semantics — the sink is keyed by window, not by event, so applying the same event twice recomputes the same delta, not an additional one.
How do you choose the grace period length?
Measure observed event-time lag (p99 network/ad-server delay) and set the grace period just past that — too short undercounts stragglers, too long delays dashboard finality.
Why not just make the batch job the only source of truth and skip streaming?
Advertisers expect near-real-time spend visibility to manage budgets live; a batch-only system with hours of lag fails that requirement even though it's simpler.
How would you detect bot or fraudulent click traffic?
A separate fraud-detection pipeline scores events on signals like click velocity per IP/device and click-through anomalies, then flags or excludes them before billing — decoupled from core counting.
How does this scale if event volume grows 10x?
Add Kafka partitions and stream-processor parallelism first (cheap, linear), then shard the aggregate store by ad_id range before considering edge pre-aggregation.
