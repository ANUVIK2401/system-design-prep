Design Bitly

Tests whether you can hit <10ms global read latency on a read-heavy system while treating 301 vs 302 as a product decision, not a coin flip.

35-45 min interview Medium
Say this in the first 2 minutes
  • The core insight: 301 vs 302 is a business decision, not a technical one — it trades browser-cached speed against click analytics visibility
  • Key scale number to anchor on: peak redirect traffic can hit 1M RPS on a single viral link, far above steady-state
  • Primary tradeoff being probed: strong click-count consistency vs an async pipeline that never touches the redirect's critical path
  • Component most candidates forget: CDN edge caching in front of Redis — Redis alone doesn't survive a true viral spike
  • Failure mode most candidates skip: what happens when the Kafka analytics pipeline backs up — it must degrade silently, never block a redirect

Requirements

Functional

Non-Functional

  • 99.99% availability for redirects (writes can tolerate less)

What NOT to build: a full marketing analytics suite (funnels, cohorts, A/B test integration), team/workspace collaboration features, or link-in-bio landing pages. This is shorten + redirect + click counting only.

Step 4 of 6 — Data Flow

Data flow

Read path — cache + CDN first
Write path — low volume, strong uniqueness
Analytics — fully decoupled, fire-and-forget
  1. Client requests /{short_code} — hits CDN edge first
  2. CDN hit — returns cached 302, origin never touched
  3. CDN miss → Redirect Svc → Redis — checks hot mapping cache
  4. Response sent to client — 302 issued immediately once the mapping is resolved
  5. Click event fired to Kafka — happens after the response is already sent, never adds latency

Step 6 of 6 — Deep Dives

Deep dive: Snowflake ID generation

1sign
41timestamp (ms)
10node ID
12sequence
1 bit unused, keeps ID positive
41 bits — ms since custom epoch, ~69 years
10 bits — up to 1024 nodes
12 bits — up to 4096 IDs/node/ms
  1. Read current timestamp — ms since a custom epoch, not Unix epoch
  2. Attach node ID — fixed per generator instance at boot
  3. Increment sequence — local counter reset every millisecond
  4. Bit-pack and return — no network call, no lock contention
  5. Base62-encode for the short code — the 64-bit int becomes the human-facing code
Clock drift is the real failure mode: if a node's clock moves backward, it must refuse to generate IDs until it catches up, or two different timestamps could produce a colliding ID.

Deep dive: cache hit vs miss

✓ Cache hit — ~1-3ms
✗ Cache miss — ~15-30ms

Deep dive: analytics pipeline never blocks the redirect

  1. Fire-and-forget emit — the redirect service publishes a click event after already sending the 302, non-blocking
  2. Kafka buffers the burst — a viral link spiking to 100k clicks/sec just queues, doesn't overwhelm consumers
  3. Consumer batches writes — analytics consumer bulk-inserts into ClickHouse every 1-5s instead of row-by-row
  4. ClickHouse aggregates on read — dashboards query grouped counts, exactly what a column store is built for

Interview moves

  • 01
    "Before I design anything, let me clarify the read/write ratio — this is roughly 1000:1 read-heavy which changes the entire caching strategy" — anchors every downstream decision to a stated number
  • 02
    "302 vs 301 here is a business decision, not a technical one — 301 would get cached by the browser and we'd lose click visibility" — ties a technical choice directly to a stated requirement
  • 03
    "I'll decouple click tracking into Kafka so a slow analytics write never adds latency to the redirect's critical path" — demonstrates instinct for isolating hot paths from side effects
  • 04
    "Custom aliases go into a separate table from generated codes so a user-chosen alias can never silently collide with a Snowflake-derived one" — a subtlety most candidates miss entirely
  • 05
    "At 1M peak RPS on a single viral link, even Redis needs help — I'd lean on CDN edge caching so most of that traffic never reaches our infrastructure" — shows caching has layers, not just one tier
  • 06
    "If the Kafka pipeline backs up, I'd rather drop click events than let the redirect service block waiting on it" — states a concrete, correct failure-mode priority unprompted

Follow-up questions

How do you prevent short code enumeration attacks?
Rate limit per IP/user at the gateway. Base62 with 7+ chars gives 62^7 ≈ 3.5 trillion combinations, making brute-force scanning impractical.
What if the same long URL is submitted by two different users?
Each gets a distinct short code by default. Dedup by hashing long_url would save storage but adds a lookup on every write and complicates per-user ownership.
How would you extend this to support 10x the scale?
Shard Postgres by short_code prefix, extend the code to 8+ characters, add more Snowflake node IDs, make CDN edge caching mandatory for all traffic.
How do you handle a malicious long URL (phishing, malware)?
Check against a Safe Browsing API synchronously at shorten time, re-scan periodically, and allow user reports to flag and auto-disable a short code.
Why not just use a DB auto-increment given the write QPS is relatively low?
It works at 12k/s on a single writer but is a single point of contention with no horizontal scaling path — Snowflake removes that ceiling for a small added operational cost.
How do you keep the CDN and Redis cache consistent when a URL is deleted?
Short TTL (~60s) on both as a safety net, plus an explicit purge/invalidate call fired on delete rather than relying on TTL expiry alone.
