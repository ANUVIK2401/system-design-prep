Design Bitly

Tests whether you can hit <10ms global read latency on a read-heavy system while keeping the write path collision-free at massive scale.

35-45 min interview Medium
Say this in the first 2 minutes
  • This is read-heavy at roughly 1000:1 — every decision downstream optimizes the redirect path, not the shorten path
  • ID generation is the actual hard problem — not the redirect logic, which is a one-line cache lookup
  • I'd pick Snowflake IDs over hashing or a DB counter to avoid both collisions and a single-writer bottleneck
  • 301 vs 302 is a real tradeoff between browser caching and analytics visibility, not a coin flip
  • Click analytics must never sit on the redirect's critical path — that goes through Kafka, fully async

Data model

-- Postgres: source of truth, indexed for O(1) short_code lookup
CREATE TABLE urls (
  id          BIGINT PRIMARY KEY,           -- Snowflake ID (64-bit)
  short_code  VARCHAR(10) UNIQUE NOT NULL,  -- base62(id) — e.g. "gH3xK2p"
  long_url    TEXT NOT NULL,
  user_id     BIGINT,                       -- nullable, anonymous allowed
  is_custom   BOOLEAN DEFAULT FALSE,        -- true = user-chosen alias
  created_at  TIMESTAMP DEFAULT NOW(),
  expires_at  TIMESTAMP                     -- null = never expires
);
CREATE UNIQUE INDEX idx_short_code ON urls(short_code);   -- redirect hot path

-- Separate namespace: prevents a custom alias colliding with a Snowflake-derived code
CREATE TABLE custom_aliases (
  alias       VARCHAR(50) PRIMARY KEY,
  url_id      BIGINT REFERENCES urls(id)
);

-- ClickHouse: append-only, wide, optimized for aggregation not point lookups
CREATE TABLE click_events (
  short_code  String,
  ts          DateTime,
  country     LowCardinality(String),
  referrer    String,
  user_agent  String
) ENGINE = MergeTree ORDER BY (short_code, ts);
Why Postgres for URLs: low volume of writes relative to reads (12k/s vs 120k/s), strong uniqueness constraint needed on short_code, and rows are small — a single primary with read replicas comfortably handles 300M rows (~180GB/yr). Why ClickHouse for clicks: billions of append-only rows/day where the query pattern is "count/group by," which is exactly what a column store is built for.

Deep dive: Snowflake ID generation

A 64-bit integer split into fields so IDs are unique across every generator node without a central counter or DB round-trip.

1sign
41timestamp (ms)
10node ID
12sequence
1 bit unused (always 0, keeps ID positive)
41 bits — ms since custom epoch, good for ~69 years
10 bits — up to 1024 generator nodes
12 bits — up to 4096 IDs per node per millisecond
  1. Read current timestamp: milliseconds since a custom epoch (e.g. service launch date, not Unix epoch — buys more usable years)
  2. Attach node ID: fixed per generator instance at boot (e.g. derived from a Kubernetes pod ordinal or ZooKeeper-assigned slot)
  3. Increment sequence: a local counter reset every millisecond, allowing up to 4096 IDs per node without collision in the same tick
  4. Bit-pack and return: no network call, no lock contention across nodes — the ID is unique by construction
  5. Base62-encode for the short code: the resulting 64-bit int is converted to base62 to produce the human-facing 7-11 character code
Clock drift is the real failure mode: if a node's clock moves backward (NTP correction), it must refuse to generate IDs until it catches up, or two different timestamps could produce a colliding ID.

Deep dive: cache strategy — hit vs miss

The entire <10ms p99 target depends on the vast majority of redirects resolving in the top branch.

✓ Cache hit — ~1-3ms
✗ Cache miss — ~15-30ms
At Bitly's scale, cache hit rate needs to stay above ~95% — a small fraction of links (the viral ones) drive the majority of traffic, so LRU eviction naturally keeps the hot set in Redis.

Deep dive: analytics pipeline

Counting a billion clicks a day without ever touching the redirect's response time.

  1. Fire-and-forget emit: the redirect service publishes a click event to Kafka after already sending the 302 response — the publish is non-blocking
  2. Kafka buffers the burst: a viral link spiking to 100k clicks/sec doesn't overwhelm downstream consumers — it just queues
  3. Consumer batches writes: the analytics consumer reads in batches (e.g. every 1-5s or every N events) and bulk-inserts into ClickHouse rather than one row at a time
  4. ClickHouse aggregates on read: dashboards query pre-aggregated or on-the-fly grouped counts — this is what column stores are built for
  5. Denormalized click_count on the urls row: updated asynchronously from the same consumer for a cheap "total clicks" read without hitting ClickHouse on every profile view

Follow-up questions

How do you prevent short code enumeration attacks (someone scanning all possible codes)?
Rate limit per IP/user at the gateway. Base62 with 7+ chars gives 62^7 ≈ 3.5 trillion combinations, making brute-force scanning economically infeasible even without rate limiting.
What happens if the same long URL is submitted by two different users?
Each gets a distinct short code by default — dedup by hashing long_url would save storage but adds a lookup on every write and complicates per-user ownership and expiration.
How would you extend this to support 10x the scale?
Shard Postgres by short_code prefix, extend the code to 8+ characters, add more Snowflake node IDs, and make CDN edge caching mandatory rather than optional for all traffic.
How do you handle a malicious long URL (phishing, malware)?
Check against Google Safe Browsing (or similar) synchronously at shorten time, and re-scan periodically; allow user reports to flag and auto-disable a short code.
Why not just use the DB's auto-increment for IDs given the write QPS is relatively low?
It would work at 12k/s on a single writer, but it's a single point of contention and doesn't scale horizontally — Snowflake costs little extra complexity and removes that ceiling entirely.
How do you keep the CDN cache and Redis cache consistent when a URL is deleted?
Set a short TTL (~60s) on both layers as a safety net, and fire an explicit purge/invalidate call to the CDN and a Redis DEL on delete rather than relying on TTL alone.
