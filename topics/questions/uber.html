Design Uber

Tests whether you split a write-heavy geospatial stream (driver location) from a read-heavy geospatial query (ride matching) — and whether you can reason about ETA and surge pricing as their own deep problems, not afterthoughts.

45-55 min interview Hard
Say this in the first 2 minutes
  • The core insight: driver location is a write-heavy geospatial stream, ride matching is a read-heavy geospatial query — different access patterns need different stores and different optimization strategies
  • Key scale number to anchor on: 3M active drivers pinging every ~4s is 750k location writes/sec at peak — an entirely separate hot path from the 5M trips/day it eventually feeds
  • Primary tradeoff being probed: durability vs write speed for driver location — a dropped ping is irrelevant since a fresher one lands in 4s
  • Component most candidates gloss over: ETA is a live composition of routing, traffic, and driver behavior, not a single distance formula — and surge pricing is a separate signal fed by the same demand data
  • Consistency model most candidates get wrong: trip state needs strong consistency (no double-dispatch), but surge pricing is fine with data that's a few seconds stale — don't apply one consistency bar to both

Step 4 of 6 — Data Flow

Data flow

Driver location writes (blue) and ride-matching reads (mint) hit the same Redis GEO index but are otherwise fully independent flows.

Driver/rider apps — persistent WebSocket for location, REST for requests
Redis GEO — one index, written by the location path, read by the matching path
  1. Driver pings location every ~4s — over the already-open WebSocket connection
  2. Location Svc writes to Redis GEO — GEOADD refreshes position and TTL, also streams to Kafka for ETA and surge aggregation
  3. Rider requests a trip — Trip Service creates the trip row in "requested" state, reads the current surge multiplier
  4. Matching Svc queries Redis GEO — finds nearby available drivers by expanding outward from the rider's geohash cell
  5. Best candidate is offered the trip — pushed over their WebSocket connection with a 15s accept window

Step 6 of 6 — Deep Dives

Deep dive: geohash grid for ride matching

A city is divided into a grid of geohash cells. A driver ping lands in exactly one cell; a ride request expands outward ring by ring until enough available drivers are found.

  1. Driver ping lands in a cell — GPS coordinate encoded to a geohash string, GEOADD places the driver at that key
  2. Ride request arrives with its own cell — rider's pickup location encoded the same way
  3. Query expands outward ring by ring — own cell first, then the 8 neighbors, then the next ring, until the candidate count is enough
  4. Rank candidates — actual road distance and current driver idle time, not just geohash proximity
  5. Offer and lock — top candidate gets a 15s accept window; next candidate tried on decline or timeout
Geohash precision is a tradeoff, not a constant: 7 characters gives ~150m cells, tight enough for dense urban matching but requiring more ring expansions in sparse suburbs. Fewer characters (coarser cells) reduce ring-expansion cost but blur "nearby" into cells that can be a kilometer wide — precision should flex with local driver density, not be fixed citywide.

Once a trip reaches "Arrived Pickup," cancellation is no longer offered to the rider without a fee — the driver has already committed travel time and fuel. This must be enforced at the state machine level, not just hidden in the UI, or a race condition could let a free cancellation slip through after the driver has arrived.

Deep dive: surge pricing pipeline

Demand density per geohash cell — open ride requests divided by available drivers — drives a pricing tier multiplier. This signal can be a few seconds stale; trip state cannot.

Kafka + aggregator — continuous streaming computation
SurgeCell in Redis — cheap to read, tolerant of a few seconds of lag
Trip Service — the one place the multiplier becomes strongly consistent, frozen at request time
  1. Every location ping and ride request streams to Kafka — the same event stream that feeds ETA also feeds surge
  2. Density Aggregator maintains rolling counters per cell — open_requests and available_drivers, refreshed on a ~5s window
  3. Ratio crosses a threshold — e.g. requests-to-drivers ratio above 1.5 triggers the next pricing tier for that cell
  4. Trip Service reads the multiplier at request time — and freezes it into the trip row so the price shown to the rider never changes mid-trip
  5. Multiplier decays as supply catches up — the same rolling window that triggered surge naturally relaxes it
Why staleness is fine here: surge pricing only needs to be "roughly right, updated frequently" — a multiplier computed on data that's 3-5 seconds old still reflects real demand pressure, and the cost of being briefly wrong (slightly over- or under-priced) is far lower than the cost of making every rider request block on a strongly consistent citywide count. Contrast with trip state, where a stale read could double-dispatch a driver — that's why trip transitions use Postgres with optimistic locking while surge uses eventually-consistent rolling counters in Redis.

Follow-up questions

How do you prevent two riders from being matched to the same driver simultaneously?
Optimistic locking on the driver assignment — the UPDATE only succeeds if the driver is still unassigned; a losing request immediately retries the next candidate.
What happens if a driver's app crashes mid-trip?
A timeout on state transitions and location updates flags the trip for ops review; an on-call SOP contacts both parties and force-resolves the trip to Cancelled or Completed based on available evidence.
Why is it safe for surge pricing to use stale data when trip state can't be?
Surge is a pricing signal where being briefly wrong just means a slightly off multiplier; trip state staleness can cause a double-dispatch, which is a hard correctness violation, not a soft one.
How would you detect a driver who appears available but has actually lost connection?
A TTL-based heartbeat on the driver's Redis GEO key — if it's not refreshed within ~15s, the key expires and the driver silently drops out of the matching pool.
What's the fallback if the ETA service is slow or down?
A circuit breaker trips and the request flow returns a cached or static distance-based estimate immediately rather than blocking the rider on a live computation.
How would you cap surge so it doesn't feel exploitative?
A hard multiplier ceiling per tier, shown transparently to the rider before they confirm, with the multiplier frozen into the trip at request time so it can't change mid-ride.
