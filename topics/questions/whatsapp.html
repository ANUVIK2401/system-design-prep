Questions

Design WhatsApp

Tests whether you can guarantee at-least-once, in-order message delivery over millions of long-lived connections without losing a single message when a recipient is offline.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: delivery guarantees (at-least-once) and per-conversation ordering are the hard problems here — not the chat UI
  • Key scale number to anchor on: ~150M daily active users, each holding one long-lived WebSocket connection concurrently at peak
  • Primary tradeoff being probed: strong per-message delivery guarantees vs the cost of holding millions of persistent connections in memory
  • Component most candidates forget: a durable per-user offline queue — messages must survive a dead connection, not just a dead request
  • Failure mode most candidates skip: what happens when a connection service instance dies mid-session with thousands of sockets attached to it

Step 4 of 6 — Data Flow

Data flow

Send path (blue) writes durably before acking; receive path (mint) fans out to whichever connection servers hold the recipient's live sockets.

Send path — durable write before ack
Receive path — session-routed fan-out
Offline — queued, delivered on reconnect
  1. Sender writes over WebSocket — Connection Svc forwards the frame to Message Svc
  2. Message Svc assigns an ordered ID and writes to Cassandra — the message is not acked as "sent" until this commits
  3. Session Router looks up the recipient — checks which connection server, if any, holds their live socket
  4. Online: direct push — the owning connection server writes the frame straight down the socket, no polling
  5. Offline: queued instead — message ID pushed onto the recipient's Redis list; a push notification is fired so the OS wakes the app

Step 6 of 6 — Deep Dives

Deep dive: delivery receipt state machine

Every message moves through three states, driven by explicit client acknowledgments — never inferred silently by the server.

  1. Sent — set the moment Message Svc's write to Cassandra commits, before any fan-out happens
  2. Delivered — recipient's Connection Svc pushes the frame and the client sends an application-level ACK back over the same socket
  3. Read — fired only when the recipient actually opens that conversation thread, a separate explicit client action from delivery
  4. Receipt fan-out to sender — each transition is itself a small message routed back to every one of the sender's active devices
  5. Multi-device reconciliation — a device that was offline during a transition catches up via its stored sync cursor rather than replaying every intermediate receipt
Read receipts must never be inferred from delivery — a message can be pushed to a locked phone in someone's pocket. Treating "delivered" as "read" is the single most common correctness bug candidates propose here.

Deep dive: offline message queue and reconnect flow

A message sent to an offline user must be durably queued, not dropped — and delivered with receipts firing correctly once the recipient reconnects.

Interview moves

  • 01
    "The hard part here isn't chat UI, it's guaranteeing at-least-once delivery with per-conversation ordering across connects, disconnects, and multiple devices" — names the actual problem before diving into components
  • 02
    "I'd never ack a send back to the sender until the message is durably written to Cassandra — the WebSocket ack and the durability guarantee are two separate events" — shows the write path is correctness-first, not latency-first
  • 03
    "Delivered and read are two distinct, explicit client actions — I will not infer 'read' from 'delivered,' that's the most common correctness bug in this design" — preempts a classic interviewer probe
  • 04
    "Connection Svc is the one genuinely stateful tier in this system — everything else stays stateless so it scales trivially behind a load balancer" — demonstrates the architecture isn't uniformly stateless out of habit
  • 05
    "Redis holds the offline queue for speed, but Cassandra is always the fallback source of truth if that queue is ever lost — I never treat the cache as durable" — states a clear durability boundary unprompted
  • 06
    "At-least-once means the client has to dedupe by message_id — I'd rather redeliver a message the client already has than risk losing one" — picks the correct side of the tradeoff and defends it

Follow-up questions

How do you handle a user with two devices connected simultaneously?
Both devices get their own socket and their own session entry; the Message Svc fans out to every active device, and each tracks its own sync cursor independently.
Why Cassandra over Postgres for the message log?
Write volume (~460k/s) and the append-only, partition-scoped access pattern favor a wide-column store; Postgres would need heavy sharding to match that write throughput.
How do you prevent a message from being lost if Connection Svc crashes mid-delivery?
The message is already durably written to Cassandra before any push attempt, so a crashed connection server just means the client reconnects and the normal offline-drain path re-delivers it.
How would you add group chat on top of this design?
Fan out the same send path to N recipients instead of one, and change conversation_id semantics to a group ID — the durability, ordering, and offline-queue mechanics stay identical.
How do you scale to 10x the concurrent connections?
Add more Connection Svc instances (horizontal, bounded by memory per socket), shard the Redis session table further, and geo-partition connection clusters by region.
What happens if the sender and recipient are on the same Connection Svc instance?
Still routes through Message Svc for the durable write and ID assignment — no shortcut bypassing Cassandra, since correctness must hold even when this coincidence doesn't.
