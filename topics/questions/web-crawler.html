Questions

Design a Web Crawler

Tests whether you treat the URL frontier and dedup index as the hard distributed-systems problems they are, instead of hand-waving "we just fetch pages with HTTP."

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: this is a distributed BFS over the web, constrained by politeness — the frontier and the dedup index are the hard data-structure problems, not the HTTP fetching
  • Key scale number to anchor on: 10B pages to crawl, ~5,000 pages/sec sustained across the fleet, which is fetch-worker-bound, not bandwidth-bound
  • Primary tradeoff being probed: dedup accuracy vs memory — a naive hash-set of 10B seen URLs doesn't fit in memory, so a Bloom filter trades a tunable false-positive rate for a 20-30x memory win
  • Component most candidates forget: politeness — the frontier must be sharded/queued per domain so the fetcher fleet never hammers one site, no matter how parallel it is
  • Failure mode most candidates skip: a crawler trap — a site that generates near-infinite URLs via query parameters or calendar links can consume the entire fetch budget if not capped

Step 4 of 6 — Data Flow

Data flow

Write path (blue) is a fetch-and-parse cycle that feeds itself — new links flow back into the frontier (amber). Read path (mint) is a page lookup, fully decoupled from crawling.

Frontier — politeness-gated queue
Content store + Postgres metadata
Downstream consumers — fully decoupled from crawling
  1. Fetcher claims a URL — only from a domain whose politeness delay has already elapsed
  2. Fetcher retrieves the page — respects robots.txt disallow rules pulled from the cached DomainPolicy
  3. Parser extracts links and normalizes them — strips tracking params, resolves relative paths to absolute URLs
  4. Each extracted URL is checked against the Bloom filter — likely-seen URLs are dropped before touching the frontier
  5. Genuinely new URLs are pushed back into the frontier — this is what makes the crawl a self-feeding BFS instead of a fixed list

Step 6 of 6 — Deep Dives

Deep dive: URL frontier and politeness

The frontier isn't one global queue — it's effectively one priority queue per domain, and the scheduler will only ever pop from a domain that's past its politeness delay.

  1. Each domain gets its own logical queue — scored by next_allowed_time, the earliest moment a fetch is allowed again
  2. Scheduler merges all per-domain queues — but only exposes entries whose domain's time has already passed
  3. Fetcher claims one URL — via the claim endpoint, never by reading the frontier directly
  4. Domain's next_allowed_time is pushed forward — by the domain's crawl_delay (from DomainPolicy), immediately after the claim
  5. Blocked domains simply wait — no matter how many fetcher workers are idle, a domain inside its delay window yields nothing
Politeness has to be enforced at the frontier/scheduler level, not left to individual fetcher workers — if each worker independently decided "am I allowed to hit this domain," a large fleet would still race past the limit in aggregate before any single worker noticed.

Deep dive: dedup at scale with Bloom filters

A naive hash-set of 10B seen URLs at ~64 bytes/entry (hash + overhead) is well over half a terabyte — too large to keep resident for a sub-millisecond membership check on every discovered link.

Sized from the standard formulas m = -(n · ln p) / (ln 2)² and k = (m/n) · ln 2. At n=10B and p=0.1%, that's m ≈ 1.44×10¹¹ bits ≈ 18GB — versus roughly 640GB to store 10B full 64-byte hash-set entries. Close to a 35x memory reduction for a 1-in-1000 chance of wrongly treating a genuinely new URL as already seen.

  1. Candidate URL hashed by k independent functions — each sets/checks one bit position in the shared bit array
  2. Any unset bit means definitely not seen — the vast majority of genuinely new URLs resolve here in O(k), no external lookup needed
  3. All bits set means maybe seen — this is the only case that pays for a definitive lookup, keeping the expensive path rare by design
  4. Durable store gives the authoritative answer — a Postgres unique-index lookup on url_hash confirms or refutes the match
  5. A false positive silently drops one new URL — accepted at a fixed, tunable rate rather than paying for a fully durable set
The false-positive rate isn't a bug to eliminate, it's a dial to set deliberately — going from 0.1% to 0.01% roughly doubles the bit array size. State the accepted rate out loud rather than promising zero false positives, since a real hash set at this scale isn't a serious option.

Deep dive: recrawl scheduling and freshness

Not every page deserves the same recrawl cadence — a news homepage that changes hourly and a static documentation page that hasn't changed in a year are wasting the same fetch budget if polled identically.

  1. Every recrawl computes a content hash — compared against the hash stored from the previous fetch
  2. Change-rate is updated as a rolling estimate — a page that changed on this check nudges its estimated change frequency up, one that didn't nudges it down
  3. Freshness score blends change-rate with importance — a high-traffic page that rarely changes can still outrank a low-traffic page that changes constantly
  4. next_recrawl_at is recomputed — shorter intervals for volatile pages, longer for stable ones, always within a min/max bound
  5. Page re-enters the frontier at its due time — competing for fetch budget against both first-time discoveries and other due recrawls
Recrawl and first-discovery share one frontier and one fetch budget on purpose — treating them as separate systems would let an aggressive recrawl policy starve discovery of brand-new pages, or vice versa.

  • 02
    "Politeness has to be enforced at the frontier or scheduler level, not trusted to individual fetcher workers, or a large fleet still races past the limit in aggregate" — shows you understand distributed rate-limiting, not just single-node rate-limiting
  • 03
    "A real hash set of 10 billion seen URLs is hundreds of gigabytes — I'd use a Bloom filter and explicitly state the false-positive rate I'm accepting" — quantifies the tradeoff instead of just naming the data structure
  • 04
    "A crawler trap — infinite URLs from query parameters or calendar links — will silently eat the whole fetch budget unless I cap max-depth and max-URLs-per-domain up front" — proactively surfaces a failure mode most candidates only discover when prompted
  • 05

    "I'd put a timeout and a per-domain circuit breaker on every fetch, because one slow-responding server can otherwise drain fetcher capacity indefinitely" — names a concrete resilience mechanism instead of hand-waving "handle timeouts"

Follow-up questions

How do you avoid crawling the same content reachable via two different URLs?
URL normalization first (sort query params, strip session IDs, lowercase host) collapses many duplicates before dedup even runs; a content-hash comparison catches the rest post-fetch.
What happens when the Bloom filter reports a false positive?
A genuinely new URL is silently never crawled — no error, no retry. It's mitigated by periodic resizing/rotation and accepted as a fixed, tunable cost rather than something to eliminate entirely.
How would you prioritize which domains get crawled first when the frontier has a backlog?
A priority score blending page importance (e.g. inbound link count), change rate, and staleness — high-value, fast-changing, overdue pages surface first within each domain's politeness budget.
How do you detect and stop a crawler trap in real time, not after the fact?
Track URL count and unique-path-depth per domain against a running baseline; crossing a hard cap on max-URLs-per-domain or max-depth halts further enqueuing from that domain automatically.
Why keep DNS resolution as a separate cached step instead of resolving inline per fetch?
A cold DNS lookup can add tens to hundreds of milliseconds — at thousands of fetches/sec that latency compounds across the fleet, so pre-resolving and caching within TTL bounds is a cheap win.
How would this design change for a scoped, single-site crawl instead of web-scale?
The Bloom filter and sharded frontier become overkill — a single in-memory set and one politeness queue suffice; the design only earns its complexity at the 10B-URL, multi-domain scale.
