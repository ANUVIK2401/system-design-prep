Design Facebook News Feed

Tests whether you can reason about fan-out on write vs read at the tail of the follower-count distribution — one design breaks the moment a celebrity posts.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: fan-out on write vs fan-out on read is not either/or — it's a hybrid keyed on follower count
  • Key scale number to anchor on: a celebrity with 50M followers makes push fan-out physically impossible on a single post
  • Primary tradeoff being probed: write amplification (push) vs read latency (pull) — and who pays the cost
  • Component most candidates forget: the feed is a precomputed, ranked cache (Redis sorted set), not a live query over the Post table
  • Failure mode most candidates skip: what happens when a fan-out worker falls behind during a viral moment — feed staleness must degrade gracefully, not block posting

Step 4 of 6 — Data Flow

Data flow

Write path (blue) fans a post out to followers asynchronously; read path (mint) merges the precomputed feed with a live pull for celebrity authors.

Read path — precomputed cache + celebrity pull merge
Write path — durable post, fan-out decoupled
Fan-out + ranking — fully async, never blocks either path
  1. Client publishes a post — Post Svc writes to Postgres and returns 201 in under 50ms
  2. Post event queued to Kafka — fan-out worker picks it up, never blocks the publish response
  3. Worker checks author follower count — under 1M: push to every follower's Redis ZSET; over 1M: skip push entirely
  4. Client loads feed — Feed Svc reads the precomputed ZSET, then merges in a live pull for any followed celebrity authors
  5. Response hydrated and ranked — post IDs resolved against Postgres/CDN for content, sorted by rank_score, paginated back

Step 6 of 6 — Deep Dives

Deep dive: fan-out write amplification and the hybrid model

Pure push fan-out is simple and fast to read — until a single post has to write into 10 million sorted sets.

1 post
published by a celebrity
10M

10M ZADDs

~15 min
time to fully fan out at 11k writes/sec/worker
✓ Normal user — push, ~2ms felt latency
✗ Celebrity (10M followers) — push would take ~15 min
The hybrid threshold (commonly ~1M followers) isn't arbitrary: below it, push cost is bounded and every follower gets an instantly warm feed; above it, the write amplification (10M+ ZADDs from one post) would starve the fan-out worker pool and delay every other user's fan-out behind it in the same queue.

Deep dive: feed ranking signal pipeline

Every candidate post gets a rank_score before it's written into (or merged into) a feed — recency alone produces a feed nobody wants to scroll.

Input signals
Ranking model
Score written as ZSET rank
  1. Recency decays exponentially — a post from 30 minutes ago outranks one from 3 days ago by default, half-life tuned to ~6 hours
  2. Affinity score computed offline — built from historical interaction rate between the specific viewer and author, refreshed periodically not per-request
  3. Engagement prediction from ClickHouse features — a lightweight model estimates P(engage) per viewer/post pair using aggregated signals
  4. Signals combine into one rank_score — a single float so it can be used directly as a Redis ZSET score, no re-ranking needed at read time
  5. Score is recomputed only on fan-out — not on every feed read, keeping the read path a pure sorted-set range query
Ranking must not run synchronously on the feed-read path — if the model call is on the critical path, a slow feature lookup directly adds to feed load latency. Score at fan-out/write time instead, and treat the feed read as a dumb, fast ZREVRANGE.

Interview moves

  • 01
    "The feed is a precomputed cache, not a live query — I'll rank and materialize it ahead of time so the read path is a cheap sorted-set range scan" — signals you understand the read/write asymmetry immediately
  • 02
    "Pure push fan-out breaks the moment someone with 10 million followers posts — that's 10 million writes fired from one event" — names the exact failure mode before being asked
  • 03
    "I'd use a hybrid: push for normal accounts under roughly a million followers, and merge in a live pull for celebrity accounts at read time" — the single answer that separates strong candidates on this question
  • 04
    "Ranking has to happen at fan-out/write time, not on the read path — otherwise every feed load pays for a model inference" — shows awareness that read latency budgets are non-negotiable
  • 05
    "I'll decouple engagement events into their own async pipeline so a slow ranking feature computation never blocks a like or comment from confirming" — demonstrates the same hot-path-isolation instinct as write/read separation
  • 06
    "If the fan-out workers fall behind during a viral spike, I'd rather show a slightly stale feed than block or slow down publishing" — states a concrete, correct failure-mode priority unprompted

Follow-up questions

How do you decide the exact follower-count threshold for push vs pull?
Based on fan-out worker throughput budget — pick the count where worst-case fan-out time stays under a few seconds, typically in the hundreds of thousands to low millions.
What happens when a user follows 500 celebrities?
The pull merge at read time queries a small, per-celebrity cached "latest posts" list rather than hitting Postgres directly, keeping the merge itself cache-backed and fast.
How do you handle a user with zero cached feed (new account, cold cache)?
Fall back to an on-demand build: pull recent posts from their follow list directly from Postgres/Social Graph, rank inline, and populate the ZSET for next time.
How do you prevent the feed from being dominated by one hyperactive friend?
Cap posts per author per feed window and fold repetition into the affinity signal so the ranking model naturally discounts over-posting from a single source.
How would you support real-time-ish updates (new post appears without refresh)?
A lightweight WebSocket/long-poll notifying the client a new item exists, prompting a background refetch — not pushing full content over the socket itself.
How do you extend this to 10x the user base?
Shard Redis and Social Graph further by user_id, push more authors into pull-only status, and move ranking feature computation to precomputed batch snapshots refreshed more frequently.
