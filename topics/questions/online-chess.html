Questions

Design Online Chess

Tests whether you treat two-player game state as a distributed consensus problem — both clients must see an identical board with zero divergence, even across packet loss and reconnects.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: this is a distributed consensus problem in miniature — two clients must converge on one authoritative board state, and packet loss must never let them diverge
  • Key scale number to anchor on: 10M concurrent games, each a stateful WebSocket session pinned to one game server — this is not a stateless request/response system
  • Primary tradeoff being probed: strong consistency on the two-player write path vs eventual consistency on the spectator read path — they should not share an architecture
  • Component most candidates forget: idempotency on the move message — an ack can be lost even when the move itself was applied, so the client must be able to safely retry without double-submitting
  • Failure mode most candidates skip: a network partition between the two players isn't a bug to catch, it's a rule to define — someone needs an adjudication policy, not just a retry loop

Step 2 of 6 — Core Entities

Core entities

Live board state lives in Redis for sub-millisecond read/write on the hot path; player identity and rating live in Postgres for durability; every move is appended to an immutable log for replay, disputes, and post-game analysis.

Step 4 of 6 — Data Flow

Data flow

A move write (blue) touches the game server and Redis directly; the same applied move is then fanned out to spectators (amber) through a separate broadcast layer that never talks to the authoritative game server.

Player client / spectators — separate connection pools
Redis — single authoritative board, written by one path, read by the fan-out layer
Pub/sub fan-out — decoupled from the hot game-write path
  1. Player submits a move — sent over the already-open game WebSocket with an idempotency key
  2. Game Server forwards to Move Validator — checks legality against the current FEN and whose turn it is
  3. Valid move is written to Redis — atomic update of FEN, turn, move_count (acts as version)
  4. Ack sent to the mover, move broadcast to the opponent — both over their own persistent connections
  5. Applied move is published to the fan-out layer — spectators receive it seconds or milliseconds later, never touching the game server

The two players hold a sticky session against one game server instance; that server is the sole writer to the game's Redis state; matchmaking and spectator fan-out are separate services that never sit on the players' hot path.

Step 6 of 6 — Deep Dives

Deep dive: move validation and delivery

A move is validated once, applied once, then delivered twice — an ack to the sender and a broadcast to the opponent. If the ack is lost, the client retries with the same idempotency key rather than resubmitting a fresh move.

Client / gateway
Cache — hot path
Async — queue / worker
  1. Mover submits a move — includes a client-generated idempotency key and the move_count it expects to apply against
  2. Game Server validates and applies — legality check, then atomic Redis update, move_count increments
  3. Ack goes to the mover, broadcast goes to the opponent — two independent sends over two independent sockets
  4. If the ack never arrives — the mover's client retry timer fires and resends the identical move with the same idempotency key
  5. Server detects the duplicate key — it does not re-validate or re-apply the move, it simply re-sends the stored ack, so the board never double-advances
Without the idempotency key, a lost ack plus a naive retry would submit the same move twice — the second submission would either be rejected as illegal (confusing the client) or, worse, silently advance the board an extra ply if validation is sloppy about turn order.

Deep dive: ELO-bucketed matchmaking

Players are placed into a rating bucket on queue entry. The matcher first looks for an opponent inside a narrow rating/latency window, then widens that window on a timer if no match is found.

Client / gateway
Cache — hot path
Database — source of truth
Async — queue / worker
  1. Player enters the queue — assigned to a rating bucket based on current ELO
  2. Matcher searches a narrow window first — e.g. ±50 ELO and a tight latency/region constraint, for match quality
  3. No match after a few seconds — the rating window widens on a timer (±50 → ±150 → ±300...)
  4. Latency constraint can relax too — a distant region becomes acceptable rather than leaving the player waiting indefinitely
  5. Match found — both players are handed off to the same Game Server instance, which creates the Game record and opens both sockets
The widening schedule is itself a tradeoff knob: a steeper widening curve favors wait time, a flatter one favors match quality — worth stating explicitly if asked "how do you pick the widening rate?"

Deep dive: spectator fan-out at 1M viewers

The two players' write path and the spectators' read path must never share infrastructure — a viral match with 1M viewers cannot be allowed to add latency or connection pressure to the two people actually playing.

Players — fixed at 2 sockets regardless of spectator count
Game State + Event — single publish point per applied move
Pub/Sub + CDN edge — the only tier that scales with viewer count
  1. Move is applied on the Game Server — exactly as it would be for an unwatched game, no extra cost per spectator
  2. One event is published — a single fan-out message per move, not one per viewer
  3. Pub/sub broker distributes to CDN edge nodes — regional nodes absorb the bulk of connection fan-out close to viewers
  4. Edge nodes hold the spectator connections — 1M viewers spread across many edge nodes, not one origin server
  5. Spectators never connect to the Game Server — a viral match adds zero load to the two players' authoritative connection
The single most common mistake here is routing spectators through the same server that holds the players' sockets "because it already has the state" — that couples an unbounded, bursty read load to a latency-critical write path and is exactly the kind of decision an interviewer is listening for you to avoid.

Interview moves

  • 01
    "This is really a distributed consensus problem wearing a chess costume — the hard part isn't the rules engine, it's guaranteeing both clients converge on one board state even under packet loss" — states the core insight unprompted, reframes the problem correctly
  • 02
    "I'd put an idempotency key on every move message so a lost ack can be safely retried without the server re-applying or double-counting the move" — names the exact mechanism that prevents the most common bug in this design
  • 03
    "The game server has to be stateful with a sticky session — this isn't a stateless request/response API, and pretending otherwise breaks reconnect and failover" — shows you're not defaulting to a generic load-balanced-stateless mental model
  • 04
    "Spectators should never connect to the same server holding the players' sockets — a viral game with a million viewers can't be allowed to add latency to the two people actually playing" — proactively separates read and write paths before being asked
  • 05
    "A network partition between two players isn't just an error to catch — someone needs to define the adjudication rule, like a disconnect-clock timeout that awards the game" — surfaces a product/policy decision most candidates treat as purely technical
  • 06
    "Cheat detection runs as an async, post-game pipeline against engine analysis — it must never sit on the live move path or it adds latency to every legitimate player to catch a small minority" — demonstrates separating a real requirement from the hot path it doesn't belong on

Follow-up questions

How do you guarantee the two players never see divergent board states?
Single writer per game (the Game Server), atomic Redis updates keyed by game_id, and move_count as a monotonic version — no client ever applies a move locally before the server confirms it.
What happens if a player's client reconnects mid-game after a dropped connection?
The client reconnects to the same sticky Game Server (or a recovered one), sends its last known move_count, and the server replies with the authoritative current FEN to resync.
How would you scale matchmaking if the player pool is very thin at extreme ratings?
Widen both the rating window and the latency/region constraint on a timer so an isolated high- or low-rated player still gets matched, trading some match quality for bounded wait time.
Why is cheat detection async instead of blocking moves in real time?
Real-time engine analysis on every move would add latency to every legitimate player to catch a small minority — running it post-game against the full move log keeps the hot path fast.
How do you prevent a hostile client from claiming an illegal move was applied?
The server is the sole source of truth — it validates every move server-side before writing to Redis, and clients only ever render what the server acks or broadcasts, never their own local guess.
What's the tradeoff in choosing Redis over Postgres for live game state?
Redis gives the sub-millisecond atomic read-modify-write the move path needs at 350k moves/sec, at the cost of weaker durability — mitigated by the append-only Postgres move log as the recoverable source of truth.
