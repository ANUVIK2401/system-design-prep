Design Facebook Live Comments

Tests whether you can fan a single comment out to millions of concurrent viewers in under a second — this is pub/sub at extreme scale, not a CRUD comments feature.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: this is fan-out delivery at extreme scale, not a comments CRUD problem — writes are cheap, broadcast is the hard part
  • Key scale number to anchor on: a single hot stream can hold 1M+ concurrent viewers, each needing every new comment pushed within ~1 second
  • Primary tradeoff being probed: strict comment ordering per stream vs horizontal write throughput — solved by partitioning on stream_id
  • Component most candidates forget: a connection server layer that holds the long-lived WebSockets, decoupled from the stateless ingestion path
  • Failure mode most candidates skip: a fan-out worker falling behind on a viral stream — it must shed load gracefully, never crash the whole pipeline

Step 4 of 6 — Data Flow

Data flow

Write path (blue) — post a comment. Read/fan-out path (mint) — that comment delivered to every viewer's open socket.

Client / viewer
Kafka — durable, ordered by stream_id
Connection servers — hold the sockets
Cassandra — comment history
  1. Client posts a comment — Ingestion Svc validates and returns 202 immediately
  2. Comment persisted to Cassandra — durable history, keyed by stream_id + comment_id
  3. Comment published to Kafka — partitioned by stream_id, so order is preserved per stream
  4. Fan-out worker consumes and looks up routing — Redis tells it which connection servers hold this stream's viewers
  5. Pushed to every connection server holding a viewer — each pushes down its open WebSockets, target <1s end-to-end

Step 6 of 6 — Deep Dives

Deep dive: connection server architecture

Millions of long-lived WebSockets can't live on one box — the pool is sharded, and Redis is the map from stream to shard.

Fan-out worker — one comment, many targets
Redis routing + connection server pool
Viewers — sharded across the pool
  1. Viewer opens the WebSocket — load balancer assigns it to whichever connection server has headroom
  2. Server registers itself in Redis — adds its ID to the set for that stream_id
  3. Fan-out worker never talks to sockets directly — it asks Redis which servers to notify, then pushes once per server, not once per viewer
  4. Each connection server fans out locally — iterates its own in-memory socket map for that stream, writes the frame to each
  5. On disconnect, the server removes itself from the Redis set — if it was the last one for that stream, the set is cleaned up
The connection server is the one genuinely stateful tier in this system — a single instance dying drops every socket it held. Clients must detect the drop and reconnect (with backoff), and the load balancer must route the reconnect to a healthy instance. This is why capacity planning targets ~50k sockets/instance, not the theoretical OS-level max — headroom for a full instance failure without cascading reconnect storms.

Deep dive: Kafka partition strategy

Partitioning by stream_id is what makes "every viewer sees comments in the same order" possible without a distributed sort.

Kafka partitions — one ordered log each
Consumers — each owns a fixed set of partitions
  1. Producer hashes stream_id — picks the partition deterministically, same stream always lands on the same partition
  2. Kafka preserves order within a partition — comments for one stream are appended in the exact order they were produced
  3. One consumer instance owns a partition at a time — guarantees a single stream's comments are never processed out of order or in parallel
  4. A viral stream can still overload its single partition — the tradeoff for ordering is a per-partition throughput ceiling
  5. Mitigation for a hot stream: sub-key by comment_id ranges — relaxes strict ordering only for that one stream, in exchange for parallelism

Interview moves

    "The hard part here isn't storing comments — it's fanning one comment out to a million open connections in under a second" — names the real problem before diving into a familiar CRUD shape
  • 02
    "I'll partition Kafka by stream_id so ordering only needs to hold within a stream, not globally — that's the scope the requirement actually needs" — shows the ordering guarantee is scoped deliberately, not accidentally weak
  • 03
    "Connection servers are the one stateful tier in this system — everything else stays stateless so it can scale freely around it" — demonstrates architectural awareness of where state actually lives
  • 04
    "Fan-out workers never talk to individual sockets — they push once per connection server, and each server fans out locally to its own viewers" — a concrete detail that shows the fan-out isn't O(viewers) at the messaging layer
  • 05
    "If a fan-out worker falls behind on a viral stream, I'd rather batch pushes into short windows than let latency creep unbounded" — proactively states a degrade-gracefully strategy before being asked
  • 06
    "A connection server dying drops every socket it holds — I size instances at ~50k sockets, not the OS max, specifically to leave headroom for that failure" — ties a concrete capacity number directly to a stated failure mode

Follow-up questions

Why not just have every viewer poll for new comments every second?
At 1M concurrent viewers that's 1M requests/sec just for one stream's polling — a persistent WebSocket amortizes that into a single push per comment instead.
How do you handle a viewer switching between streams rapidly?
Each switch closes the old socket registration in Redis and opens a new one; the connection server itself can often be reused, only the routing entry changes.
How would you extend this to support 10x the concurrent viewers?
Add more connection server instances (near-linear scaling), scale the fan-out consumer group, and sub-partition Kafka topics for any individual stream that outgrows one partition.
What happens to in-flight comments if a fan-out worker crashes mid-broadcast?
Kafka retains the offset; a new consumer in the group picks up the partition from the last committed offset, so comments are delayed, not lost.
How do you moderate a comment out in real time across a million viewers?
Delete publishes a lightweight "hide comment_id" event through the same Kafka/fan-out path, so every connected client removes it client-side within the same latency budget.
Why Cassandra for comment history instead of Postgres?
The access pattern is purely append + range-scan by stream_id, which a wide-column store handles at far higher write throughput than a relational store without sharding.
