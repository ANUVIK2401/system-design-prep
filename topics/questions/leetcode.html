Design LeetCode (Online Code Judge)

Tests whether you can safely run untrusted user code at scale — sandboxing is the actual system design problem, the CRUD around it is a distraction.

30-40 min interview Medium
Say this in the first 2 minutes
  • The core insight: this is an untrusted-workload isolation problem, not a CRUD problem — sandboxing is the whole design, not a detail
  • Key scale number to anchor on: ~5k submissions/sec at peak (contest start), each fanning out into 20-100 test case runs
  • Primary tradeoff being probed: sandbox isolation strength (VM-level) vs spin-up latency (container-level) — you can't max both
  • Component most candidates forget: a job queue between submission and execution — synchronous execution on the request thread cannot survive a contest-start spike
  • Failure mode most candidates skip: infinite loops and fork bombs — every sandbox needs hard CPU/memory/wall-clock caps enforced by the host, not the guest

  2. Client opens a WebSocket — subscribes to that submission_id for live status
  3. Judge worker dequeues the job — pulls source from S3 and test cases from the problem's bundle
  4. Sandbox executes each test case — isolated, resource-capped, no network access
  5. Result written and pushed — verdict lands in the result store, WS gateway pushes the update to the open socket

Step 6 of 6 — Deep Dives

Deep dive: sandbox isolation layers

Untrusted code gets four stacked layers of containment — any single layer failing shouldn't compromise the host.

Host OS — never directly reachable
Container — filesystem + network isolation
gVisor — intercepts every syscall
seccomp + cgroups — the hard numeric caps
  1. Container boundary — read-only root filesystem, no network namespace at all, so exfiltration and outbound abuse are structurally impossible, not just blocked by policy
  2. gVisor userspace kernel — reimplements the Linux syscall surface in userspace, so even a kernel exploit in the submitted code hits gVisor's sandbox, not the real host kernel
  3. seccomp-bpf filter — denies an explicit blocklist of dangerous syscalls (ptrace, mount, reboot, clone with CLONE_NEWUSER) before gVisor even has to interpret them
  4. cgroup resource caps — 2s CPU time, 256MB memory, 64 max processes/threads — enforced by the host scheduler, so a spinning process gets killed even if application-level timeouts fail
  5. Sandbox teardown — every sandbox is destroyed after one submission's run, never reused across tenants, so no state or memory leaks between users
The timeout must be enforced by the host (cgroup wall-clock + CPU quota), never by a timer inside the submitted code's own process — an infinite loop or intentionally hostile submission will never call its own cleanup code.

Deep dive: test case runner parallelism

A single submission with 100 test cases can't run them serially against a 10s p99 target — they fan out to independent sandboxes and results aggregate back.

  1. Judge worker claims the job — reads the test case bundle manifest for the problem (count, inputs, expected outputs)
  2. Fan out N sandboxes in parallel — one sandbox per test case, up to a per-submission concurrency cap (typically 10-20 at once) to avoid one submission monopolizing the worker's local resources
  3. Each sandbox runs independently — own CPU/memory/wall-clock caps, one case's TLE doesn't affect siblings
  4. Results stream back as each case finishes — the WS gateway can push "42/100 passed" incrementally rather than waiting for all 100
  5. Aggregation rule: first failure can short-circuit — for a straight pass/fail verdict, stop dispatching remaining cases once one fails, saving compute; contest judges instead run all cases for partial-credit scoring
Short-circuiting on first failure is a product decision, not just an optimization — LeetCode-style practice mode wants a fast fail signal, but a scored contest needs every test case's result for tie-breaking and partial credit, so it must run to completion regardless of an early failure.

  • 02

  • 03
    "Every resource limit — CPU, memory, wall-clock — has to be enforced by the host via cgroups, never trusted to code running inside the sandbox" — shows you understand who the adversary actually is
  • 04
    "I'd fan out a submission's test cases to run in parallel across sandboxes rather than serially, since 100 sequential runs would blow past our latency target" — catches an obvious latency trap before being asked
  • 05
    "Whether to short-circuit on the first failing test case is a product decision — fast feedback for practice mode, but a scored contest needs every case's result for partial credit" — reframes an implementation detail as a requirements question
  • 06
    "gVisor over raw Docker here — a container alone shares the host kernel, and that's not a strong enough boundary for code I don't trust at all" — justifies a specific, correct technology choice with the actual reason

Follow-up questions

Why not just use Docker containers without gVisor?
Containers share the host kernel — a container-escape kernel exploit compromises the host directly. gVisor intercepts syscalls in userspace, so an exploit hits gVisor's sandboxed kernel instead.
How do you keep sandbox spin-up latency low without compromising isolation?
Pre-warm a pool of idle, already-isolated sandboxes ahead of demand (especially before scheduled contests) so a submission claims one instead of paying the ~150ms cold-start cost per request.
How do you support multiple languages without N different judge images?
One base sandbox image per language runtime, selected at dispatch time; compiled languages (Java, C++) get a separate compile step with its own timeout before the run step even starts.
How do you prevent a user from submitting the same solution thousands of times to game the leaderboard?
Rate limit submissions per user per problem (e.g. one every few seconds), and hash-dedupe identical source against a user's own recent submissions to short-circuit obvious repeats.
What happens if the judge worker fleet is completely saturated during a flash-contest spike?
Submissions queue in Kafka and the UI shows an honest queue position rather than fabricating progress; autoscaling adds workers on queue depth, and pre-warmed capacity absorbs the first burst before scale-up lands.
How do you stop test case data from leaking to users through the submitted code's output?
The sandbox only returns stdout/stderr up to a size cap and a pass/fail verdict — raw expected-output test data is never mounted anywhere the submitted process can read it directly.
