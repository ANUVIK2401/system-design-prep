Design Facebook Post Search

Tests whether you can build a near-real-time search index while enforcing per-viewer privacy — the hard part isn't full-text search, it's never showing a post to someone who can't see it.

30-40 min interview Medium
Say this in the first 2 minutes
  • The core insight: this isn't a full-text search problem, it's a privacy-filtered full-text search problem — the ACL check is the hard part, not the tokenizer
  • Key scale number to anchor on: 500M posts/day created, but search must reflect a new post within ~5-10 seconds or results feel stale
  • Primary tradeoff being probed: filter privacy at index time (fast, can go stale on ACL changes) vs at query time (always correct, costs latency)
  • Component most candidates forget: a durable event stream (Kafka) between the post service and the indexer, so indexing never blocks post creation
  • Failure mode most candidates skip: what happens when a post's audience changes (public → friends-only) after it's already indexed

Read path — privacy filter applied before ranking

  1. Client posts text — Post Svc writes to Postgres and returns 201 immediately, indexing is not on this path
  2. Post-created event streamed to Kafka — fired after the DB commit, decoupling indexing latency from write latency
  3. Index Worker consumes the event — tokenizes text and snapshots the current audience into visible_to
  4. Document written to Elasticsearch — searchable within the <10s lag target
  5. Search request hits Privacy Filter first — the viewer's friend/block sets gate the query before it ever reaches ranking

Step 6 of 6 — Deep Dives

Deep dive: the inverted index

The data structure that makes "find every post containing this word" fast instead of a full scan across 10B rows.

Client / gateway
Cache — hot path
Database — source of truth
Async — queue / worker
  1. Post text is tokenized — lowercased, stemmed ("launching" → "launch"), stopwords dropped
  2. Each term maps to a posting list — a sorted list of every post_id containing that term
  3. A multi-term query intersects posting lists — "launch dashboard" only matches posts in both lists
  4. The privacy filter ANDs against visible_to — narrows the intersected set to the viewer's allowed audience before ranking
  5. Surviving docs are scored and ranked — BM25 (term frequency weighted by rarity) orders the final results
The posting list is what makes privacy filtering at query time affordable: it's a set intersection (term matches ∩ viewer's audience), not a per-document ACL check against every candidate.

Deep dive: privacy filtering — index time vs query time

Where you enforce "who can see this post" changes the entire correctness/latency tradeoff of the system.

Filter at index time

Each document stores a denormalized visible_to list computed when it's indexed. Queries filter against it directly — fast, no extra hop.

Risk: if a user's audience shrinks (friend removed, post switched to friends-only) after indexing, the stale visible_to keeps showing it to someone who should no longer see it — a real privacy leak until the next re-index event fires.

Filter at query time

Every search resolves the viewer's live audience (friend/block sets) and intersects it with matched posts at query time — always reflects the current ACL state.

Cost: an extra Redis (or graph-DB) round trip per query, and the intersection itself adds compute versus a pre-baked field. This is the safer default for a "zero leaks" requirement.

This system uses both: visible_to is denormalized into the index as a coarse pre-filter (cuts the candidate set fast), then Privacy Filter re-validates against the live friend/block cache at query time before results are returned — belt and suspenders, because a false positive here is worse than a false negative.

  • 02
    "I'll decouple indexing from post creation with Kafka so a slow indexer never adds latency to posting" — shows the instinct to isolate the write path from a downstream async system
  • 03
    "I'd rather show fewer results than leak one — so query-time privacy re-validation is non-negotiable even though it costs latency" — states a concrete, defensible priority ordering unprompted
  • 04
    "Denormalizing the audience into the index gives us a cheap pre-filter, but I still re-check against the live friend/block cache at query time" — demonstrates the index-time vs query-time tradeoff isn't binary
  • 05
    "A multi-term query is just a set intersection of posting lists — that's why the inverted index scales to billions of documents" — grounds the abstract search claim in a concrete mechanism
  • 06
    "If a post's visibility changes from public to friends-only, that has to re-enter the same indexing pipeline as a brand-new post" — catches a subtlety most candidates miss: ACL changes, not just creates/deletes, must trigger re-indexing

Follow-up questions

How do you rank results when many posts match the same terms?
BM25 (term frequency weighted by inverse document frequency) as the base signal, with recency and engagement as secondary boosts — Elasticsearch supports this natively.
What happens if the indexing worker crashes mid-batch?
Kafka consumer offsets aren't committed until the ES write succeeds, so the event is simply replayed on restart — indexing is idempotent by post_id.
How do you handle a user who blocks someone after their posts are already indexed?
The block event invalidates that pair in the Privacy Filter cache immediately; the denormalized visible_to field catches up on the next re-index sweep.
Why not just filter privacy in the Post Service before results are ever sent to Elasticsearch?
Filtering must happen per-viewer at query time, not per-post at write time — the same post is visible to different audiences depending on who's searching.
How would you support typo tolerance or fuzzy matching?
Elasticsearch's built-in fuzzy query (edit-distance matching) on the analyzed field — costs extra query latency, so it's usually a fallback when exact-match returns few results.
How do you scale this to 10x the document volume?
Increase Elasticsearch primary shard count (requires reindexing), add more Kafka partitions and indexing workers in lockstep, and push hot-query caching harder at the Search API layer.
