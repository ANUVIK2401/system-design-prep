Questions

Design Ticketmaster

Tests whether you can solve distributed locking under extreme write contention — millions of people fighting over the same 20,000 seats in the same 60 seconds.

40-50 min interview Hard
Say this in the first 2 minutes
  • The core insight: this isn't a CRUD problem, it's a distributed locking problem — the hard part is preventing two people from both "successfully" reserving the same seat
  • Key scale number to anchor on: a single stadium on-sale can see 3-5M waiting-room requests in the first 60 seconds for 20,000 seats — a 200:1+ contention ratio
  • Primary tradeoff being probed: pessimistic locking (simple, safe, contends hard) vs optimistic locking (scales better, needs conflict retry UX)
  • Component most candidates forget: a TTL-based hold expiry — if a lock never releases on abandoned checkout, seats die forever
  • Failure mode most candidates skip: double-booking during a lock-service failover — must be architecturally impossible, not just unlikely

Step 4 of 6 — Data Flow

Data flow

Read path — merges durable state with live locks
Write path — atomic lock acquire is the whole game
TTL expiry — self-healing, no cleanup job required
  1. Client requests a hold on N seats — one API call, all-or-nothing
  2. Reservation Svc attempts SETNX on every seat_id — atomic per-seat, in a fixed lock order to avoid deadlock
  3. Any seat fails to lock → roll back all acquired locks — returns 409, nothing partially held
  4. All locks acquired → hold_token + 8-minute TTL returned — checkout page opens with a visible countdown
  5. Confirm within TTL → Postgres write + Redis locks deleted — or TTL fires and seats silently return to available

Step 6 of 6 — Deep Dives

Deep dive: optimistic vs pessimistic locking

✓ Pessimistic — Redis SETNX lock
✗ Optimistic — version-check on write
Optimistic locking scales beautifully when contention is rare — but a hot on-sale is the opposite of rare contention. Retrying failed writes at 250:1 demand-to-inventory ratio just means most users pay the full cost of checkout UX before being told "sorry, taken." Pessimistic locking rejects instantly instead, which is why this design defaults to it for the hold step.

Deep dive: seat hold state machine

Available / Released — no active Redis key
Held — Redis key with live TTL
Reserved — durable, Postgres source of truth
  1. Available → Held — atomic SETNX in Redis; seat map reflects this within one poll cycle
  2. Held → Reserved — only fires after the payment gateway confirms the charge; Postgres write and Redis delete happen in the same request
  3. Held → Released — TTL expiry (no action needed) or explicit cancel; either way the Redis key simply stops existing
  4. Released → Available — there's no separate storage state for this; absence of a Redis key already means available
  5. Reserved → Released — a refund path, gated behind support tooling, not exposed on the hot path at all
The dangerous transition is Held → Reserved: if the payment call succeeds but the Postgres write fails (or times out) before the Redis lock is deleted, the seat can appear inconsistently sold. Fix: wrap the Postgres commit and Redis delete in a single idempotent operation keyed by hold_token, and make the payment webhook retry-safe so a retried confirm can't double-charge.

Interview moves

  • 01
    "The core problem here isn't the seat map, it's distributed locking under extreme contention — I want to design the lock first" — immediately signals you've identified the actual hard problem, not the decoy
  • 02
    "At 250:1 demand-to-inventory, optimistic locking would mean most users discover a conflict after doing real work — I'm going pessimistic with a Redis SETNX" — justifies a specific tradeoff with a specific number
  • 03
    "Every hold needs a TTL, or one abandoned checkout tab kills a seat forever — this has to be self-healing, not dependent on a cleanup job" — shows you think about failure modes unprompted
  • 04
    "I'd put a waiting room in front of the booking flow so the thundering herd never even reaches Redis directly" — demonstrates layered defense, not just a single strong component
  • 05
    "When acquiring locks for multiple seats in one order, I always sort seat_ids first to avoid a classic deadlock between two overlapping multi-seat holds" — a concrete correctness detail most candidates miss entirely
  • 06
    "The Held → Reserved transition is the one place a bug causes real money damage, so I'd make the Postgres write and Redis delete a single idempotent operation keyed by hold_token" — pinpoints the exact risky transition and states the fix precisely

Follow-up questions

Why not just use a Postgres row lock (SELECT ... FOR UPDATE) instead of Redis?
It works at low contention, but holding a transaction open across an 8-minute checkout window would exhaust connection pools instantly at this scale. Redis decouples the lock lifetime from any DB transaction.
How do you handle a group of friends trying to sit together?
The hold API accepts an array of seat_ids and locks all-or-nothing — if any seat in the group fails to lock, every acquired lock in that request is rolled back immediately.
What stops a bot from holding thousands of seats and never checking out?
Per-user rate limits on hold requests, CAPTCHA at the waiting-room entry, and a max-seats-per-hold cap (typically 8) enforced server-side before the lock is even attempted.
How would you extend this to support 10x the scale (a global on-sale)?
Shard Redis by event_id so hot events don't starve others, run per-region waiting rooms, and pre-provision dedicated capacity ahead of any known high-demand on-sale time.
How do you keep the seat map view from showing stale availability?
Short polling (2-3s) or a lightweight WebSocket push on state change; the view is allowed to be a few seconds stale since the lock itself is the real source of truth at hold time.
What happens if the payment gateway times out but may have actually charged the card?
Never assume failure on timeout — poll the gateway's idempotency-keyed status endpoint before releasing the hold, so a slow success can't result in a charged-but-unseated customer.
