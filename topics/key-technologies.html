Mental model

Technologies are tools. The mistake is reaching for a tool because you know it — not because the problem needs it. Every choice must be followed by "because the access pattern is X and the tradeoff I'm accepting is Y."

SQL vs NoSQL

PostgreSQL — consistency vs availability
Strong consistencyAvailability
DynamoDB / Cassandra — consistency vs availability
Strong consistencyAvailability
Read-optimized vs write-optimized
Read-optimized (SQL + replicas)Write-optimized (NoSQL sharded)
SQL (PostgreSQL, MySQL)
  • Structured, relational data with joins
  • ACID transactions — critical for financial or inventory data
  • Schema is known upfront and stable
  • Complex queries (aggregations, filtering)
  • Scales well to ~10k writes/sec with read replicas
NoSQL (DynamoDB, Cassandra, MongoDB)
  • High write throughput (>10k writes/sec)
  • Flexible or evolving schema
  • Simple key-value or document access pattern
  • Horizontal sharding required from day one
  • Eventual consistency is acceptable
Common trap: using NoSQL because it sounds "more scalable." SQL scales further than most systems need. Justify the choice with a specific scale number or access pattern requirement.

Kafka — when and why

Producer (decoupled from consumers)
Kafka topic (persisted, replayable)
Independent consumer groups
  1. Fan-out: One event (user sign-up) triggers multiple consumers — email, analytics, fraud check — without the producer knowing about them
  2. Durability: Messages persisted to disk — consumers can replay from any offset
  3. Decoupling: Producer and consumer scale independently; consumer can be slow without blocking producer
  4. Ordering: Guaranteed within a partition — partition by user_id for per-user ordering
Use Kafka when you need fan-out, replay, or >100k events/sec. For simple point-to-point task queues (<10k/sec), SQS or RabbitMQ is simpler and sufficient.

CDN — when it changes your design

CDN — latency vs freshness
Always fresh (origin)Lowest latency (edge cache)
What CDN solves
  • Serves static assets from edge nodes (~5ms vs ~100ms)
  • Absorbs read traffic — origin sees only cache misses
  • Global availability without multi-region origin infra
  • DDoS absorption at the edge
  • HTTP redirect caching (URL shortener use case)
CDN limitations
  • Cache invalidation on update is slow or requires purge API
  • Personalized content can't be cached
  • Short TTL reduces cache hit rate
  • Cost scales with egress bandwidth
  • Not a substitute for proper DB or service design

Choosing the right DB engine

✓
PostgreSQL: Default choice. Handles 99% of problems. JSONB for flexible fields. Read replicas for scale. Vertical scaling goes far.
✓
DynamoDB / Cassandra: When you need single-digit ms latency at millions of writes/sec. Access pattern must be key-value or range scan — no joins.
⚠
MongoDB: Good for document workloads with nested structure. Watch out: default eventual consistency and non-atomic multi-document operations.
✗
Never store images or video in a relational DB. Use S3 + CDN. Store only the S3 key/URL in the DB.

Interview moves

  • 01
    "I'd use PostgreSQL here — the data is relational, schema is stable, and we're at ~1k writes/sec which it handles easily" — default to SQL and justify if you deviate
  • 02
    "For the hot read path I'd put Redis in front — this cuts latency from ~10ms to ~1ms for the top 20% of keys that get 80% of traffic" — always quantify the cache benefit
  • 03
    "The user upload goes to S3 — I'd never store binary blobs in Postgres. The DB stores only the S3 key." — say this proactively; missing it is a common junior signal
  • 04
    "I'd use Kafka here for fan-out — when a user posts, we need to notify their followers, update search index, and fire analytics without coupling the write path to all that work" — name the specific reason
  • 05
    "For global latency I'd put a CDN in front of the redirect service — edge nodes serve most requests in <10ms regardless of user location" — CDN is often overlooked; mention it proactively for read-heavy systems

Common follow-up questions

When would you use Cassandra over DynamoDB?
Cassandra when you need multi-region active-active writes (LWT for lightweight transactions) or when you want to avoid vendor lock-in. DynamoDB when you're on AWS and want zero ops overhead. Performance is comparable.
What's a write-through vs write-behind cache?
Write-through: write to cache and DB together — consistent but higher write latency. Write-behind (write-back): write to cache, DB updated asynchronously — lower latency but risk of data loss on crash.
What happens if Redis goes down?
Read traffic falls back to DB — latency increases but system stays up. Implement a circuit breaker on the cache layer. Redis Cluster with replication gives HA. Redis Sentinel for automatic failover.
How does Kafka guarantee exactly-once delivery?
Idempotent producers (dedup by sequence number) + transactional API for atomic produce + consumer offset commits. At-least-once is the practical default — make your consumers idempotent instead.
When would you use a search index like Elasticsearch vs a DB full-text search?
DB full-text (Postgres tsvector) works fine up to ~10M rows with infrequent queries. Elasticsearch for: fuzzy matching, autocomplete, faceted search, >10M docs, or >100 search QPS. Never use ES as your primary store.
What's the difference between a message queue and an event log?
Queue (SQS, RabbitMQ): messages consumed and deleted — each message processed once. Event log (Kafka): messages persisted, consumers maintain their own offset — any consumer can replay from any point in history.
