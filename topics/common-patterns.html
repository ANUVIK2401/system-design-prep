Common Patterns

The 8 recurring architectural patterns that appear across almost every system design question.

15 min read Patterns
Pattern recognition cues
  • Read-heavy (>10:1 R:W) → cache aggressively, read replicas, CDN
  • Write-heavy (>10k/sec) → async writes, Kafka buffer, NoSQL
  • "Notify all followers" → fan-out (push vs pull — decide at interview time)
  • "Never lose a transaction" → saga pattern, idempotent retries, outbox
  • "Different read/write shapes" → CQRS — separate read and write models

Mental model

Patterns are the recurring moves in chess — you don't invent them mid-game, you recognize the position and apply the right pattern. Name the pattern out loud in the interview; it signals experience.

Pattern 2 — Write-heavy system

Triggered by: metrics ingestion, log collection, IoT telemetry, click tracking

Clients (high volume)
→
Kafka / write buffer
→
Consumers (batch write)
→

Never write directly to a relational DB at >10k writes/sec. Buffer in Kafka, batch into a columnar store, or use Cassandra with a write-optimized LSM tree.

Pattern 3 — Fan-out (notification / feed)

Triggered by: Twitter-style feed, notification system, group chat

Fan-out on write (push)
  • When user posts, write to each follower's feed immediately
  • Read is O(1) — precomputed feed
  • Cost: write amplification — 1 post × 10M followers = 10M writes
  • Use for: accounts with <~10k followers
Fan-out on read (pull)
  • When user opens feed, pull latest posts from all followed accounts
  • Write is cheap — only 1 write per post
  • Cost: read is expensive — merge N timelines on each load
  • Use for: celebrity accounts with millions of followers
Production answer: hybrid. Push for normal users, pull for mega-accounts (>1M followers). Twitter uses this exact approach.

Pattern 4 — Saga (distributed transactions)

Triggered by: payment + inventory + order all needing to succeed or all roll back

Order Service
→
Payment Service
→
Inventory Service
→
Confirm
  1. Step: Each service completes its local transaction and publishes a success event
  2. Compensating transaction: If any step fails, each previous service executes a rollback operation (e.g., refund payment)
  3. Choreography vs orchestration: Choreography = services react to events (decoupled). Orchestration = a saga coordinator calls each service in order.
Saga only gives eventual consistency — not ACID. Compensating transactions are complex to write correctly. If you need true atomicity, put everything in one DB.

Pattern 5 — CQRS (Command Query Responsibility Segregation)

Triggered by: write and read shapes are very different; scale needs differ between reads and writes

Write (Command)
→
Write DB (normalized)
→ async →
Read DB (denormalized)
←
Read (Query)
Example: e-commerce writes normalized order data to Postgres; an async job denormalizes it into Elasticsearch for flexible search queries. Read and write models evolve independently.

Pattern 6 — Event sourcing

Triggered by: audit log required, need to replay state, financial ledger

ConceptDescription
Event logStore every state change as an immutable event — never update/delete
Current stateComputed by replaying events from the beginning (or from a snapshot)
SnapshotPeriodic materialized state to avoid replaying 10M events every read
ProjectionA read model built by consuming the event log — one per query shape
Event sourcing is powerful but complex. Only introduce it when audit trail or replay is a hard requirement. The default is CRUD — don't over-engineer.

Pattern 7 — Transactional outbox

Triggered by: must publish a message to Kafka reliably after a DB write — without dual-write risk

Write to DB
+
Write to outbox table (same TX)
→
Polling relay reads outbox
→
Publish to Kafka
Solves the dual-write problem: if you write to DB then publish to Kafka, you can't guarantee both succeed. Outbox makes the message part of the DB transaction — guaranteed to be published eventually.

Pattern 8 — Consistent hashing

Triggered by: distributing keys across a cache cluster or sharded DB

Consistent hashing wins
  • Adding/removing a node only remaps K/n keys (K=total, n=nodes)
  • No global rehash — cache stampedes avoided
  • Virtual nodes distribute load more evenly across heterogeneous machines
Naive modulo hashing
  • Adding 1 node remaps ~50% of all keys
  • Cache stampede: all remapped keys are cache misses simultaneously
  • Simple to implement — fine for static cluster size

Interview moves

  • 01
    "This is a read-heavy system — I'd cache aggressively in Redis and add read replicas before considering any partitioning" — name the pattern, state the implication
  • 02
    "For the feed, I'd use fan-out on write for most users and fan-out on read for celebrity accounts above 1M followers — same hybrid Twitter uses" — shows awareness of the tradeoff and real-world precedent
  • 03
    "Across these three services I can't use a single DB transaction — I'd use the Saga pattern with compensating transactions for rollback" — naming Saga explicitly signals distributed systems maturity
  • 04
    "To guarantee the Kafka publish happens, I'd use the transactional outbox pattern — the event is written to a DB table in the same transaction as the state change" — this is a differentiating answer most candidates don't know
  • 05
    "For the cache cluster I'd use consistent hashing — adding nodes only remaps a small fraction of keys, avoiding stampedes" — shows you know why, not just what

Common follow-up questions

When does fan-out on write break down?
When a user has millions of followers — writing to 10M feeds per post creates unacceptable write amplification and latency. Switch to pull (fan-out on read) for accounts above a threshold (~500k followers).
What's the difference between Saga choreography and orchestration?
Choreography: services emit events and react to each other's events — no central controller, harder to trace. Orchestration: a saga coordinator calls each service in sequence — easier to observe but adds a single coordinator dependency.
Is CQRS always worth the complexity?
No. CQRS adds a second data model, async sync lag, and operational overhead. Use it when: read/write load is asymmetric by 100x, or when the read and write schemas genuinely diverge. For most CRUD apps, it's overkill.
What's the difference between event sourcing and an audit log?
An audit log is append-only but doesn't necessarily derive current state from events. Event sourcing makes the event log the source of truth — current state is always derived from replaying events. Much higher complexity.
How do you handle the outbox polling delay?
Polling interval is typically 100ms-1s — acceptable for most async workflows. For lower latency, use Debezium (CDC) which streams the DB transaction log directly to Kafka without polling.
What are virtual nodes in consistent hashing?
Each physical node maps to multiple points on the hash ring. This distributes load more evenly and allows heterogeneous nodes (stronger machines get more virtual nodes). Standard in Cassandra and Redis Cluster.
