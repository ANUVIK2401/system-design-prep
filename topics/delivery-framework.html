Delivery Framework

The exact step-by-step structure to use in every system design interview — learn it once, use it every time.

12 min read Most important page
The 5-phase framework (memorize this)
  • 1. Clarify — requirements, users, scale, exclusions (5 min)
  • 2. Estimate — QPS, storage, bandwidth back-of-envelope (3 min)
  • 3. API design — 3-4 core endpoints, request/response shape (5 min)
  • 4. High-level design — components, data flow, data model (15 min)
  • 5. Deep dive — bottlenecks, failure modes, scale tactics (15 min)

Mental model

The framework is your GPS. You might deviate based on traffic (interviewer cues), but you always know where you are in the journey — and so does your interviewer.

Phase 1 — Clarify (5 min)

Ask aboutExample question
Core feature"Is this read-heavy or write-heavy? What's the primary user action?"
Users"Are we building for consumers, businesses, or internal teams?"
Scale"How many DAU should I assume? 1M, 10M, 100M?"
Exclusions"Should I design auth, payments, and notifications — or focus on the core flow?"
SLA"Is strong consistency required, or is eventual consistency acceptable?"
Geography"Single-region or globally distributed?"
Stop after 5 minutes even if you haven't asked everything. State your assumptions and move forward. Over-clarifying is a time management red flag.

Phase 2 — Estimate (3 min)

~1150

~365GB
storage/yr (1KB × 10M × 365)
These numbers drive your decisions. A system at 100 writes/sec needs no sharding. At 100k writes/sec, you need sharding, Kafka, and a dedicated write path. Say this out loud.

Phase 3 — API design (5 min)

Define 3-4 core endpoints. RESTful by default unless the problem demands otherwise (streaming, bidirectional = WebSockets).

What to defineExample (URL Shortener)
Method + pathPOST /urls
Request body{ long_url, custom_alias?, expires_at? }
Response{ short_url, short_code, expires_at }
AuthBearer token in header; anonymous for redirect endpoint
API design shows you think from the client's perspective first. It also constraints your data model — define it early.

Phase 4 — High-level design

Draw on the whiteboard while narrating. Cover these in order:

Components to cover
  • Client + load balancer + API layer
  • Core service(s) and their responsibilities
  • Database(s) — type, schema, why
  • Cache layer — what's cached, eviction policy
  • Async components (queues, workers) if needed
Common omissions
  • Forgetting to specify the data model
  • Not saying why you chose SQL vs NoSQL
  • Skipping the cache entirely
  • Not labeling the read path vs write path
  • Drawing components without explaining data flow

Phase 5 — Deep dive

✓
Proactively identify bottlenecks: "The hot path here is X — if this gets 100x traffic, here's what breaks and how I'd fix it."
✓
Cover failure modes: "What happens if the DB goes down? If the cache is cold? If a downstream service is slow?" Mention detection + recovery for each.
⚠
Follow the interviewer's lead: If they say "tell me more about the DB design", drop everything else and go deep. They're telling you where they want to probe.
✗
Don't go wide here: The deep dive is for depth on one component, not adding more components to your diagram. One thing, done well.

Interview moves

  • 01
    "Let me start by clarifying a few things before I design anything" — say this verbatim as your first sentence, every single interview
  • 02
    "I'll assume 10M DAU — quick math gives me ~115 writes/sec, so no sharding needed initially" — shows you connect estimates to design decisions
  • 03
    "Let me define the API first — it'll constrain my data model" — interviewers love this sequencing; it shows design discipline
  • 04
    "I'm going to draw the write path first, then the read path separately" — separating paths shows clarity; most systems have different optimization needs per path
  • 05
    "Before I go further — is there a component you'd like me to go deeper on?" — check in at the 25-min mark; lets the interviewer steer the deep dive
  • 06
    "The main bottleneck I see is [X]. Here's how I'd handle it at 10x scale..." — proactive bottleneck identification is a strong senior signal

Common follow-up questions

What if the interviewer interrupts my framework flow?
Follow them. The framework is your default path, not a script. If they say "skip ahead to the DB design", do it. You can always say "I'll circle back to X if we have time."
Do I need to cover all 5 phases in every interview?
Clarify and estimate are always required. The interviewer may shortcut others. Watch for "that's fine, let's move on" — it means they're satisfied and want to go deeper elsewhere.
How detailed should the data model be?
Primary entities, key fields, and the main foreign key / partition key. Don't model every column. State your storage choice and justify it. That's enough at the high-level phase.
What's the difference between a system design interview and a design review?
Speed and ambiguity. A design review has a week of context and a 1-pager. An interview has 2 minutes of context and a blank whiteboard. The framework keeps you from spending time you don't have.
When should I bring up monitoring and observability?
In the deep dive, after bottlenecks and failure modes. Say: "I'd instrument this with latency histograms and error rate alerts on the hot path." Proactively mentioning observability is a strong production engineering signal.
How do I avoid running out of time?
Set mental checkpoints: 5 min (requirements done), 8 min (estimates done), 13 min (API done), 28 min (high-level done), last 15 min for deep dive. Glance at the clock after each phase.
