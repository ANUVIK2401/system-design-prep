Key Technologies

Redis

An in-memory data structure server — not just a cache. Strings, hashes, sorted sets, streams, persistence, and cluster sharding.

15-20 min read Medium
Say this in the first 2 minutes
  • Redis is a data structure server, not a generic cache — the structure you pick (ZSET, Hash, Stream) often eliminates app-side logic
  • Single-threaded event loop means every command is atomic — no locking needed for single-key ops
  • Persistence is opt-in and tunable: RDB for fast restarts, AOF for durability, or both
  • Cluster mode shards via 16384 hash slots, not consistent hashing — rebalancing moves whole slots
  • Cache stampede on a hot key expiring is a real failure mode — mitigate with jitter or locking, not just TTL

Mental model

Redis is RAM with a vocabulary. A plain key-value cache only knows GET and SET; Redis adds verbs — push, increment, intersect, rank — so the data itself carries structure instead of your application re-deriving it from a blob on every read.

Eviction policies

When Redis hits maxmemory, the eviction policy decides what gets dropped.

PolicyBehavior
noevictionReject writes with an error once memory is full — reads still work. Default; safest for data you can't lose.
allkeys-lruEvict the least-recently-used key across the entire keyspace, regardless of TTL
volatile-lruEvict least-recently-used key, but only among keys with a TTL set — keys without expiry are protected
allkeys-lfuEvict least-frequently-used key — better than LRU for workloads with stable hot keys and occasional scans
volatile-ttlEvict the key with the nearest expiration time first
volatile-lru is the common production default: it protects keys you intentionally made permanent (e.g. config) while still reclaiming space from expiring cache entries.

Persistence: RDB vs AOF

RDB — point-in-time snapshot
  • Compact single binary file, fast to load on restart
  • Low runtime overhead — forks a child process to write
  • Great for backups and fast cluster node recovery
  • Data loss window: everything since the last snapshot (e.g. 5 min)
AOF — append-only command log
  • Logs every write command — replay on restart rebuilds exact state
  • Configurable fsync: always (durable, slow), everysec (default, ~1s loss window), no
  • Larger file, slower restart — needs periodic rewrite/compaction
  • Much smaller data-loss window than RDB alone
Production default is usually both: AOF for durability, RDB snapshots for fast full-cluster bootstrap and backups. Pure in-memory with no persistence is fine only if Redis is a pure cache backed by a real source of truth.

Cluster mode — hash slot sharding

Masters — own a slot range, serve reads/writes
Replicas — async copy, promoted on master failure
  1. Slot assignment: 16384 total hash slots are statically divided across master nodes at cluster setup
  2. Routing: client hashes the key (CRC16 mod 16384), connects to the master owning that slot directly — no proxy hop
  3. Replication: each master streams writes asynchronously to 1+ replicas for failover, not for read scaling by default
  4. Gossip + failover: nodes exchange heartbeats; if a master misses enough pings, the cluster votes and promotes its replica
  5. Resharding: adding/removing a node moves whole slot ranges, not individual keys — predictable but can be a multi-minute operation at scale

Tradeoffs

Wins
  • Sub-millisecond latency — everything is in RAM
  • Rich data structures eliminate app-side aggregation logic
  • Atomic single-key operations with no external locking
  • Pub/sub and Streams give you messaging for free
  • Cluster mode scales horizontally with predictable sharding
Costs
  • Dataset must fit in RAM — expensive at large scale vs disk-based stores
  • Single-threaded command execution — one slow command (e.g. KEYS *) blocks everything
  • Without persistence, a crash loses all data
  • Cluster mode loses multi-key transactions across different slots
  • Operational overhead: failover, resharding, memory tuning

When to use

✓
USE: sub-ms hot-path reads, leaderboards/ranking, rate limiting, session storage, pub/sub fan-out, or any workload that maps cleanly onto one of Redis's native structures.
✗
AVOID: as a primary system of record for data you can't afford to lose without persistence configured, or for datasets that don't fit in available RAM cost-effectively.
⚠
WATCH OUT: a single very hot key (celebrity user, trending item) can saturate one shard even with cluster mode — needs client-side local caching or key splitting.

  • 04

  • 05
    "Single-threaded means I need to avoid O(N) commands like KEYS or large SMEMBERS in the hot path — I'd use SCAN instead" — operational awareness

Common follow-up questions

What is a cache stampede and how do you prevent it?
Many requests hit a DB simultaneously when a hot key expires. Prevent with a short-lived lock (only one request repopulates) or TTL jitter so keys don't all expire at the same instant.
Why is Redis single-threaded, and is that a problem?
Avoids locking overhead and context switches for in-memory ops, which is faster for typical small commands. It's a problem only for O(N) commands on large collections — use SCAN/pagination instead.
How does Redis Cluster differ from Redis Sentinel?
Sentinel provides HA/failover for a single primary-replica set with no sharding. Cluster mode adds sharding across the 16384 hash slots plus its own failover via gossip — use Cluster when data exceeds one node's RAM.
Can you do transactions in Redis?
MULTI/EXEC queues commands and executes them atomically, but without rollback on individual command failure. In cluster mode, all keys in a transaction must hash to the same slot (use hash tags {tag}).
How would you implement a rate limiter with Redis?
Sliding window with a ZSET: score = request timestamp, trim entries older than the window, count remaining. Or simpler token bucket via INCR + EXPIRE for fixed windows.
What happens on a Redis Cluster master failure?
Other masters detect missed heartbeats via gossip, reach quorum, and promote the failed master's replica. Writes to that slot range fail for the brief detection+failover window (typically a few seconds).
