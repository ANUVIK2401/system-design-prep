Key Technologies

ZooKeeper

A coordination service for distributed systems — leader election, distributed config, and service discovery, built on a simple hierarchical znode tree.

12-18 min read Medium
Say this in the first 2 minutes
  • ZooKeeper's value isn't the database — it's the coordination primitives: leader election, distributed locks, config propagation
  • Leader election uses ephemeral sequential znodes — lowest sequence number wins, and the watch mechanism notifies the next-in-line on failure
  • Ephemeral nodes disappear automatically when a client disconnects — that's the basis for liveness-based service discovery
  • Watches let clients react to changes (config update, node death) without polling
  • It's increasingly replaced by etcd/Consul in newer systems, but the coordination patterns it embodies are exactly what interviewers want you to reason through — name ZooKeeper or etcd, the pattern is what matters

Mental model

ZooKeeper is a shared bulletin board with a strict filing system: everyone can post a note in a folder, see who else posted, and get tapped on the shoulder the instant something changes — which turns "who's in charge right now" into a simple, race-free read of who has the lowest ticket number.

Znode tree and node types

Znode typeBehavior
PersistentExists until explicitly deleted — used for stable config and structure
EphemeralAutomatically deleted when the creating client's session disconnects — the basis for liveness detection
SequentialZooKeeper appends a monotonically increasing counter to the name — guarantees a global, gapless ordering
Ephemeral + SequentialCombines both — the standard building block for leader election and distributed locks
WatchA one-time trigger a client sets on a znode — fires on change, client must re-register to keep watching

Leader election via ephemeral sequential znodes

Current leader — lowest sequence number under /election
Candidates — watch only their immediate predecessor
  1. Register: each candidate creates an ephemeral sequential znode under a shared parent (e.g. /election/n_)
  2. Compare: a candidate lists its siblings; if its own sequence number is lowest, it's the leader
  3. Watch the predecessor, not the leader: every other candidate sets a watch only on the znode immediately before its own — this avoids every node waking up on every change (the "herd effect")
  4. Failover: if the leader's session disconnects (crash, network partition), its ephemeral znode is automatically deleted
  5. Promote: the watch on the deleted znode fires for the next-in-line candidate, which re-checks its rank and becomes leader if it's now lowest

Other coordination patterns

PatternHow ZooKeeper implements it
Distributed configurationConfig stored in a persistent znode; clients set a watch and get notified instantly when it changes — no polling
Service discoveryEach service instance registers an ephemeral znode on startup — it vanishes automatically if the instance crashes or loses connectivity, giving accurate liveness without heartbeats to a separate system
Distributed lockSame ephemeral-sequential pattern as leader election — lowest sequence number holds the lock, releases by disconnecting or deleting its znode
Barrier / group membershipClients register under a shared znode; the count of children tells every member who else is currently part of the group

Tradeoffs

Wins
  • Battle-tested, strongly consistent coordination primitives
  • Watch mechanism avoids polling for config/leadership changes
  • Ephemeral nodes give accurate liveness detection for free
  • Underpins major systems (Kafka historically, HBase, Hadoop)
Costs
  • Operationally heavy — JVM-based, needs an odd-numbered quorum (3, 5) of dedicated nodes
  • Not designed for high-volume data storage — small values only, coordination metadata
  • etcd (Raft, simpler ops model, gRPC API) has displaced it in many newer systems (Kubernetes uses etcd, not ZK)
  • Kafka itself removed its ZooKeeper dependency (KRaft mode) — a sign of the broader trend

When to use

✓
USE: any distributed system needing leader election, distributed locks, or coordinated config — whether via ZooKeeper, etcd, or Consul, the pattern is the point.
✗
AVOID: as a general-purpose data store — it's built for small amounts of coordination metadata, not application data at volume.
⚠
WATCH OUT: a watch only fires once — if you don't re-register it immediately after handling the event, you can miss the next change (a classic correctness bug).

Common follow-up questions

Why use ephemeral sequential znodes specifically for leader election, instead of just ephemeral?
Sequential gives a deterministic, race-free ordering (lowest number wins) without any node needing to coordinate who goes first — pure sequential ephemeral alone can't resolve ties between simultaneous registrations.
What happens during a network partition?
ZooKeeper uses ZAB (a Paxos-like consensus protocol) requiring a quorum majority to make progress — the minority partition can't elect a new leader or accept writes, avoiding split-brain.
How is ZooKeeper different from etcd?
etcd uses Raft (generally considered easier to reason about than ZAB), exposes a gRPC/HTTP API instead of a custom protocol, and has become the default in cloud-native systems like Kubernetes.
Why did Kafka remove its ZooKeeper dependency?
KRaft mode moved metadata management into Kafka itself using a Raft-based quorum of brokers, removing the operational burden of running and scaling a separate ZooKeeper cluster.
How would you implement a distributed lock with ZooKeeper?
Same ephemeral-sequential pattern as leader election: create a sequential znode, you hold the lock if you're lowest, watch your predecessor, and the lock releases automatically if your session dies.
What's the "herd effect" and how does ZooKeeper avoid it?
If every client watched the same znode, one change would wake up and re-trigger logic in all of them simultaneously. Watching only your immediate predecessor means a change wakes at most one other client.
