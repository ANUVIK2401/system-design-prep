Key Technologies

Cassandra

A wide-column, AP-leaning store built for write-heavy, high-availability workloads — schema is designed around the query, not the entity.

15-20 min read Medium-Hard
Say this in the first 2 minutes
  • Cassandra is query-first schema design — you model tables around the exact queries you'll run, denormalizing aggressively, because there are no joins
  • Partition key decides which node owns the data; clustering columns decide sort order within that partition
  • Consistency is tunable per-query (ONE, QUORUM, ALL) — it's not a fixed property of the database
  • Writes are fast by design: append to commit log + memtable, no read-before-write, no random disk I/O on the write path
  • It's AP-leaning on the CAP spectrum — favors availability and partition tolerance over strict consistency by default

Mental model

Cassandra is a write-optimized ledger, not a queryable spreadsheet. You don't ask it new questions after the fact — you decide the questions up front, build a table per question, and it answers each one in constant time because the data is already laid out for that exact lookup.

Wide-column model

ConceptWhat it means
Partition keyDetermines which node(s) own the row — all rows with the same partition key live together, on the same node
Clustering columnsSort order of rows within a partition — enables efficient range scans (e.g. latest 50 events for a user)
DenormalizationSame data duplicated across multiple tables, one per query pattern — storage is cheap, joins are not available
No joinsQuery patterns must be known at schema design time; changing access patterns later often means a new table

Tunable consistency

Cassandra default posture — consistency vs availability (CAP)
Strong consistencyAvailability
Consistency levelBehavior
ONEAcked by a single replica — fastest, weakest guarantee, used for high-throughput non-critical writes
QUORUMAcked by majority of replicas (e.g. 2 of 3) — common production default, balances latency and consistency
ALLAcked by every replica — strongest read-after-write guarantee, lowest availability (any replica down blocks the request)
Read QUORUM + write QUORUM gives you strong consistency for that specific query, since the read and write replica sets are guaranteed to overlap by at least one node. This is set per-query, not globally.

Write path

Commit log — durability, replayed on crash recovery
Memtable — in-memory, what makes writes fast
SSTable + compaction — eventual on-disk durable form
  1. Commit log: write appended sequentially to disk first — guarantees durability if the node crashes before flush
  2. Memtable: write also applied to an in-memory sorted structure — this is what makes the write path so fast, no random I/O
  3. Ack: write is acknowledged to the client once both succeed, satisfying the requested consistency level across replicas
  4. Flush: when the memtable fills up, it's flushed to disk as an immutable SSTable
  5. Compaction: background process merges SSTables over time, discarding overwritten or tombstoned (deleted) data

Tradeoffs

Wins
  • Extremely fast writes — no read-before-write, sequential disk I/O
  • Linear horizontal scalability — add nodes, capacity grows proportionally
  • No single point of failure — masterless, peer-to-peer architecture
  • Multi-region active-active writes are a first-class capability
Costs
  • No joins — every new query pattern often needs a new denormalized table
  • Eventual consistency by default — strong consistency costs latency (QUORUM/ALL)
  • Deletes create tombstones that must be compacted away — can degrade read performance if mismanaged
  • Schema changes are harder once query patterns shift — redesign means backfilling new tables

When to use

✓
USE: write-heavy, high-availability workloads with known query patterns — time-series/IoT sensor data, activity feeds, messaging history.
✗
AVOID: ad-hoc analytical queries, workloads needing joins, or anything where the access pattern isn't known upfront and changes frequently.
⚠
WATCH OUT: heavy delete-and-rewrite workloads generate tombstones that slow reads until compaction catches up — design around updates/appends where possible.

Common follow-up questions

Why is Cassandra's write path so much faster than a typical relational DB?
No read-before-write and no random disk I/O — it's a sequential append to a commit log plus an in-memory write, with reconciliation (compaction) deferred to the background.
How do you choose a partition key?
High cardinality to spread load evenly across nodes, but matching your primary query's filter (e.g. user_id) so a single query hits one partition instead of scanning the whole cluster.
What's a tombstone and why does it matter?
A marker written on delete instead of immediate removal — reads must skip past tombstones until compaction physically removes them. Too many tombstones in a partition slows reads significantly.
How does Cassandra achieve high availability with no master?
Peer-to-peer gossip protocol — every node can accept reads/writes for the data it owns or coordinates. Replication factor (commonly 3) spreads copies so node failure doesn't mean data loss.
What happens if you need a new query pattern you didn't design for?
Usually means creating a new denormalized table for that pattern and backfilling it — there's no ad-hoc join to fall back on. This is the main schema-evolution cost of the model.
Cassandra vs DynamoDB — when would you pick one over the other?
Cassandra for multi-region active-active writes outside AWS or to avoid vendor lock-in. DynamoDB for zero-ops on AWS with comparable performance and built-in autoscaling.
