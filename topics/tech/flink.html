Key Technologies

Flink

A true stream processing engine — continuous computation over unbounded data, with windowing and checkpointed state for low-latency exactly-once results.

15-20 min read Hard
Say this in the first 2 minutes
  • Flink treats streams as the primitive — batch is just a stream with a known end, not the other way around
  • Windowing buckets an infinite stream into finite chunks for aggregation: tumbling (fixed, non-overlapping), sliding (fixed, overlapping), session (gap-based, dynamic)
  • Keyed state lets each key (e.g. user_id) maintain its own running state across events, partitioned like Kafka partitions
  • Checkpointing periodically snapshots all operator state — recovery replays from the last checkpoint, giving exactly-once semantics
  • Reach for Flink over Kafka Streams or Spark Streaming when you need true low-latency processing with complex, large keyed state

Mental model

Batch processing is reading a finished book cover to cover. Stream processing is reading a book as pages arrive forever — you maintain a running understanding (state) and periodically save your place (checkpoint) so a crash doesn't mean starting over.

Windowing

Windows bucket an unbounded stream into finite chunks so aggregations (count, sum, average) have a defined scope.

Window typeBehaviorUse case
TumblingFixed size, non-overlapping — each event belongs to exactly one window"Requests per 5-minute bucket"
SlidingFixed size, overlapping by a slide interval — an event can belong to multiple windows"Rolling 5-min average, updated every 30s"
SessionDynamic size, closes after a gap of inactivity — no fixed boundary"Group a user's clicks into one session"

Stream → window → sink, with checkpointing

Keyed state — per-key, partitioned like Kafka partitions
Windowing — buckets the stream for aggregation
Checkpointing — periodic durable snapshot for recovery
  1. Source: consume an unbounded stream, typically from Kafka
  2. keyBy: partition the stream by key so each key's state and processing is independent and parallelizable
  3. Window: bucket events by time (tumbling/sliding/session) and apply an aggregation function
  4. Checkpoint: Flink periodically injects a barrier into the stream; as it flows through each operator, that operator snapshots its state to durable storage
  5. Recover: on failure, restart from the last checkpoint and replay the source from that offset — combined with idempotent sinks, this gives exactly-once results

Tradeoffs

Wins
  • True low-latency stream processing — milliseconds, not micro-batches
  • Rich, large keyed state managed natively with efficient checkpointing
  • Exactly-once semantics via checkpoint barriers, even across complex pipelines
  • Flexible windowing (event-time, watermarks for out-of-order events)
Costs
  • Steep operational learning curve — cluster, state backends, checkpoint tuning
  • Heavier to run than Kafka Streams for simple stateless transformations
  • Large state can make checkpoints slow if not tuned (incremental checkpointing helps)
  • Smaller ecosystem/talent pool than Spark

When to use

✓
USE: true low-latency stream processing with complex stateful logic — fraud detection, real-time recommendations, sessionization, anomaly detection.
✗
AVOID: simple stateless filtering/transformation embedded in one service — Kafka Streams is a lighter-weight library with no separate cluster to operate.
⚠
WATCH OUT: out-of-order or late-arriving events need watermarks and an explicit lateness policy, or window results will be silently wrong/incomplete.

Common follow-up questions

How does Flink achieve exactly-once semantics?
Checkpoint barriers flow through the pipeline; every operator snapshots state when the barrier passes. On failure, the job restarts from the last consistent checkpoint and replays the source from that point.
Flink vs Spark Streaming — what's the core difference?
Flink is a native stream processor with true per-event processing. Spark Structured Streaming uses micro-batching — processes small batches on an interval, which adds latency Flink avoids.
Flink vs Kafka Streams — when would you pick Flink?
Kafka Streams is a lightweight library embedded in your app, simpler to deploy. Flink is a separate cluster better suited for complex stateful jobs, large state, and advanced windowing/watermark needs.
What's a watermark and why does it matter?
A watermark is Flink's estimate of "we've seen all events up to time T" — it lets windows close and emit results despite some events arriving out of order, with a configurable allowed lateness.
How does keyed state scale?
State is partitioned by key, similar to Kafka partitions — each parallel task owns a subset of keys and their state, so scaling out adds more parallel state partitions.
What happens if checkpoints become too slow due to large state?
Use incremental checkpointing (RocksDB state backend) which only persists state deltas since the last checkpoint, rather than a full snapshot every time.
