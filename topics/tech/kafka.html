Key Technologies

Kafka

A distributed, partitioned, replicated commit log — partitions are the unit of parallelism, consumer groups are the unit of scale.

15-20 min read Medium
Say this in the first 2 minutes
  • A topic is split into partitions — ordering is guaranteed only within a partition, never across the whole topic
  • Partition count is your parallelism ceiling: more consumers than partitions in a group just sit idle
  • Each partition is consumed by exactly one consumer within a consumer group — that's how horizontal scaling works
  • At-least-once + idempotent consumer is the practical default; exactly-once (idempotent producer + transactions) is for when duplicates are unacceptable and can't be deduped downstream
  • Consumer lag (how far behind the latest offset a group is) is your single most important Kafka health metric

Mental model

A Kafka topic is a set of append-only logs, not a queue. Consumers don't remove messages — they track their own bookmark (offset) and can rewind it. That's what makes replay, multiple independent consumers, and durability all fall out of the same simple structure.

Partitions and consumer groups

ConceptWhat it means
PartitionAn ordered, append-only log. The unit of parallelism — more partitions = more concurrent consumers possible
Ordering guaranteeOnly within a single partition. Across partitions, no ordering is guaranteed at all
Partition keyHash of the key picks the partition — same key always lands on the same partition, preserving per-key order
Consumer groupA set of consumers sharing the work of a topic — Kafka assigns each partition to exactly one consumer in the group
Consumer lagDifference between the latest produced offset and a consumer group's committed offset — the key health signal
RebalancingReassigning partitions across consumers when one joins/leaves the group — briefly pauses consumption
Picking partition count: if you expect to scale to 12 consumers eventually, provision at least 12 partitions up front — you can add partitions later, but that breaks key-based ordering for existing keys since the hash-to-partition mapping changes.

Producer → partitions → consumer group

Partitions — ordered logs, the parallelism unit
Consumer group — 1:1 partition-to-consumer mapping
  1. Produce: producer hashes the message key, deterministically picks a partition
  2. Append: message is written to the end of that partition's log, replicated to followers before ack (if acks=all)
  3. Assign: Kafka's group coordinator assigns each partition to exactly one consumer in the group
  4. Consume + commit: consumer reads sequentially, processes, commits its offset — restart resumes from the last committed offset
  5. Scale: adding a 4th consumer to a 3-partition topic leaves it idle — partition count is the hard ceiling on parallelism

Delivery guarantees

At-least-once + idempotent consumer
  • Default and simplest — retries on failure may redeliver a message
  • Consumer dedups by a unique key (e.g. UPSERT on order_id) so reprocessing is harmless
  • Lower overhead, easier to reason about, scales further
  • Good enough for the vast majority of real systems
Exactly-once semantics (EOS)
  • Idempotent producer (sequence numbers dedupe retries) + transactional writes across produce + offset commit
  • Needed when duplicates genuinely can't be tolerated and can't be deduped downstream (e.g. financial ledger entries)
  • Higher latency and throughput cost — full transaction coordination per batch
  • Only spans Kafka-to-Kafka; doesn't make external side effects (an API call) exactly-once
Most interview answers should default to "at-least-once with an idempotent consumer" and only reach for EOS when explicitly pressed on duplicate-sensitive correctness (e.g. payments).

Tradeoffs

Wins
  • Decouples producers from consumers — independent scaling and deploys
  • Durable, replayable log — new consumers can read from the beginning
  • Massive throughput — sequential disk I/O, batched writes
  • Multiple independent consumer groups can read the same topic
Costs
  • No ordering across partitions — must design partition keys carefully
  • Operational complexity: brokers, ZooKeeper/KRaft, partition rebalancing
  • Rebalancing pauses consumption briefly when group membership changes
  • Overkill for simple point-to-point task queues (<10k msgs/sec) — SQS/RabbitMQ is simpler

When to use

✓
USE: event-driven fan-out (one event, many independent consumers), audit/event log, decoupling write-heavy producers from slower consumers, >100k events/sec.
✗
AVOID: simple request-response or point-to-point task queues at modest scale — a managed queue (SQS) is operationally simpler and sufficient.
⚠
WATCH OUT: growing consumer lag means consumers can't keep up with producers — alert on it before it becomes hours of backlog, not after.

Interview moves

  • 01
    "I'd partition by user_id so all events for one user stay ordered, while different users process in parallel" — shows you understand the ordering-vs-parallelism tradeoff explicitly
  • 02

  • 03
    "At-least-once with an idempotent consumer is the default here — exactly-once only matters if duplicates are truly unrecoverable, like a payment ledger" — calibrates the right level of guarantee instead of over-engineering
  • 04
    "I'd alert on consumer lag, not just broker uptime — a healthy cluster with a backed-up consumer group is still a production incident" — operational maturity

Common follow-up questions

Why does Kafka only guarantee ordering within a partition?
Each partition is a single append-only log handled by one broker leader at a time. Guaranteeing global order across partitions would mean serializing all writes through one node, killing throughput.
What causes consumer rebalancing and why is it disruptive?
Triggered when a consumer joins, leaves, or is considered dead (missed heartbeat). All partitions get reassigned, briefly pausing consumption across the whole group during the rebalance.
How do you choose the number of partitions?
Base it on target throughput ÷ per-partition throughput, and your expected max consumer count, since consumers beyond partition count sit idle. Err generous — you can add but not safely shrink.
What's the difference between Kafka and a traditional message queue?
A queue removes a message once consumed — one logical consumer. Kafka retains messages for a configured period; multiple independent consumer groups can each read the full stream at their own pace.
How does Kafka achieve durability?
Each partition is replicated across multiple brokers (replication factor, typically 3). A write is acknowledged once it's written to the leader and, with acks=all, replicated to enough followers.
When would you NOT use Kafka?
Low-volume task queues, simple background jobs, or when you need true exactly-once delivery to an external non-Kafka system without building idempotency yourself — a managed queue or workflow engine is simpler.
