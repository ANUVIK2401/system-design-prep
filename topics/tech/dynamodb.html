Key Technologies

DynamoDB

A fully managed, partition-key-sharded NoSQL store built for predictable-access-pattern workloads at massive scale with zero ops overhead.

12-18 min read Medium
Say this in the first 2 minutes
  • Partition key hashes to a physical partition — pick a high-cardinality key or you create a hot partition
  • GSI = different partition/sort key, its own throughput, eventually consistent; LSI = same partition key, alternate sort key, can be strongly consistent, must exist at table creation
  • Two capacity modes: provisioned (predictable cost, needs autoscaling) vs on-demand (pay-per-request, handles spiky traffic with zero planning)
  • Single-table design is the advanced DynamoDB pattern — multiple entity types in one table to support several access patterns with one query each
  • It's fully managed — no ops overhead, but you trade away joins and ad-hoc queries for that simplicity

Mental model

DynamoDB is a giant filing cabinet sorted by a label you choose up front. If you know exactly which drawer (partition key) you need, retrieval is instant — but you can't flip through every drawer looking for something; you must have planned your labeling system before you needed it.

Partition keys and hot partitions

ConceptWhat it means
Partition keyHashed to determine the physical partition — all items with the same key live together
Sort key (optional)Orders items within a partition, enables range queries (e.g. all orders for a user, newest first)
Hot partitionLow-cardinality or skewed partition key (e.g. a single popular tenant_id) concentrates traffic on one physical partition, throttling it
Choosing a keyPick high-cardinality, evenly-accessed values — composite keys (e.g. userId#date) spread load when a natural key alone is too coarse

GSI vs LSI

Global Secondary Index (GSI)Local Secondary Index (LSI)
Partition keyDifferent from the base tableSame as the base table
Sort keyDifferent attribute, your choiceAlternate attribute, same partition
ConsistencyEventually consistent onlyCan be strongly consistent
When createdAnytime, even after table creationOnly at table creation — cannot be added later
CapacityIts own provisioned throughputShares the base table's throughput
LSI's "must exist at table creation" constraint is the single biggest reason teams default to GSIs — you rarely know every access pattern upfront, and GSIs let you add new query paths as requirements evolve.

Partition routing and capacity modes

Healthy partitions — high-cardinality key spreads load
Hot partition — low-cardinality key concentrates traffic
Provisioned + auto-scaling
  • You set read/write capacity units; auto-scaling adjusts within bounds based on utilization
  • Cheaper at steady, predictable traffic
  • Requires capacity planning and tuning scaling policies
  • Can still throttle on sudden spikes faster than auto-scaling reacts
On-demand
  • Pay per request, no capacity planning needed
  • Instantly absorbs traffic spikes — scales to previous peak automatically
  • More expensive per-request at sustained high, predictable volume
  • Best for unpredictable or new workloads where you don't know the traffic shape yet

Tradeoffs

Wins
  • Fully managed — no servers, patching, or cluster ops
  • Single-digit millisecond latency at virtually unlimited scale
  • Seamless horizontal scaling via automatic partition splitting
  • Built-in global tables for multi-region active-active replication
Costs
  • No joins, no ad-hoc queries — access patterns must be designed upfront
  • AWS vendor lock-in
  • Hot partitions throttle even when overall table capacity is fine
  • Single-table design (the advanced pattern) has a real learning curve and reduces query flexibility for ease of scaling

When to use

✓
USE: massive-scale workloads with well-known, stable access patterns — session stores, shopping carts, gaming leaderboards, IoT device state — especially already on AWS.
✗
AVOID: ad-hoc analytical queries, evolving/unknown access patterns, or anything requiring complex joins across entities.
⚠
WATCH OUT: a low-cardinality partition key (e.g. status: "active"/"inactive") creates a hot partition that throttles regardless of how much total table capacity you provision.

  • 02

  • 03

  • 04
    "I'd briefly mention single-table design as an option here, but keep it simple with separate tables unless we have a specific reason to optimize for fewer round trips" — shows awareness without over-engineering

Common follow-up questions

What is single-table design and why would you use it?
Storing multiple entity types in one DynamoDB table using generic partition/sort key names, so a single query can retrieve related entities together. Reduces round trips but adds real schema complexity.
How do you fix a hot partition?
Add a random or calculated suffix to the partition key to spread writes (write sharding), or redesign the key to a higher-cardinality attribute that better matches actual access patterns.
Why can't you add an LSI after table creation?
LSIs share the base table's partition structure and storage internally — DynamoDB needs to know about every LSI when items are first written so it can maintain them inline, hence the upfront requirement.
DynamoDB vs Cassandra — how do you choose?
DynamoDB for zero-ops on AWS with comparable performance. Cassandra for multi-cloud/on-prem flexibility or when you specifically need tunable consistency per query rather than DynamoDB's simpler consistency model.
How does DynamoDB handle strong consistency?
Reads default to eventually consistent (cheaper, faster) but support a strongly consistent read flag that reads from the leader replica — costs roughly 2x the read capacity of an eventually consistent read.
What's DynamoDB Streams used for?
A change log of item-level modifications, consumable by Lambda or Kinesis — common for triggering downstream processing, replicating to Elasticsearch, or building event-driven workflows off table changes.
