Key Technologies

Elasticsearch

A distributed search engine built on an inverted index — fed from your real database, never the source of truth itself.

15-20 min read Medium
Say this in the first 2 minutes
  • Elasticsearch is a secondary, read-optimized index — never the system of record
  • It's fast at full-text search because of the inverted index: term → list of doc IDs, not row scans
  • Data lands in ES via CDC or dual-write from the real DB, with a near-real-time refresh delay (~1s default)
  • Sharding (primary + replica) gives both horizontal scale and read availability
  • A relational DB's native full-text search (e.g. Postgres tsvector) is "good enough" up to ~10M rows — justify ES with relevance scoring, fuzzy match, or facets, not just "search"

Mental model

A relational DB index speeds up "find the row where id = 5." An inverted index flips that: it answers "find every document containing this word" by storing the word once and pointing to every document it appears in — like a book's back-of-the-page index instead of reading cover to cover.

Inverted index

A forward index maps doc → words. An inverted index maps word → docs. Search becomes a set lookup instead of a scan.

ES vs a database's native search

NeedPick
Exact match, small dataset (<10M rows)DB native (WHERE, LIKE, tsvector) — one less system to operate
Relevance-ranked full-text searchElasticsearch — BM25 scoring out of the box
Faceted search / aggregations at scaleElasticsearch — built for filter + aggregate combos
Fuzzy matching, typo tolerance, autocompleteElasticsearch — edit-distance and n-gram analyzers
Strong read-your-writes consistencyDB native — ES refresh interval adds lag (see below)

Sharding

Primary shards — fixed at index creation, can't reshard without reindexing
Replica shards — read scale + failover
Primary shard count is set at index creation and is expensive to change — pick it based on projected data volume, not current volume. Replica count can be changed anytime.

Tradeoffs

Wins
  • Relevance-ranked full-text search out of the box (BM25)
  • Fast faceted/aggregation queries across large datasets
  • Horizontally scales via sharding, built-in replication
  • Fuzzy matching, autocomplete, geo queries built in
Costs
  • Near-real-time only — default 1s refresh interval before new docs are searchable
  • Operationally heavy: JVM tuning, shard sizing, cluster management
  • No real transactions or strong consistency — eventually consistent by design
  • Must stay in sync with the real DB — adds a CDC/dual-write pipeline to maintain

When to use

✓
USE: product search with relevance ranking, log/event analytics, autocomplete, or any query mixing full-text + filters + aggregations at scale.
✗
AVOID: as your primary transactional store — no real ACID guarantees, and losing the index just means rebuilding it from the source of truth.
⚠
WATCH OUT: read-your-write expectations break — a user who just created a record may not see it in search results for up to the refresh interval (~1s default, tunable but costs indexing throughput).

Interview moves

  • 01
    "Postgres stays the source of truth; I'd feed Elasticsearch via CDC so search never blocks the write path" — the single most important sentence in any ES design
  • 02
    "Given we need typo-tolerant search with relevance ranking, a DB LIKE query won't cut it — that's exactly what ES's inverted index and BM25 scoring are for" — justifies the technology with the specific requirement
  • 03
    "There's a near-real-time gap — if the product spec needs read-your-writes, I'd flag that ES isn't the right fit for that specific flow" — shows you understand the consistency tradeoff, not just the happy path
  • 04
    "I'd size primary shards for projected 2-year data volume since resharding means reindexing" — operational foresight

Common follow-up questions

How do you keep Elasticsearch in sync with the source database?
CDC (Debezium reading the DB's write-ahead log) into a queue, then a consumer indexes into ES. Dual-write is simpler but risks the two stores drifting apart on partial failure.
What happens if Elasticsearch goes down?
Search degrades or fails, but the source DB is untouched since ES isn't authoritative. Recovery is a reindex from the DB or CDC replay — no data is permanently lost.
Why can't you just reshard an ES index like you would a database?
Primary shard count is fixed at index creation because document routing (hash → shard) depends on it. Changing it requires creating a new index with the new shard count and reindexing all data.
How does Elasticsearch rank results?
BM25 by default — scores based on term frequency, inverse document frequency, and field length normalization. Custom scoring functions (boosts, decay by recency) can be layered on top.
When would plain Postgres full-text search be enough?
Up to roughly 10M rows with moderate query volume and no need for complex facets — Postgres tsvector + GIN index handles it without adding an entire extra system to operate.
What's the refresh interval and why does it matter?
New/updated documents aren't searchable until the next refresh (default ~1s) — it batches index segment writes for efficiency. Lowering it improves freshness but increases indexing overhead.
