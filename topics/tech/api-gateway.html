Key Technologies

API Gateway

A single front door that centralizes auth, rate limiting, routing, and request shaping so backend services don't each reimplement them.

12-18 min read Easy-Medium
Say this in the first 2 minutes
  • The gateway exists to centralize cross-cutting concerns — auth, rate limiting, routing — so every backend service isn't reimplementing them
  • It validates JWTs/OAuth tokens once at the edge; backend services trust a verified identity header instead of re-checking credentials
  • Rate limiting belongs at the edge (token bucket or sliding window) — stops abusive traffic before it reaches any backend
  • It becomes a single point of failure and a latency bottleneck if not designed for horizontal scale — mention this proactively
  • Request transformation (protocol translation, payload shaping) lets backends evolve independently of the public-facing contract

Mental model

An API gateway is a building's front desk: it checks ID once, decides which floor you're allowed on, and meters how many people go up at a time — so no individual office has to re-verify every visitor itself.

Core responsibilities

ResponsibilityWhat it does
AuthenticationValidates JWT/OAuth token signature and expiry once; forwards a trusted identity header downstream
Rate limitingToken bucket or sliding window per client/API key, rejecting excess requests before they reach backends
RoutingPath-based (/users/* → user service) or header-based (API version, tenant) routing to the right backend
Request transformationProtocol translation (REST↔gRPC), request/response reshaping, header injection
ObservabilityCentralized logging, tracing, and metrics for every request crossing the edge
TLS terminationDecrypts HTTPS once at the edge; internal traffic can run plain HTTP/gRPC inside a trusted network

Request flow

Gateway — auth, rate limit, route, transform
Backend services — trust the gateway's verified identity
  1. Authenticate: validate JWT signature and expiry against the issuer's public key — reject with 401 before touching any backend
  2. Rate limit: check the client's token bucket / sliding window — reject with 429 if exceeded
  3. Route: match the request path/headers to the target backend service
  4. Transform: reshape the request (strip internal headers, translate protocol) and forward with a trusted identity header
  5. Aggregate (optional): some gateways fan out to multiple services and merge responses (BFF pattern) for mobile-optimized payloads

Tradeoffs

Wins
  • Auth/rate-limiting logic lives in one place, not duplicated across N services
  • Backend services can change internal protocols without breaking public clients
  • Centralized observability — every request is logged/traced at one chokepoint
  • Simplifies client code — one endpoint instead of N service URLs
Costs
  • Single point of failure if not deployed redundantly across instances/regions
  • Adds a network hop of latency to every request
  • Can become a bottleneck under load if it does too much synchronous work
  • Another piece of infrastructure to operate, version, and deploy safely
Mitigate the SPOF/bottleneck risk: deploy multiple stateless gateway instances behind a load balancer, keep gateway logic lightweight (no heavy business logic), and consider regional gateways for global traffic to cut cross-region latency.

When to use

✓
USE: any system with more than a couple of backend services that all need auth, rate limiting, or a unified public API surface.
✗
AVOID: putting heavy business logic or stateful processing inside the gateway — it should stay thin; logic belongs in backend services.
⚠
WATCH OUT: a misconfigured gateway (bad routing rule, broken auth check) takes down access to every backend simultaneously — treat gateway config changes with the same rigor as a DB migration.

Common follow-up questions

How does the gateway avoid becoming a bottleneck?
Keep it stateless and horizontally scaled behind a load balancer; avoid synchronous heavy work (DB calls, business logic). Cache auth verification keys locally instead of calling an auth service per request.
What's the difference between an API gateway and a load balancer?
A load balancer distributes traffic across instances of one service. A gateway routes across many different services and adds auth, rate limiting, and transformation — often sits in front of multiple load balancers.
How would you implement rate limiting at the gateway?
Token bucket or sliding window counter keyed by API key/user ID, backed by Redis for shared state across gateway instances. Return 429 with a Retry-After header when exceeded.
How do backend services trust the identity the gateway asserts?
The gateway signs an internal trusted header (or re-issues a short-lived internal JWT) after validating the original token — services verify that signature instead of re-validating the original credential.
What is the Backend-for-Frontend (BFF) pattern and how does it relate?
A BFF is a gateway variant tailored per client type (mobile vs web) that aggregates multiple backend calls into one optimized response — reduces client-side round trips and over-fetching.
How do you handle gateway config changes safely?
Treat routing/auth rules as versioned config deployed through CI/CD with canary rollout and automated rollback — a bad rule can silently break every service behind it at once.
