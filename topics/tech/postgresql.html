Key Technologies

PostgreSQL

The default choice until a specific requirement forces a distributed store — vertical scaling, read replicas, and partitioning carry it further than most interviews assume.

15-20 min read Medium
Say this in the first 2 minutes
  • Default to Postgres unless a specific requirement forces a distributed DB — most systems never hit its ceiling
  • B-tree is the default index; GIN is for full-text search and JSONB containment queries
  • Streaming replication (physical, byte-for-byte) for read replicas; logical replication for selective, cross-version replication
  • Sync replication guarantees zero data loss on failover at a latency cost; async is faster but risks losing the last unreplicated writes
  • Don't reach for sharding until vertical scaling + read replicas + partitioning are exhausted — that ceiling is much higher than people assume

Mental model

Postgres is a single very capable office that you can keep expanding — more staff (read replicas), better filing (indexes), bigger filing cabinets (partitioning) — before you ever need to open a second office (sharding) with all the coordination overhead that implies.

Indexing strategies

Index typeUse case
B-tree (default)Equality and range queries on ordered data — the default for almost every column you index
GINFull-text search (tsvector), JSONB containment queries (@>), array membership
GiSTGeometric data, range types, nearest-neighbor search
Partial indexIndexes only rows matching a WHERE clause (e.g. WHERE deleted_at IS NULL) — smaller, faster for that subset
Covering indexIncludes extra columns via INCLUDE so the query is answered from the index alone — avoids a heap fetch
A partial index on WHERE status = 'pending' on a table where 95% of rows are 'completed' can be 20x smaller than a full index — and faster for the queries that actually filter on that condition.

Replication: write path vs read path

Primary — single writer, source of truth
Replicas — scale reads, replay the write-ahead log (WAL)
Replication typeBehavior
Streaming (physical)Byte-for-byte WAL replay — replica is an exact copy, must match Postgres major version
LogicalReplicates at the row/table level — supports selective replication and cross-version upgrades
SynchronousPrimary waits for replica ack before committing — zero data loss, higher write latency
AsynchronousPrimary commits immediately, replica catches up — lower latency, risks losing last writes on failover

When Postgres scales "enough"

Default bias — stay on Postgres vs reach for a distributed DB
Stick with PostgresGo distributed
  1. Vertical scaling first: modern hardware (64+ cores, TBs of RAM, NVMe) carries a single Postgres instance through tens of thousands of writes/sec for many workloads
  2. Read replicas next: offload read traffic horizontally — most systems are read-heavy, so this alone buys huge headroom
  3. Partitioning next: split a huge table by range (date) or hash on a single instance — keeps indexes small and vacuum fast without distributing anything
  4. Only then, sharding/distributed DB: reach for Citus, CockroachDB, or a NoSQL store when write volume genuinely exceeds one primary, or you need true multi-region active-active writes
Common interview trap: jumping straight to "we'll shard the database" for a system that would comfortably run on a single well-tuned Postgres instance with replicas. Justify the jump with a specific number, not a vibe.

Tradeoffs

Wins
  • Full ACID transactions, mature query planner, joins that actually work
  • JSONB gives semi-structured flexibility without leaving SQL
  • Enormous ecosystem — extensions, tooling, managed offerings everywhere
  • Vertical + replica + partition scaling goes further than most systems need
Costs
  • Single-writer architecture — write throughput ultimately caps on one primary
  • Multi-region active-active writes aren't native — needs an external solution
  • Vacuum/bloat management requires ongoing operational attention
  • Horizontal write scaling (sharding) isn't built-in — needs Citus or app-level sharding

When to use

✓
USE: the default starting point for almost any system — relational data, need for transactions, unclear future access patterns.
✗
AVOID (alone): when you have a confirmed requirement for multi-region active-active writes or write volume that demonstrably exceeds single-primary capacity even after partitioning.
⚠
WATCH OUT: async replication means failover can lose the last few writes — if that's unacceptable, you need at least one synchronous replica, accepting the latency cost.

Interview moves

  • 01
    "I'd default to Postgres here and only introduce a distributed store if a specific requirement — write volume or multi-region — forces it" — the single highest-signal sentence in this whole topic
  • 02
    "Before sharding, I'd exhaust read replicas and partitioning — those alone often carry a system another order of magnitude in scale" — shows the scaling ladder, not a binary choice
  • 03
    "For this JSONB column with containment queries, I'd add a GIN index rather than a B-tree, since B-tree can't efficiently support @> lookups" — concrete indexing knowledge
  • 04
    "Given we can't lose a single committed write here, I'd make at least one replica synchronous, accepting the added write latency" — explicit tradeoff reasoning, not just a feature list

Common follow-up questions

When would you actually need to shard Postgres?
When sustained write throughput exceeds what a single well-tuned primary can handle even after partitioning — typically tens of thousands of writes/sec sustained, or a hard multi-region active-active write requirement.
What's the difference between partitioning and sharding?
Partitioning splits a table across multiple physical segments on the same instance (still one writer). Sharding splits data across multiple independent database instances/nodes.
How do you decide between synchronous and asynchronous replication?
Synchronous when zero data loss on failover is a hard requirement (financial transactions) — accept the latency cost. Asynchronous when slightly stale failover is acceptable for lower write latency.
What is replication lag and how do you handle it?
The delay between a write on the primary and its visibility on a replica. For read-your-writes needs, route that specific read to the primary or track a "read after write" hint to wait for replica catch-up.
What's the difference between streaming and logical replication?
Streaming replicates the entire WAL byte-for-byte (must match major version, used for HA/read replicas). Logical replicates row changes selectively, enabling cross-version replication and use with CDC tools like Debezium.
How would you handle a table that's grown to hundreds of millions of rows?
Range or hash partition it (e.g. by month for time-series data) to keep individual indexes and vacuum operations fast, before considering sharding across instances.
