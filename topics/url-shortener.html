URL Shortener

Convert long URLs to short aliases and redirect users at scale

30-40 min interview

Say in the first 5 min

  • Clarify read vs write ratio — shorteners are read-heavy (100:1), cache aggressively
  • Pick hash length early: 7 chars base62 = 62^7 = 3.5 trillion URLs
  • ID generation is the hard problem — not the redirect logic
  • Expiration + analytics are the two features that complicate the data model
  • A CDN in front of the redirect service cuts latency to under 10ms globally

Data model

-- urls table (PostgreSQL)
CREATE TABLE urls (
  id          BIGINT PRIMARY KEY,        -- Snowflake ID
  short_code  VARCHAR(10) UNIQUE NOT NULL, -- base62(id), e.g. "gH3xK2p"
  long_url    TEXT NOT NULL,
  user_id     BIGINT,                    -- nullable (anonymous)
  created_at  TIMESTAMP DEFAULT NOW(),
  expires_at  TIMESTAMP,                 -- null = never
  click_count BIGINT DEFAULT 0           -- denormalized for quick reads
);

CREATE INDEX idx_short_code ON urls(short_code);  -- lookup hot path
Storage estimate: 100M rows × ~500B avg = 50GB. Fits on a single Postgres instance with read replicas for scale. Partition by created_at monthly if you hit 1B+ rows.

Key design decisions

ID generation: Snowflake over hash
Hashing (MD5, SHA) risks collisions and requires a uniqueness check on every write. Snowflake IDs are globally unique by construction — no DB round-trip needed to verify.
Tradeoff: Snowflake requires a coordination service (or node ID config). If you use DB auto-increment instead, it's simpler but becomes a write bottleneck above ~10k writes/s.
301 (permanent) vs 302 (temporary) redirect
302 means every click hits your servers (analytics works). 301 means browsers cache it and you lose click visibility. Default to 302 unless the client explicitly opts into 301.
Tradeoff: 302 = more server load. 301 = no analytics after first visit per user. For business-critical analytics, always 302.
Cache at redirect service + CDN layer
Two-layer caching: Redis holds the hot 20% of URLs in memory. CDN caches the HTTP 302 responses at edge nodes for globally fast reads.
Tradeoff: 302 responses can be cached by CDN but stale cache is a problem on URL deletion. Solve with a low TTL (60s) or purge API call on delete.
Custom aliases: separate namespace
User-defined aliases (e.g. /my-brand) go into a separate custom_aliases table to avoid colliding with Snowflake-generated codes.
Tradeoff: Two lookup paths in redirect service. Keep it simple with a single DB query that checks both tables via UNION or a type column.

Bottlenecks and scale tactics

BottleneckTriggerFix

URL expiration cleanup100M expired rows accumulateBackground job (cron) deletes expired rows in batches. Soft-delete with deleted_at to avoid blocking live reads.

Consistency model

Writes: strong consistency — new short URL is immediately readable (write to primary Postgres, return to client).
Reads after cache miss: eventual — read replicas may lag primary by ~100ms. Acceptable because a brand-new URL's traffic ramps up slowly.
Deletes: cache TTL + CDN purge means a deleted URL can still resolve for up to 60s. Document this in your SLA.

API design

POST/api/v1/shorten
Request: { "long_url": "https://...", "custom_alias": "my-link", "expires_at": "2026-12-31" }
Response: { "short_url": "https://sho.rt/gH3xK2p", "short_code": "gH3xK2p", "expires_at": "..." }
GET/{short_code}
Response: HTTP 302 with Location: https://original-long-url.com/...
Error: HTTP 404 if not found or expired
DELETE/api/v1/urls/{short_code}
Auth required. Soft-deletes the record, triggers CDN purge. Response: HTTP 204
GET/api/v1/urls/{short_code}/stats
Response: { "clicks": 42810, "top_countries": [...], "referrers": [...] } (served from ClickHouse, async)

Interview follow-ups

How would you prevent someone from generating billions of short codes to exhaust your namespace?
Rate limit creates per user/IP. Require auth for custom aliases. At 1k writes/s, exhausting 62^7 codes takes ~111,000 years.
What if the same long URL is submitted twice — do you return the same short code?
Design decision. Simple: always generate a new code. Deduplicated: hash long_url, check DB before creating. Dedup saves storage but adds a DB read on every write. Pick based on whether analytics per-link matters.
How do you handle geographic latency?
CDN handles redirects at edge globally. For writes, route to nearest regional write cluster. Use Postgres logical replication to sync regions with ~100ms lag.
What changes if you need to support 1 billion URLs?
Shard Postgres by short_code prefix (16 shards). Upgrade to Redis Cluster with 32+ nodes. Extend short code to 8 chars (62^8 = 218T capacity). CDN becomes mandatory, not optional.
How would you implement URL expiration efficiently?
Store expires_at in DB and Redis TTL. At redirect time, check expiry inline (no extra query if Redis TTL handles it). Background job scans expired rows daily for hard deletes in batches of 10k.
How do you instrument this system?
Metrics: redirect p99 latency, cache hit rate, write QPS, error rate by type. Alerts: cache hit rate <70%, p99 >200ms, 5xx error rate >0.1%. Trace: every redirect tagged with short_code, cache hit/miss, region.
