URL Shortener

Convert long URLs to short aliases and redirect users at scale.

30-40 min interview Easy
Say in the first 5 min
  • Clarify read vs write ratio — shorteners are read-heavy (100:1), cache aggressively
  • Pick hash length early: 7 chars base62 = 62^7 = 3.5 trillion URLs
  • ID generation is the hard problem — not the redirect logic
  • Expiration + analytics are the two features that complicate the data model
  • A CDN in front of the redirect service cuts latency to under 10ms globally

Mental model

A URL shortener is a phone book in reverse: instead of looking up a number by name, you mint a short, unique key and store what it points to. The entire system exists to make that lookup fast at massive read scale.

Data model

CREATE TABLE urls (
  id          BIGINT PRIMARY KEY,         -- Snowflake ID
  short_code  VARCHAR(10) UNIQUE NOT NULL,  -- base62(id), e.g. "gH3xK2p"
  long_url    TEXT NOT NULL,
  user_id     BIGINT,                     -- nullable (anonymous)
  created_at  TIMESTAMP DEFAULT NOW(),
  expires_at  TIMESTAMP,                  -- null = never
  click_count BIGINT DEFAULT 0            -- denormalized for quick reads
);

CREATE INDEX idx_short_code ON urls(short_code);  -- lookup hot path
Storage estimate: 100M rows × ~500B avg = 50GB. Fits on a single Postgres instance with read replicas for scale. Partition by created_at monthly if you hit 1B+ rows.

Key design decisions

ID generation: Snowflake over hash
Why: Hashing (MD5, SHA) risks collisions and requires a uniqueness check on every write. Snowflake IDs are globally unique by construction — no DB round-trip needed to verify.
Tradeoff: Snowflake requires a coordination service (or node ID config). DB auto-increment is simpler but becomes a write bottleneck above ~10k writes/s.
301 (permanent) vs 302 (temporary) redirect
Why: 302 means every click hits your servers (analytics works). 301 means browsers cache it and you lose click visibility. Default to 302 unless the client explicitly opts into 301.
Tradeoff: 302 = more server load. 301 = no analytics after first visit per user.
Cache at redirect service + CDN layer
Why: Two-layer caching: Redis holds the hot 20% of URLs in memory. CDN caches the HTTP 302 responses at edge nodes for globally fast reads.
Tradeoff: Stale CDN cache is a problem on URL deletion. Solve with a low TTL (60s) or purge API call on delete.
Custom aliases: separate namespace
Why: User-defined aliases (e.g. /my-brand) go into a separate custom_aliases table to avoid colliding with Snowflake-generated codes.
Tradeoff: Two lookup paths in redirect service. Keep it simple with a single DB query that checks both tables via UNION or a type column.

Failure modes

⚠
Redis cluster down: latency spikes, redirects fall to DB. Detect via cache miss rate >80% alert. Mitigate with a circuit breaker to prevent DB overload.
⚠
Primary DB down: writes fail, reads degrade to replicas. Detect via replication lag + write error spikes. Mitigate by promoting a read replica.
⚠
CDN misconfiguration: all redirects hit origin servers — a 10x RPS spike. Mitigate with rate limiting at the gateway and auto-scaled redirect pods.
✗
Malicious URL abuse: integrate Google Safe Browsing API at write time, allow reporting, auto-disable flagged URLs.

Common follow-up questions

How would you prevent someone from generating billions of short codes to exhaust your namespace?
Rate limit creates per user/IP. Require auth for custom aliases. At 1k writes/s, exhausting 62^7 codes takes ~111,000 years.
What if the same long URL is submitted twice — do you return the same short code?
Design decision. Simple: always generate a new code. Deduplicated: hash long_url, check DB before creating. Dedup saves storage but adds a DB read on every write.
How do you handle geographic latency?
CDN handles redirects at edge globally. For writes, route to nearest regional write cluster. Use Postgres logical replication to sync regions with ~100ms lag.
What changes if you need to support 1 billion URLs?
Shard Postgres by short_code prefix (16 shards). Upgrade to Redis Cluster with 32+ nodes. Extend short code to 8 chars (62^8 = 218T capacity). CDN becomes mandatory, not optional.
How would you implement URL expiration efficiently?
Store expires_at in DB and Redis TTL. At redirect time, check expiry inline (no extra query if Redis TTL handles it). Background job scans expired rows daily for hard deletes in batches of 10k.
How do you instrument this system?
Metrics: redirect p99 latency, cache hit rate, write QPS, error rate by type. Alerts: cache hit rate <70%, p99 >200ms, 5xx error rate >0.1%. Trace: every redirect tagged with short_code, cache hit/miss, region.
